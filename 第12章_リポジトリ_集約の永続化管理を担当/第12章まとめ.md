# 第12章「リポジトリ」〜集約の永続化管理を担当〜

## 第12章扉（page_0167）

前章では集約の生成を「ファクトリ」として紹介した。本章では、集約の保存・削除・取得を担当するオブジェクト「リポジトリ（Repository）」について解説する。集約をどのデータストアにどのように保存するかという実装の詳細をリポジトリに閉じ込めることで、ドメインモデルをクリーンに保つことができる。

---

## 12.1 リポジトリとは（page_0167〜page_0168）

一般的に「リポジトリ（貯蔵庫・保管場所）」を意味する。DDDにおける Apache Subversion などバージョン管理システムの「リポジトリ」とは異なる。DDDのリポジトリは、集約の永続化（ストレージへの保存・検索・削除）を担当する**ファーストクラスオブジェクト**である。

### リポジトリで集約を操作する流れ

【図 12.1 リポジトリの役割と流れ】

- **A. 新しい集約をデータベースに記録する場合**
  1. クライアントからファクトリに集約の生成を依頼
  2. ファクトリが新しい集約を生成して返す
  3. クライアントがリポジトリに集約の追加を依頼
  4. リポジトリがデータベースに集約を保存

- **B. データベースから集約を取得して利用する場合**
  1. クライアントがリポジトリに集約の取得を依頼
  2. リポジトリがデータベースから集約を取得して返す
  3. クライアントが集約に何らかの処理を依頼
  4. クライアントが集約の変更をリポジトリに保存依頼

「A. 新しい集約をデータベースに記録する場合」の補足：クライアントから見て、どこにデータが格納されているかを意識しなくてよい。それはリポジトリに閉じ込められている。複数1回の処理で追加メソッドを実行し、完了したらデータベースに保存されています。

### Evans 氏の DDD本におけるリポジトリの説明

IDDDの本では、リポジトリは以下のように定義されている：

> グローバルにアクセスできる必要があるオブジェクトに対して、1つのタイプのオブジェクトに対して1つのリポジトリを作成する。あるオブジェクトを「知っている」インフラストラクチャを提供して、そのオブジェクトへの参照を持ったオブジェクトに提供できるようにしましょう。

これを踏まえ、DDD・DDDのリポジトリの特徴は次の通りである。クライアント（アプリケーションサービスなど）から見れば、データストアの種類（RDB、NoSQLなど）や保存の詳細を知らずに集約を操作できる。データストアに対する操作詳細（クエリ、コネクション管理等）はリポジトリが完全に隠蔽する。

このため「コレクション指向」を意識することがDDD的リポジトリの特徴と言える。データベースに対する操作（永続化のためのクエリや手続き）はリポジトリの内部に閉じている。コレクション（配列・リスト）を操作しているかのような API を提供することが理想である（IDDD本の用語「コレクション指向リポジトリ」）。

### リポジトリと DAO の違い

リポジトリと DAO（Data Access Object）はデータアクセスを担うという点で似ているが、設計の意図が異なる。

| | リポジトリ | DAO |
|--|-----------|-----|
| 責務の主体 | ドメインモデル側 | データベース側 |
| 抽象化の対象 | 集約コレクション | テーブル／ストレージ操作 |
| Hibernate等のO/Rマッパー | 利用可能（実装詳細を隠蔽） | 利用可能（O/Rマッパー自体がDAO的） |

DAO はどちらかというとデータストアのテーブル単位での操作を意識しており、集約という単位の概念がない。対してリポジトリは集約単位で操作するため、DDD的に適切な抽象化レベルを持つ。

## 12.2 リポジトリを利用する「アプリケーションサービス」（page_0169〜page_0170）

そのため、リポジトリを利用するクライアントは Java（C#）だと次のようになる。リスト12.1では `IdentityApplicationService`（Java）が示されている。

### リスト 12.1 リポジトリを呼び出すアプリケーションサービス（Java）

```java
// ■ユーザーIDに関するアプリケーションサービス
public class IdentityApplicationService {

    // ▼リポジトリ（Spring Framework の DI にて設定）
    @Autowired
    private UserRepository userRepository;

    // ▼ユーザーを登録する（トランザクション宣言をアノテーション指定）
    @Transactional
    public User registerUser(RegisterUserCommand aCommand) {

        Tenant tenant = this.existingTenant(aCommand.getTenantId());

        // ユーザーの集約を生成（テナント集約のファクトリから生成）
        User user =
            tenant.registerUser(
                aCommand.getInvitationIdentifier(),
                aCommand.getUsername(),
                （中略）
            )

        // ユーザーをリポジトリに登録
        this.userRepository().add(user);
        return user;
    }
```

このコードから分かるように、アプリケーションサービスはファクトリで生成した集約を受け取り、リポジトリの `add()` メソッドを呼ぶだけでよい。データベースへの保存処理はリポジトリ内部に隠蔽されているため、アプリケーションサービスはデータストアの詳細を意識しない。

### リポジトリのモジュール構成

【図 12.2 リポジトリに関するレイヤ/モジュール（パッケージ名省略）構成】

DDDのレイヤ構造上、リポジトリは次のように配置される：

- **ドメイン層**：リポジトリの**インターフェイス**を定義（`UserRepository` など）
- **インフラストラクチャ層（DIP）**：リポジトリの**実装クラス**を置く（`HibernateUserRepository` など）
- **アプリケーション層**：リポジトリのインターフェイスを通じて集約を操作

A に「ドメインモデル」、B に「アプリ」、C（DIPアーキテクチャ）のポートやアダプターに「リポジトリ実装」が位置する。インターフェイスをドメイン層に置くことで、インフラ依存をドメインから切り離せる（依存性逆転の原則）。

### データベースのトランザクション管理

リポジトリのメソッドはそれぞれトランザクション処理を含まない。むしろトランザクションはアプリケーションサービス側で管理する（Java/Spring の `@Transactional` アノテーション等）。アノテーションを宣言することで、Spring がトランザクション開始・コミット・ロールバックを自動管理する。リポジトリはあくまでストレージへのアクセスを担当し、トランザクション境界はアプリケーション層に委ねる。

## 12.3 コレクション指向のリポジトリ（page_0172〜page_0175）

「コレクション指向」のリポジトリとは、対象ストレージが RDB（などのO/Rマッパーを使えるもの）のときに使用し、集約の情報を複数のテーブルに分割して保存する。

### インターフェイスの設計

コレクション指向のリポジトリ・インターフェイス（Java / C#）の設計は次の通り。

```java
// ■ユーザー集約を管理するリポジトリ
public interface UserRepository {
    // ▼リポジトリに追加
    public void add(User aUser);

    // ▼リポジトリから削除
    public void remove(User aUser);

    // ▼名前でユーザー集約を取得
    public User userWithUsername(
        TenantId aTenantId,
        String aUsername);
    （中略）
}
```

メソッドは `add`（追加）、`remove`（削除）、`userWithUsername`（検索）の3種が基本。コレクション（リスト）を操作するような API になっている点がポイント。`save`（上書き保存）ではなく `add`（追加）メソッドを使うのが特徴で、これはO/Rマッパーが変更追跡（ダーティチェッキング）を行うため、明示的な `save` が不要なことを前提にしている。

### コレクション指向の実装クラス

実装コード（Java）は O/R マッパーを利用する `HibernateUserRepository` クラス。Hibernate を利用して保存や削除を呼び出している。取得においても、クエリを組み立てて条件に合った集約を取り出している。

### リスト 12.3 「コレクション指向のリポジトリ」の実装クラス（Java）

```java
// ■ユーザー集約を管理するリポジトリの実装クラス
public class HibernateUserRepository
    extends AbstractHibernateSession implements UserRepository {

    // ▼リポジトリに追加
    @Override
    public void add(User aUser) {
        try {
            this.session().saveOrUpdate(aUser);
        } catch (ConstraintViolationException e) {
            throw new IllegalStateException("User is not unique.", e);
        }
    }

    // ▼リポジトリから削除
    @Override
    public void remove(User aUser) {
        this.session().delete(aUser);
    }

    // ▼名前でユーザー集約を取得
    @Override
    public User userWithUsername( TenantId aTenantId,  String aUsername) {
        Query query = this.session().createQuery(
            "from  com.saasovation.identityaccess.domain.model.identity.User as _obj_ "
            + "where _obj_.tenantId = ? "
            + "and _obj_.username = ?");
        query.setParameter(0, aTenantId);
        query.setParameter(1, aUsername, Hibernate.STRING);
        return (User) query.uniqueResult();
    }
    （中略）
}
```

## 12.4 永続指向のリポジトリ（page_0175〜page_0177）

「永続指向のリポジトリ」は、対象ストレージが NoSQL などのキーバリューストアのときに使用し、集約の情報全体をまとめて保存する。

### インターフェイスの設計

### リスト 12.4 「永続指向のリポジトリ」の公開インターフェイス（Java）

```java
// ■プロダクト集約を管理するリポジトリ
public interface ProductRepository {

    // ▼プロダクト集約を取得（テナント ID＋プロダクト ID）
    public Product productOfId(TenantId aTenantId, ProductId aProductId);

    // ▼リポジトリに保存
    public void save(Product aProduct);

    // ▼リポジトリから削除
    public void remove(Product aProduct);
    （中略）
}
```

集約全体を MongoDB や LevelDB の 1 つのレコード（ドキュメント）として保存するので、`add` メソッドではなく `save` メソッドを使う。それ以外はコレクション指向のインターフェイス設計と同じ。

### 永続指向のリポジトリ実装クラス

続けて、実装クラスのコード（Java）を見てみる。ここでは NoSQL の 1 つである LevelDB 向けの実装となっている。

### リスト 12.5 「永続指向のリポジトリ」の実装クラス（Java）

```java
// ■LevelDBでの実装リポジトリ
public class LevelDBProductRepository
    extends AbstractLevelDBRepository implements ProductRepository {

    private static final String PRIMARY = "PRODUCT#PK";

    // ▼コンストラクタで、DB の場所として、コンテキスト名を使用
    public LevelDBProductRepository() {
        super(LevelDBDatabasePath.agilePMPath());
    }

    // ▼プロダクト集約を取得（テナント ID＋プロダクト ID）
    @Override
    public Product productOfId(TenantId aTenantId, ProductId aProductId) {
        // 取得に使うキーを設定する
        LevelDBKey primaryKey =
            new LevelDBKey(PRIMARY, aTenantId.id(), aProductId.id());
        Product product =
            LevelDBUnitOfWork.readOnly(this.database())
                .readObject(primaryKey.key().getBytes(), Product.class);
        return product;
    }

    // ▼プロダクトをリポジトリに保存
    @Override
    public void save(Product aProduct) {
        LevelDBKey lockKey = new LevelDBKey(PRIMARY, aProduct.tenantId().id());
        LevelDBUnitOfWork uow = LevelDBUnitOfWork.current();
        uow.lock(lockKey.key());
        this.save(aProduct, uow);
    }
    （中略）
}
```

ここでは LevelDB に対する保存／削除／取得処理を記述している。NoSQL では、データの格納と取得のために一意なキーの設計が必要です。IDDD 本では「境界づけられたコンテキストの短縮名」「集約の名前」「一意な識別子」を組み合わせる案を提示している。

このプロダクトという集約の例であれば、まず DB そのものをコンテキスト名（`LevelDBDatabasePath.agilePMPath` の部分）で分離し、それ以外の集約を示す定数「`PRODUCT#PK`」と一意な識別子「テナント ID」＋「プロダクト ID」の組み合わせをキーとして使用している。

## 12.5 まとめ（page_0178）

本章では DDD におけるリポジトリについて紹介した。集約の永続化を管理するにあたって、いくつかの設計指針、実装パターンがあることが理解できたと思う。次章では「境界づけられたコンテキストの結合」について紹介する。

### 章のポイント整理

| 項目 | 内容 |
|------|------|
| リポジトリの役割 | 集約の永続化（追加・削除・取得）を担う。クライアントからデータストアの詳細を隠蔽する |
| 配置 | インターフェイスはドメイン層、実装はインフラ層（依存性逆転） |
| トランザクション管理 | アプリケーション層で管理（`@Transactional` など） |
| コレクション指向 | RDB向け。O/Rマッパーを利用。`add`/`remove`メソッドを提供 |
| 永続指向 | NoSQL向け。集約全体を1レコードで保存。`save`メソッドを提供 |
| DAOとの違い | リポジトリは集約単位、DAOはテーブル単位の操作を意識する |
