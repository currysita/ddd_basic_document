# 第6章「値オブジェクト」〜振る舞いを持つ不変オブジェクト〜

## 章の概要

前章で一章まるまる「エンティティ」を取り上げた。本章では、DDDのもう一方の重要な構成要素である「値オブジェクト（Value Object）」を取り上げる。エンティティとともに、DDDの「値オブジェクト」はオブジェクト指向設計を基礎としながらも、より高いレベルの抽象化によってドメインモデルを豊かに表現するための手法である。

---

## 6.1 値オブジェクトとは

DDDにおける「値オブジェクト」とは、属性を持ち、定義された振る舞いも備えたオブジェクトである。ただし「エンティティ」とは異なり、一意な識別子（ID）を持たない。

DDD 本（Eric Evans 著 "Domain-Driven Design"）の定義：
> 「ある物の側面を表現するオブジェクトで、概念的な同一性を持たないもの」

3点の特徴として挙げられている：
1. オブジェクトが何であるかではなく、**その属性のみが重要**
2. **インスタンスの等値性**（属性が同じなら等しいとみなす）
3. **不変性**（一度生成されたら状態を変えない）

これらの特性により、値オブジェクトはシンプルで予測しやすく、テストしやすいオブジェクトになる。DDDでは値オブジェクトを積極的に利用することが推奨されている。

---

## 6.2 値オブジェクトの特徴

値オブジェクトには4つの特徴がある。

### 表6.1 値オブジェクトが持つ特性

| No. | 値オブジェクトの特性 | 説明 |
|-----|------------------|------|
| 1 | 計測／定量化／説明 | ドメイン内の物事を計測・定量化・説明する |
| 2 | 不変性 | 状態を変えることができない |
| 3 | 概念的な統一体 | 関連する属性を一つの概念的な統一体に結合する |
| 4 | 交換可能性 | 計測値や説明が変わると、全体として別のオブジェクトに置き換えられる |
| 5 | 等値性 | 含まれる属性の値ですべてのインスタンスを比較できる |
| 6 | 副作用のない振る舞い | メソッドの結果は新しい値オブジェクトを返し、自身は変化しない |

### DDD 本における値オブジェクトの解説

値オブジェクトにはプログラミングでいうと数値・文字列なども含みうるが、DDDにおける値オブジェクトはより豊かなもの。`AreaSize`（面積サイズ）クラスを例に、Evans は値オブジェクトが持つ振る舞いについて解説している。

---

### (1) 計測／定量化／説明

値オブジェクトは、コンテキスト内の物事を計測・定量化・説明できるものである。まず、値オブジェクトを見分けるときに重要なのは「そのドメインの専門家がどのような言葉を使うか」を理解することである。

### 表6.2 値オブジェクトを見分ける方法

| 区分 | 内容／例 |
|-----|---------|
| 計測 | 数量（重さ、長さ、金額など）の計測 |
| 定量化 | 指定された単位での定量化（USD100、5kg など） |
| 説明 | 品名・製品仕様・色・サイズなど、物を説明する値 |

値オブジェクトが計測・定量化・説明のいずれかに当てはまる場合、エンティティではなく値オブジェクトとして設計・実装できるかを検討すること。

### (2) 不変

値オブジェクトはプロパティを変更することはできない。そのため、エンティティのようにセッターメソッドを持ってはならず、コンストラクタでのみ値をセットする。DDDの多くのプロジェクトでは、値を「変更」したいときはオブジェクトを**丸ごと置き換える**（交換可能性）。

なお、この不変性はユビキタス言語において自然に感じられるように設計することが重要。

### 表6.3 不変オブジェクトの利点

| 利点 | 説明 |
|-----|------|
| スレッドセーフ | 状態が変わらないのでロックが不要 |
| キャッシュ | 変わらないのでキャッシュして使い回せる |
| 共有可能 | 複数の箇所で同じインスタンスを共有できる |

### (3) 概念的な統一体

値オブジェクトは複数の関連する属性を一つにまとめた「統一体」として機能する。例えば「金額」は「数値」と「通貨単位」の組み合わせであり、これらをまとめて一つの値オブジェクトとして扱う。

### (4) 交換可能性

値オブジェクトは不変なので、値を「変えたい」ときは新しいオブジェクトに**丸ごと置き換える**。これを交換可能性と呼ぶ。

```java
// 変更前
Money price = new Money(100, "USD");
// 値の変更 → 置き換える
price = new Money(200, "USD");
```

### (5) 等値性

値オブジェクトは同一性（ID）ではなく、**属性の値によって等値かどうかを判断**する。二つのオブジェクトが全く同じ属性を持つ場合、それらは等しいとみなす。

```java
Money a = new Money(100, "USD");
Money b = new Money(100, "USD");
// a と b は等しい（エンティティなら別のオブジェクト）
```

等値性を実現するには `equals()` メソッドをオーバーライドすることが多い。

### (6) 副作用のない振る舞い

「副作用のない関数（純粋関数）」と同様に、値オブジェクトのメソッドは**自身の状態を変えず、新しい値オブジェクトを返す**。

```java
// NG：セッターで内部状態を変える
priority.setRank("High");

// OK：新しいオブジェクトを返す
BusinessPriority newPriority = priority.withRating(new BusinessPriorityRatings(...));
```

## 6.3 複数コンテキストの結合を緩やかにする値オブジェクト

値オブジェクトはコンテキスト間の依存を緩める効果がある。あるコンテキストで定義されたエンティティが別のコンテキストと連携する際、エンティティ全体を渡すのではなく、**値オブジェクトとして必要な情報のみを渡す**ことで疎結合になる。

### 図6.1 コンテキスト間を橋渡しする値オブジェクト（IDDD 図6-2を参照）

【図】コラボレーションコンテキストとプロジェクト管理コンテキストが値オブジェクト（`Author` など）を経由してやりとりする構造。エンティティをそのまま渡さず、値オブジェクトに変換して受け渡す。

- コラボレーションコンテキストが「ユーザー」「モデレーター」などのエンティティを持つ
- プロジェクト管理コンテキストは「Author（著者）」「Owner（所有者）」などの値オブジェクトを経由して受け取る
- これにより両コンテキスト間の結合が緩くなる

この設計では、コンテキスト間の変換（翻訳）を値オブジェクトが担う。

## 6.4 ドメインモデルの区分や種類を表す「標準型（タイプコード）」

ドメインモデリングを進めていくと、何らかの「区分」や「種類」を表す値が必要になる場面が多い。そのような場合にシンプルな文字列や整数値で表すのではなく、DDDでは**標準型（タイプコード）**として値オブジェクトを使う方法を推奨する。

### 図6.2 標準型（タイプコード）の実装例（C#）

```csharp
public class GroupMemberType
{
    public static readonly GroupMemberType Member = new GroupMemberType("Member");
    public static readonly GroupMemberType Moderator = new GroupMemberType("Moderator");

    private string _memberType;

    protected GroupMemberType() { }

    private GroupMemberType(string memberType)
    {
        _memberType = memberType;
    }

    public bool IsMember    => _memberType.Equals("Member");
    public bool IsModerator => _memberType.Equals("Moderator");
}
```

列挙型（enum）は多くの言語で提供されているが、ドメインロジックを持てないなどの制約がある。Martin Fowler 氏の著書などでも「タイプコードをクラスで表現する」パターンが紹介されており、拡張性が高い。

### 列挙型（enum）で標準型を実装

```java
// Java
public enum GroupMemberType {
    Member,
    Moderator
}
```

### 値オブジェクト（クラス）として標準型を実装

クラスとして実装することで、ドメイン固有の振る舞い（`isMember()` など）を持たせることができる。また、コンストラクタを `private` にすることで外部からのインスタンス生成を防ぎ、定義済みの値のみを使用させることができる。

## 6.5 値オブジェクトの実装コード

本節では、値オブジェクトの実装例を示す。ここではプロジェクト管理ドメインの「ビジネス優先度（`BusinessPriority`）」クラスを例に取る。

### 図6.4 ビジネス優先度に関する値オブジェクト（IDDD 図6-5を参照）

【図】`BusinessPriority`（値オブジェクト）と `BusinessPriorityRatings`（値オブジェクト）の関係図。

- `BusinessPriority` は `BusinessPriorityRatings` を内包する
- `BusinessPriorityRatings` は `benefit`・`penalty`・`cost`・`risk` の4つの数値属性を持つ

### `BusinessPriorityTotals` クラス（値オブジェクト）の実装

`BusinessPriorityTotals` はビジネス優先度の集計値（合計）を保持する値オブジェクト。

### 図6.5 `BusinessPriorityTotals`「値オブジェクト」の実装（C#）

```csharp
// BusinessPriorityTotalsの実装例（概略）
public class BusinessPriorityTotals
{
    private decimal _totalBenefit;
    private decimal _totalPenalty;
    private decimal _totalCost;
    private decimal _totalRisk;
    private decimal _totalValue;
    private decimal _totalPercent;

    // コンストラクタでのみ値をセット（不変性）
    public BusinessPriorityTotals(
        decimal totalBenefit, decimal totalPenalty,
        decimal totalCost,    decimal totalRisk,
        decimal totalValue,   decimal totalPercent)
    {
        _totalBenefit  = totalBenefit;
        _totalPenalty  = totalPenalty;
        _totalCost     = totalCost;
        _totalRisk     = totalRisk;
        _totalValue    = totalValue;
        _totalPercent  = totalPercent;
    }
    // セッターなし、ゲッターのみ
}
```

全属性をコンストラクタで受け取り、セッターを持たないことで不変性を実現している。

### `BusinessPriorityRatings` クラス（値オブジェクト）

```csharp
// BusinessPriorityRatingsの実装例（概略）
public class BusinessPriorityRatings
{
    private int _benefit;
    private int _penalty;
    private int _cost;
    private int _risk;

    public BusinessPriorityRatings(int benefit, int penalty, int cost, int risk)
    {
        _benefit = benefit;
        _penalty = penalty;
        _cost    = cost;
        _risk    = risk;
    }
    // セッターなし
}
```

### 属性を変更した新しいオブジェクトを返す `with` メソッド

値オブジェクトで「属性の一部を変えた新しいインスタンス」を返すには、**コピーコンストラクタ**や **`with` メソッド**パターンを使う。

```csharp
// コピーコンストラクタ例
public BusinessPriority(BusinessPriority businessPriority)
{
    // 自分自身を引数として受け取り、全属性をコピー
    _ratings = businessPriority._ratings;
}
```

このようなコピーコンストラクタは、値オブジェクトに不変性があるかテストを書くときによく使用される。（なお、値オブジェクトのテストでは、ドメインエキスパートとビジネス的な話をして、その仕様を満たすようにテストコードを記載するように努める。）
