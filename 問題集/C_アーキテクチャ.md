## C-01　レイヤ化アーキテクチャの4層（知識確認）

**問い**

DDDのレイヤ化アーキテクチャを構成する4つの層を、上位から順に列挙し、各層の役割を説明してください。

---

**模範解答**

| 層 | 役割 |
|---|---|
| **ユーザーインターフェイス層** | UIの入出力を担当。ドメインオブジェクトを直接渡すか、プレゼンテーションモデルに変換してUIに渡す |
| **アプリケーション層** | ユースケースの調整・トランザクション管理を担当。ビジネスロジックはドメイン層に委譲する |
| **ドメイン層** | すべてのビジネスロジックとビジネスルールを含む最重要な層。エンティティ・値オブジェクト・ドメインサービス・イベントが配置される |
| **インフラストラクチャ層** | データの永続化（リポジトリ実装）・メッセージング・メールなど技術的な関心事を担当 |

## C-02　依存関係逆転の原則（DIP）の定義（知識確認）

**問い**

依存関係逆転の原則（DIP）を定義してください。

---

**模範解答**

Robert C. Martin が提唱した SOLID 原則の「D」。上位モジュールが下位モジュールに依存するのではなく、両者が抽象（インターフェイス）に依存するよう設計する原則。プラグインのように依存性を注入することで、インフラストラクチャ層をドメイン層から切り離せる。

---

## C-03　ヘキサゴナルアーキテクチャの定義と基本的な考え方（知識確認）

**問い**

ヘキサゴナルアーキテクチャとは何か。基本的な考え方と3つの主なメリットを説明してください。

---

**模範解答**

Alistair Cockburn が提唱したアーキテクチャ。ドメインをシステムの中心（ヘキサゴン）に配置し、外部との接続を「ポート」と「アダプター」によって実現する。「ポート＆アダプター」アーキテクチャとも呼ばれる。

**3つのメリット**

1. ドメインオブジェクトがアーキテクチャ全体を通じてピュアに保たれる
2. インフラストラクチャに依存しないためテストがしやすい
3. 複数の入出力ポートを自由に追加できる

## C-04　CQRSの定義とCQSとの違い（知識確認）

**問い**

CQS（コマンドクエリ分離）とCQRS（コマンドクエリ責務分離）をそれぞれ定義し、両者の違いを説明してください。

---

**模範解答**

- **CQS（Command Query Separation）**：Bertrand Meyer が提唱した設計原則。メソッドは状態を変更する「コマンド」か値を返す「クエリ」のどちらか一方であるべきという原則。メソッドレベルの分離。
- **CQRS（Command Query Responsibility Segregation）**：CQSをアーキテクチャレベルに拡張したもの。コマンド（Write）とクエリ（Read）を別々のモデル・データストアに分離するパターン。

**違い**：CQSはメソッド設計の原則、CQRSはアーキテクチャレベルの分離パターン。

## C-05　イベントソーシングの定義（知識確認）

**問い**

イベントソーシングとは何か、従来の状態保存方式と何が異なるかを説明してください。

---

**模範解答**

**イベントソーシング**：ドメインオブジェクトの現在の状態を保存するのではなく、状態を変化させたイベントのログを記録することで状態を管理する手法。

**従来方式との違い**

| | 従来の状態保存 | イベントソーシング |
|---|---|---|
| **保存内容** | 現在の状態 | 状態変化を引き起こしたイベントのログ |
| **過去の再現** | 困難 | イベントを再生（リプレイ）することで任意の時点を再構築できる |
| **監査ログ** | 別途実装が必要 | イベントログがそのまま監査証跡になる |

## C-06　イベント駆動アーキテクチャの定義（知識確認）

**問い**

イベント駆動アーキテクチャとは何か、またパイプ＆フィルターパターンとの関係を説明してください。

---

**模範解答**

**イベント駆動アーキテクチャ**：システム内の状態変化を「イベント」として発行し、それを受け取ったサブスクライバが処理を行うアーキテクチャスタイル。非同期処理と疎結合を実現しやすい。

**パイプ＆フィルターパターン**：イベント駆動アーキテクチャのシンプルな実装スタイルのひとつ。フィルター（処理ステップ）がデータを受け取り処理してパイプ（チャンネル）で次のフィルターに渡す。各フィルターは独立して動作し、並列化やテストが容易。

---

## C-07　長期プロセス（サーガ）の役割（知識確認）

**問い**

長期プロセス（サーガ）とは何か、どのような課題を解決するために使うのかを説明してください。

---

**模範解答**

**長期プロセス（サーガ）**：複数のサブステップからなる長時間かかる処理を管理するパターン。各サブプロセスをイベントで連鎖させ、状態を管理しながら非同期に実行する。

**解決する課題**

- 分散トランザクションの代替として、複数のBCにまたがる処理を整合性を保ちながら実行する
- 処理に時間がかかる場合（例：電話番号リストからの条件一致処理）のユーザーストレスやタイムアウトを防ぐ
- 各サブプロセスを独立して非同期実行することで、システム全体の応答性を高める

## C-08　DIPがなぜドメイン層の独立性を高めるのか（理解確認）

**問い**

依存関係逆転の原則（DIP）を適用することで、ドメイン層がインフラストラクチャ層に依存しなくなる仕組みを説明してください。

---

**模範解答**

**DIPなしの問題**

通常のレイヤ化アーキテクチャでは、ドメイン層のリポジトリ実装がインフラストラクチャ層（DB）に直接依存するため、ドメイン層がインフラ技術の変更に影響される。

**DIPを適用した仕組み**

1. ドメイン層にリポジトリの「インターフェイス（抽象）」を定義する
2. インフラストラクチャ層がそのインターフェイスを実装する
3. ドメイン層はインターフェイスにのみ依存し、実装（DB技術）を知らなくて済む

結果として、上位モジュール（ドメイン層）と下位モジュール（インフラ層）が両方とも抽象（インターフェイス）に依存する形になり、依存の向きが逆転する。

## C-09　CQRSがなぜ読み取りと書き込みを分離するのか（理解確認）

**問い**

CQRSで読み取り（クエリ）と書き込み（コマンド）を分離することで何が解決されるのか、またその代償は何かを説明してください。

---

**模範解答**

**解決されること**

- 書き込みモデル（ドメインモデル）は正規化されているため、複雑な読み取りクエリが非効率になる
- 読み取り専用の非正規化データストアを用意することで、画面表示に最適化した高速クエリが可能になる
- 読み取りと書き込みを独立してスケールアウトできる

**代償（注意点）**

- コマンド実行後、読み取りモデルが非同期で更新されるため「結果整合性」になる
- 厳密な整合性が必要な場面では使いにくい
- すべてのシステムに採用すべきではなく、オーバーエンジニアリングになりうる

---

## C-10　イベント駆動アーキテクチャとイベントソーシングの関係（理解確認）

**問い**

イベント駆動アーキテクチャとイベントソーシングはどう異なり、どのように関係するかを説明してください。

---

**模範解答**

| | イベント駆動アーキテクチャ | イベントソーシング |
|---|---|---|
| **目的** | イベントを介したコンポーネント間の疎結合・非同期連携 | 状態変化をイベントとして永続化し、過去の再現を可能にする |
| **主な関心事** | 処理の流れ（どのコンポーネントが何のイベントを処理するか） | 状態の記録（どのイベントが状態をどう変えたか） |

**関係**：イベントソーシングはイベント駆動アーキテクチャと組み合わせることが多い。イベントソーシングで記録したイベントをイベント駆動で各コンポーネントに配信することで、変更の追跡と非同期処理を両立できる。一方、イベント駆動アーキテクチャがイベントソーシングを必須とするわけではない。

## C-11　レイヤ化→DIP→ヘキサゴナルへのアーキテクチャ進化（理解確認）

**問い**

レイヤ化アーキテクチャからDIPに基づくレイヤ化アーキテクチャ、そしてヘキサゴナルアーキテクチャへと進化する過程で、何が改善されていくかを説明してください。

---

**模範解答**

**レイヤ化アーキテクチャ**

- 上位層から下位層への依存のみ許容する基本構造
- ドメイン層がインフラストラクチャ層に直接依存するため、インフラ変更の影響を受ける

**DIPに基づくレイヤ化アーキテクチャ**

- DIPを適用してドメイン層にインターフェイスを定義し、インフラ層がそれを実装する
- ドメイン層がインフラ技術から切り離され、独立性が高まる

**ヘキサゴナルアーキテクチャ**

- ドメインを中心に置き、すべての外部とのやり取りをポート（インターフェイス）とアダプター（実装）で行う
- REST API・メッセージング・UIなど複数の入出力を同じ方式で扱えるため、テストと本番で同じインターフェイスを使用できる
- ドメイン層の独立性が最も高い状態

---

## C-12　ポートとアダプターそれぞれの責務の違い（理解確認）

**問い**

ヘキサゴナルアーキテクチャにおける「ポート」と「アダプター」それぞれの責務を説明してください。

---

**模範解答**

- **ポート**：ドメインが外部と通信するためのインターフェイス（契約）。ドメイン層に定義される。具体的な技術（REST・DB・メッセージングなど）を知らない。
- **アダプター**：ポート（インターフェイス）を実装し、具体的な技術とドメインを繋ぐ部品。インフラストラクチャ層に配置される。

**関係**：アダプターはポートに着脱可能な形で提供されるため、インフラ技術を変更してもドメインに影響しない。テスト時はモックのアダプター、本番時は実際のDBアダプターを差し替えて使用できる。

---

## C-13　要件に基づく適切なアーキテクチャの選択と設計（応用）

**問い**

以下の要件を持つシステムを新規開発します。適切なアーキテクチャを選択し、その理由を説明してください。

- 医師が入力した診断内容が患者ポータルのダッシュボードにリアルタイムで近い速度で反映される必要がある
- 監査当局から「過去3年間の診断変更履歴をすべて再現できること」を求められている
- モバイルアプリ・Webアプリ・外部パートナーAPI の3種の入口が必要

---

**模範解答**

**推奨アーキテクチャ**：ヘキサゴナルアーキテクチャ + CQRS + イベントソーシング

**理由**

| 要件 | 採用するアーキテクチャ要素 | 理由 |
|---|---|---|
| ダッシュボードのリアルタイム近い反映 | CQRS（クエリモデル） | 読み取り専用の非正規化データストアで高速なダッシュボード表示を実現。コマンドとクエリを分離し、表示に最適化する |
| 過去履歴の再現 | イベントソーシング | 状態変化をすべてイベントとして記録し、任意の時点の状態をリプレイで再現できる |
| 3種の入口 | ヘキサゴナルアーキテクチャ（複数ポート） | モバイル・Web・外部APIを別々のアダプターとして実装。ドメインに影響を与えず入口を追加できる |

## C-14　マイクロサービス設計でCQRSやイベントソーシングを活用する（応用）

**問い**

ECサイトを「注文サービス」「在庫サービス」「通知サービス」のマイクロサービスに分割します。CQRSとイベントソーシングを活用して、注文確定から在庫更新・顧客通知までの流れを設計してください。

---

**模範解答**

**設計**

1. **注文サービス（CQRS + イベントソーシング）**
   - コマンド側：注文確定コマンドを受け取り、ドメインモデルで処理。イベントソーシングで「注文確定イベント」をイベントストアに記録
   - クエリ側：注文一覧ページ用の非正規化データストアを別途持ち、高速に返す
   - 「注文確定イベント」を発行する

2. **在庫サービス（イベント駆動）**
   - 「注文確定イベント」をサブスクライブして在庫を引き当てる
   - 「在庫引き当て完了イベント」を発行する

3. **通知サービス（イベント駆動）**
   - 「在庫引き当て完了イベント」をサブスクライブして顧客にメール通知を送る

**特徴**：サービス間が直接呼び出しではなくイベントで連携するため、各サービスが疎結合に保たれる。注文サービスはイベントソーシングにより変更履歴を完全に保持できる。

## C-15　ドメイン層がインフラ層に依存してしまう問題の解決（応用）

**問い**

以下のコードは、ドメイン層のサービスが直接DBアクセスクラスに依存しています。DIPを適用して問題を解決してください。

```csharp
// ドメイン層
public class OrderService {
    private SqlOrderRepository _repository = new SqlOrderRepository(); // 直接依存
    
    public void ConfirmOrder(Guid orderId) {
        var order = _repository.FindById(orderId);
        order.Confirm();
        _repository.Save(order);
    }
}
```

---

**模範解答**

**問題点**：`OrderService`（ドメイン層）が`SqlOrderRepository`（インフラ層）を直接インスタンス化しており、ドメイン層がSQL実装に依存している。

**DIPを適用した解決策**

```csharp
// ドメイン層にインターフェイスを定義
public interface IOrderRepository {
    Order FindById(Guid orderId);
    void Save(Order order);
}

// ドメイン層のサービスはインターフェイスに依存
public class OrderService {
    private readonly IOrderRepository _repository;

    public OrderService(IOrderRepository repository) { // コンストラクタ注入
        _repository = repository;
    }

    public void ConfirmOrder(Guid orderId) {
        var order = _repository.FindById(orderId);
        order.Confirm();
        _repository.Save(order);
    }
}

// インフラストラクチャ層でインターフェイスを実装
public class SqlOrderRepository : IOrderRepository { ... }
```

`OrderService`はインターフェイスにのみ依存し、SQL実装を知らない。テスト時はモックの`IOrderRepository`を注入でき、本番ではDIコンテナが`SqlOrderRepository`を注入する。

## C-16　イベントソーシングを採用すべき場面と採用コストのトレードオフ（応用）

**問い**

以下の2つのシステムについて、イベントソーシングを採用すべきかどうかを判断し、理由を説明してください。

- **システムA**：銀行の入出金管理システム。すべての取引履歴の完全な追跡と監査が法律で義務づけられている
- **システムB**：社内の会議室予約システム。現在の予約状況が参照できれば十分

---

**模範解答**

**システムA（入出金管理）：採用を推奨**

- 法的監査の要求に対してイベントソーシングは直接応える（すべての状態変化をイベントとして記録）
- 取引の完全な追跡・任意時点の状態再現が容易になる
- 採用コスト（設計・実装の複雑さ）をビジネス要件が正当化する

**システムB（会議室予約）：採用不要**

- 現在の予約状況が参照できれば要件を満たせる（履歴の完全再現は不要）
- イベントソーシングの実装・運用コストがメリットを上回る（オーバーエンジニアリング）
- 通常のCRUD設計で十分

**判断軸**：監査・コンプライアンス要件の有無、過去の再現の必要性、ビジネス要件とコストのバランス。

## C-17　CQRSと結果整合性を組み合わせた読取モデルの設計（応用）

**問い**

CQRSを採用したシステムで、コマンド実行後に読み取りモデルが更新される仕組みを説明してください。また、読み取りモデルが更新される前にクエリが来た場合の対処方針も述べてください。

---

**模範解答**

**読み取りモデルの更新仕組み**

1. UIからコマンド（例：注文確定）を受け付ける
2. コマンドハンドラがドメインモデルを呼び出し、状態変化を起こす
3. ドメインイベント（例：注文確定イベント）を発行する
4. サブスクライバがイベントを非同期で受信する
5. サブスクライバが読み取りモデル用の非正規化データストアを更新する

この流れにより、書き込みと読み取りの更新は非同期（結果整合性）となる。

**コマンド直後にクエリが来た場合の対処方針**

- **UIで楽観的更新**：コマンドが成功したとみなしてUI側で先に表示を更新し、サーバーの同期を待つ
- **短時間のリトライ**：「まだ更新中です」と通知し、一定時間後に再取得するよう誘導する
- **許容度を設計として割り切る**：わずかな遅延は許容できるビジネス要件であると明確に定義し、UIに反映する

厳密な整合性が必要な部分（例：在庫残数の確認）にはCQRSを部分的に適用しない選択も有効。
