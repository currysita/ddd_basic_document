# カテゴリA 学習メモ：DDDの基本思想とユビキタス言語

---

## A-01　DDDの3原則

### 第1周（2026-06-13）△
- 「コア」ドメインという言葉を落とした → DDDはすべてに力を注ぐのではなく、競争力の源泉となる核心に絞るという点が本質
- 「モデルを探究する」という目的意識が抜けていた → 協力すること自体が目的ではなく、モデルを深めることが目的
- 「明示的な境界づけられたコンテキスト」と言えなかった → BCという概念が3原則に直接含まれている

## 質疑応答メモ

### ドメイン知識とは何か（A-01, A-02 周辺）

**問い**：印刷工場のシステムを作る場合、印刷の知識があっても、金融法や電波通信事業法を知らないとせっかく作ったものを公開できないことがある。ドメイン知識とはその業務知識だけなのか？

**考えたこと**：
- ドメインとは「ソフトウェアが解くべき問題空間全体」であり、法規制・コンプライアンス・業界標準もドメインの一部
- 法規制を知らずに作ったシステムが公開できないなら、それはドメインの失敗
- 法規制は「外部要因」ではなく「ドメインの不変条件（インバリアント）」として設計に織り込むべきもの

**ドメインエキスパートは複数人でよい**：
```
印刷工場システムのドメインエキスパート構成例：
├── 工場長         → 生産工程・品質管理
├── 営業部長       → 受注・価格設定・顧客ルール
├── 法務担当者     → 関連法規・コンプライアンス
└── 経理担当者     → 請求・与信・税務
```

### ドメインエキスパートは育てるもの（A-02 周辺）

**問い**：ほとんどの人は最初はドメインエキスパートではないので、どのようなドメインエキスパートを育てるかになるのではないか？

**考えたこと**：
- Evansの理想はエキスパートがすでに存在する前提だが、現実は「互いに手探り」が多い
- DDDのユビキタス言語を作るプロセス自体が、ドメインエキスパートを育てる場として機能する
- 「エキスパートを見つける」のではなく「協働の中でエキスパートが育つ」のがDDDの本質的な価値の一つ

```
設計の議論
  ↓
「この用語はどういう意味？」と問われる
  ↓
業務担当者が言語化を迫られる
  ↓
曖昧だった業務ルールが明確になる
  ↓
担当者自身がドメインエキスパートになっていく
```

**リスク**：誰も「本当のルール」を知らないまま設計が進む危険がある。だからテストコードをドメインエキスパート候補が読んで確認できる形にする（A-07のテストファーストの考え方）が重要。

### コンテキストとは何か（A-01 の3原則「ユビキタス言語を語る」周辺）

**問い**：コンテキストとはどういう意味か？明示的に境界づけられていることがそんなに重要なのか？

**考えたこと**：
- コンテキスト＝「言葉が意味を持つ文脈・場」
- 同じ「顧客」という言葉でも、営業・請求・サポートでは指す対象もルールも異なる
- 境界が曖昧だと「同じ言葉なのに違う意味」という混乱がシステム全体に静かに広がる
- 明示的な境界は、その混乱を封じ込める壁

**境界が曖昧な場合の問題例**：
```
営業システムと請求システムが同じ「顧客テーブル」を共有
  ↓
請求側が「契約ステータス」フィールドを追加
  ↓
営業側の見込み客データが壊れる
  ↓
「なんか顧客データがおかしい」→ 原因特定に数日
```

---

### コンテキストの境界をどう見つけるか

**問い**：文脈によって意味が異なる単語があれば、それはコンテキストが分かれている可能性があるのか？

**考えたこと**：
- はい、それがBCを発見するための強力な手がかりになる
- Evansも「ユビキタス言語の乱れがコンテキストの境界を示す」という考え方を示している

**実践的な問いかけ例**：
```
「この『注文』は、キャンセルできる？」
  → 営業: できる（商談中だから）
  → 倉庫: できない（出荷済みだから）
→ 同じ「注文」でも状態・ルールが違う → コンテキストが分かれているサイン
```

**注意点**：「言葉の意味が違う」＝「必ずコンテキストを分ける」ではない
- 意味の違いがビジネスルールの違いを生んでいる → コンテキストを分ける
- 単なる呼び方の違いで、扱いは同じ → 用語を統一してそのまま

「同じ言葉で話が噛み合わない場面」を意図的に探すことが、BCを設計する出発点になる。
