## E1-01　集約の定義（知識確認）

**問い**

DDDにおける「集約」とは何か、その目的と特徴を説明してください。

---

**模範解答**

**集約（Aggregate）**：関連するオブジェクト群（エンティティ・値オブジェクト）をトランザクション整合性の観点でひとまとめにしたグループ。集約内のオブジェクトは常に一貫した状態を保つ。

**目的**：トランザクションの境界を明確に定義し、集約内のビジネスルール（業務ルールの整合性・一意性・妥当性・制約）を一貫して保護する。

**特徴**：
- 外部からは集約ルート（ルートエンティティ）経由でのみ操作できる
- 集約の外部にいるオブジェクトは集約ルートの識別子しか持てない
- 1つのトランザクションで変更できるのは原則1つの集約のみ

---

## E1-02　集約ルートの定義と役割（知識確認）

**問い**

集約ルート（ルートエンティティ）とは何か、その役割を説明してください。

---

**模範解答**

**集約ルート（Aggregate Root）**：集約の代表となるエンティティ。外部からの操作は必ず集約ルートを通じて行われ、集約内部オブジェクトへの直接アクセスは禁止される。

**役割**：
- 集約へのアクセスの唯一の入口として機能する
- 集約内部の整合性を維持する責任を持つ
- 外部から集約内部への直接参照を禁止することで不正な状態変化を防ぐ
- 集約の識別子（一意な識別子）を持つ

---

## E1-03　トランザクション整合性の定義（知識確認）

**問い**

集約における「トランザクション整合性」とは何か、集約の境界との関係を説明してください。

---

**模範解答**

**トランザクション整合性（Transaction Consistency）**：1つのトランザクション内で集約の状態が一貫していることを保証すること。

**集約の境界との関係**：集約の境界はトランザクションの境界と一致する。1つのトランザクションで変更できるのは原則1つの集約のみ。これにより：
- 集約内部のビジネスルールが確実に守られる
- トランザクションのスコープが明確になり、競合が起きにくくなる
- 複数集約の同時変更はドメインイベント＋結果整合性で対処する

---

## E1-04　結果整合性の定義と集約との関係（知識確認）

**問い**

「結果整合性（Eventual Consistency）」とは何か、複数集約間の整合性にどう適用するかを説明してください。

---

**模範解答**

**結果整合性**：異なる集約間の整合性は、即時ではなく最終的（結果的）に整合するという方針。

**複数集約間への適用**：

1つのトランザクションで変更できる集約は1つという原則があるため、複数の集約にまたがる操作は即時整合が取れない。

- 集約Aの変更完了時にドメインイベントを発行する
- 非同期でサブスクライバがイベントを受信する
- サブスクライバが集約Bを変更する（別トランザクション）

これにより即時の整合ではなく最終的な整合を保つ。各集約の自律性を守りながら、システム全体の整合性を実現する。

## E1-05　デメテルの法則の定義（知識確認）

**問い**

デメテルの法則とは何か、集約設計においてなぜ重要なのかを説明してください。

---

**模範解答**

**デメテルの法則（Law of Demeter）**：オブジェクトは直接知っているオブジェクトのメソッドのみを呼び出すべきという原則。「友人にのみ話しかけ、見知らぬ人には話しかけない」。

**集約設計での重要性**：

連鎖参照（`backlogItem.getTask().getEstimate().setHours(5)`）を禁止することで：
- 集約内部の構造が外部に漏れない（カプセル化の維持）
- 集約の内部変更が外部コードに影響しない（疎結合）
- 集約ルートを経由する設計が自然に維持される

## E1-06　「命じる、たずねるな」の指針（知識確認）

**問い**

「命じる、たずねるな（Tell-Don't-Ask）」とは何か、集約設計における具体例とともに説明してください。

---

**模範解答**

**Tell-Don't-Ask**：オブジェクトに状態を問い合わせてから呼び出し側で判断するのではなく、オブジェクト自身に行動を命じるべきという設計指針。

**悪い例（Ask）**：

```java
// 外部で状態を確認してから判断する
if (backlogItem.getStatus() == Status.COMMITTED) {
    backlogItem.setStatus(Status.DONE);
}
```

**良い例（Tell）**：

```java
// オブジェクトに行動を命じる（判断はオブジェクト内部で行う）
backlogItem.markAsDone();
// BacklogItem.markAsDone() 内でステータスチェックと変更を行う
```

カプセル化が高まり、ビジネスルールがオブジェクト内に留まる。

## E1-07　巨大な集約の問題点（知識確認）

**問い**

集約を巨大にしすぎた場合に生じる3つの問題点を説明してください。

---

**模範解答**

1. **モデリングエラーが入り込む**：集約が大きくなるほど、本来別のトランザクション境界にすべきオブジェクトが混在し、ビジネスルールの責務が不明確になる

2. **トランザクション競合が発生する**：多くのオブジェクトが1つの集約に属するため、異なるユーザーが同時に集約の異なる部分を変更しようとしても、集約全体がロックされて競合・失敗が起きる

3. **メモリ使用量が高くなる**：集約全体をメモリに展開するため、大きな集約は多量のメモリを消費し、パフォーマンスに悪影響を与える

## E1-08　集約内部のオブジェクトを値オブジェクトにする理由（知識確認）

**問い**

集約内のオブジェクトを可能な限り値オブジェクトとして設計することが推奨される理由を説明してください。

---

**模範解答**

- **テストしやすい**：値オブジェクトは不変で副作用がないため、単体テストが容易
- **副作用がない**：値オブジェクトはメソッドが自身の状態を変えないため、集約内での予期しない状態変化が起きない
- **シンプル**：識別子管理（ID）が不要なため、集約内の構造がシンプルになる
- **並行安全**：不変なので複数スレッドから参照しても安全

これにより集約内部のエンティティ数を最小化し、整合性の管理が容易になる。

---

## E1-09　巨大な集約と小さな集約の比較（理解確認）

**問い**

「巨大な集約」と「小さな集約（複数に分割）」を設計難易度・トランザクション整合性・スケーラビリティ・競合頻度の4軸で比較してください。

---

**模範解答**

| 比較軸 | 巨大な集約 | 小さな集約 |
|---|---|---|
| **設計難易度** | 難（どこまで含めるか判断が難しい） | 易（各集約の責務が明確） |
| **トランザクション整合性** | 集約内では高いが管理が困難 | 集約内は保証、集約間は結果整合性 |
| **スケーラビリティ** | 低（大きな集約をロックするため） | 高（小さな単位で並行処理できる） |
| **競合頻度** | 高（多くの操作が1つの集約に集中する） | 低（操作が複数の小さな集約に分散する） |

## E1-10　集約の設計ルール1：小さな集約を設計する（理解確認）

**問い**

「集約は小さく設計する」というルールを守ることでパフォーマンスやスケーラビリティが改善される理由を説明してください。また、集約をどの程度の大きさにすべきかの目安も述べてください。

---

**模範解答**

**パフォーマンス・スケーラビリティが改善される理由**：

小さな集約はロックする範囲が狭いため、複数のユーザーが同時に操作しても別々の集約をロックするだけで競合が起きない。大きな集約は多くのユーザーの操作が同一の集約ロックを奪い合い、スループットが低下する。

また、小さな集約はメモリ展開量が少なく、読み込み・保存のコストが低い。

**大きさの目安**：

1つの集約ルート（エンティティ）と、それに直属する少数のオブジェクト（値オブジェクトや関連エンティティ）のみにする。「1つのトランザクションで整合性を保つ必要がある最小単位」が集約の境界の目安。

## E1-11　集約の設計ルール2：他集約は識別子で参照する（理解確認）

**問い**

「他の集約へは直接オブジェクト参照でなく、一意な識別子で参照する」というルールがある理由を説明してください。

---

**模範解答**

**直接参照の問題**：

他集約を直接オブジェクト参照すると、その集約全体がメモリに展開される。コードが他集約のメソッドを呼び出しやすくなり、1つのトランザクションで複数の集約を変更してしまいがちになる。

**識別子参照の利点**：

- 必要なときだけリポジトリ経由で他集約を取得するため、不要なメモリ展開が起きない
- トランザクション境界が明確になる（識別子で参照する限り、うっかり他集約を変更できない）
- 集約間の結合度が下がり、独立して変更できる

```csharp
// NG：直接参照
public ProductId ProductId { get; } // 識別子（値オブジェクト）← OK
public Product Product { get; }     // 直接参照 ← NG
```

## E1-12　1つのトランザクションで1つの集約という原則の理由（理解確認）

**問い**

「1つのトランザクションで変更できるのは原則1つの集約のみ」という原則がある理由を説明してください。

---

**模範解答**

**理由**：

1. **整合性の保証**：集約の境界はトランザクションの境界と一致する。1つの集約は1つのトランザクションで整合性が保証されるように設計されている。複数の集約をまたぐと、どこかで失敗した場合に部分的な整合性しか取れない

2. **競合の防止**：複数の集約を同時にロックすると、デッドロックや競合の原因になる

3. **スケーラビリティ**：単一集約の変更に限定することで、ロックの範囲が最小化され並行処理が容易になる

複数集約にまたがる変更が必要な場合は、ドメインイベント＋結果整合性で対処する。

---

## E1-13　集約の例外パターン：UIの利便性を優先する場合（理解確認）

**問い**

「UIの利便性を優先して複数集約を一度に表示・操作する」という例外パターンが認められる場合と、その際の注意点を説明してください。

---

**模範解答**

**認められる場合**：

複数の集約を一度に取得して画面に表示する必要がある場合（例：ダッシュボードで複数の集約の情報を一画面に表示する）。読み取り専用の非正規化データを返すクエリ専用のモデルを用意する方法が適切。

**注意点**：

- 表示（読み取り）のために複数集約を取得することは問題ない
- 1回の操作で複数の集約を「変更」することは避ける
- 変更が必要な場合は、それぞれ別のトランザクションで処理するか、ドメインイベントで連鎖させる

## E1-14　集約の例外パターン：パフォーマンス改善のための直接参照（理解確認）

**問い**

「クエリのパフォーマンス改善のために集約の境界をまたいだ直接参照が有効な場合がある」という例外パターンを説明してください。その際のリスクも含めて述べてください。

---

**模範解答**

**有効なケース**：

読み取り専用のクエリで、複数集約にまたがる複雑なJOINやデータ取得が必要な場合。非正規化データストアで直接参照するケースや、レポート生成のためのクエリサービスが例として挙げられる。

**リスク**：

- 読み取りのために境界をまたいだとしても、変更にまで使ってしまうリスクがある
- 直接参照が増えると集約間の結合度が上がり、設計の一貫性が崩れる

**対処方針**：読み取りと書き込みのモデルを明確に分離し、読み取り用の参照はクエリ専用のモデルに限定する。

## E1-15　BacklogItemの集約ルート設計の分析（理解確認）

**問い**

以下のBacklogItemクラスのコードを分析し、集約設計のどのルールが守られているかを説明してください。

```csharp
public class BacklogItem : EntityWithCompositeId {
    public BacklogItemId BacklogItemId { get; private set; }
    public TenantId TenantId { get; private set; }
    public ProductId ProductId { get; private set; }
    public BacklogItemType BacklogItemType { get; private set; }
    public BacklogItemStatus BacklogItemStatus { get; private set; }
    public string Category { get; private set; }
}
```

---

**模範解答**

**守られているルール**：

1. **集約ルートがエンティティ**：`BacklogItem`クラス自体がルートエンティティ（`EntityWithCompositeId`を継承）

2. **他集約は識別子で参照**：`TenantId`、`ProductId`という値オブジェクト（識別子）を通じて他集約を参照している。`Tenant`オブジェクトや`Product`オブジェクト自体は保持していない

3. **setterがprivate**：`private set`により外部から直接プロパティを変更できない。変更は集約ルートのメソッドを通じてのみ行う（カプセル化）

4. **値オブジェクトの使用**：`BacklogItemType`・`BacklogItemStatus`をいずれも値オブジェクトとして保持している

## E1-16　集約境界の見つけ方（理解確認）

**問い**

- **Yes**：同じ集約に含める
- **No**：別の集約とし、必要なら結果整合性で対処する

**判断の観点**：

| 問い | 同じ集約 | 別の集約 |

| 片方の変更が他方を無効にするか | ✓ | |
| 別々のユーザーが独立して操作するか | | ✓ |
| 結果整合性（後から整合）で十分か | | ✓ |

## E1-17　集約外から集約内部のエンティティに直接アクセスしてはいけない理由（理解確認）

**問い**

「集約の外から集約内部のエンティティに直接アクセスしてはいけない」という制約がある理由を、ビジネスルールの観点から説明してください。

---

**模範解答**

集約のビジネスルールは「集約全体の整合性」を前提として設計されている。内部のエンティティへ直接アクセスして変更すると、集約ルートが持つバリデーションや整合性チェックをスキップしてしまう。

例：`Order`集約には「商品数が1件以上必要」というルールがある。外部から`Order.items`コレクションに直接アイテムを追加できてしまうと、このルールが無視される可能性がある。

集約ルート（`Order`）を通じて`addItem()`メソッドを呼び出すことで、常にビジネスルールが適用される。

## E1-18　デメテルの法則に違反したコードの問題（理解確認）

**問い**

以下のコードはデメテルの法則に違反しています。問題点を指摘し、修正方針を説明してください。

```java
// 外部コード
int hours = backlogItem.getTask().getEstimate().getHours();
backlogItem.getTask().getEstimate().setHours(hours + 8);
```

---

**模範解答**

**問題点**：

- 外部コードが`BacklogItem → Task → Estimate`という内部構造を知っている（カプセル化の破壊）
- `Task`や`Estimate`が内部に存在することが外部にさらけ出されている
- 集約ルートの`BacklogItem`を通じずに内部オブジェクトを直接変更している

**修正方針（Tell-Don't-Ask）**：

```java
// 集約ルートに行動を命じる
backlogItem.addHoursToTaskEstimate(8);

// BacklogItem内部でTaskとEstimateを操作する
public void addHoursToTaskEstimate(int hours) {
    this.task.addToEstimate(hours); // 内部的に処理
}
```

外部コードは集約ルートのメソッドのみを呼ぶ。内部構造の変更が外部に影響しない。

## E1-19　「命じる、たずねるな」の適用（理解確認）

**問い**

以下のコードに「命じる、たずねるな」原則を適用して改善してください。

```java
// 外部コード（アプリケーションサービスなど）
Order order = orderRepository.findById(orderId);
if (order.getStatus() == OrderStatus.DRAFT && !order.getItems().isEmpty()) {
    order.setStatus(OrderStatus.CONFIRMED);
}
```

---

**模範解答**

**問題点**：外部コードが`Order`の状態を問い合わせて（ask）判断し、setterで直接状態を変更している。ビジネスルールが`Order`の外に漏れている。

**改善後**：

```java
// 外部コード
Order order = orderRepository.findById(orderId);
order.confirm(); // Orderに行動を命じる（tell）

// Order クラス内
public void confirm() {
    if (this.status != OrderStatus.DRAFT) {
        throw new DomainException("下書き状態の注文のみ確定できます");
    }
    if (this.items.isEmpty()) {
        throw new DomainException("商品が1件以上必要です");
    }
    this.status = OrderStatus.CONFIRMED;
}
```

判断とルールが`Order`内に集まり、外部コードは「何をするか」だけを伝える。

## E1-20　集約の取得方法：リポジトリ経由とサービス経由の違い（理解確認）

**問い**

集約を取得する2つの方式（集約内からリポジトリを呼び出す方式と、集約の外からあらかじめ取得する方式）を比較し、それぞれの適用場面を説明してください。

---

**模範解答**

**方式1：集約内からリポジトリを呼び出す**

```java
// 集約内でリポジトリを使う（ドメインサービスを経由して）
// IDDD本ではサービス経由として推奨
```

- 集約の処理に必要な関連データを遅延取得できる
- リポジトリへの依存がドメイン層に入るため注意が必要

BacklogItem item = backlogItemRepository.findById(itemId);

```

- 集約がリポジトリに依存しない（テストしやすい）
- アプリケーション層で取得責任を持つ
- 一般的に推奨されるアプローチ

## E1-21　集約の境界を設計する（応用）

**問い**

ECサイトの「注文（Order）」に関して、以下のオブジェクトをどのように集約に分けるかを設計してください。

- 注文（Order）
- 注文明細（OrderItem）
- 顧客（Customer）
- 配送先住所（ShippingAddress）
- 支払い方法（PaymentMethod）

---

## E1-28　集約間をまたぐトランザクションの失敗パターンと対策（応用）

**問い**

「注文確定と同時に在庫を減らす」という処理で、注文集約と在庫集約を1つのトランザクションで変更しようとする設計の問題点と対策を説明してください。

---

**模範解答**

**問題点**：

1. **デッドロックリスク**：2つの集約を同時にロックすると、別のトランザクションが逆順にロックしようとした場合にデッドロックが起きる
2. **スケーラビリティの低下**：2つの集約をまたぐロックは範囲が広く、並行処理が制限される
3. **DBが別の場合は不可能**：集約が別のDBやサービスにある場合、そもそも同一トランザクションが使えない

**対策（結果整合性）**：

1. 注文集約をトランザクションで確定 → OrderConfirmedイベント発行
2. 在庫集約は別トランザクションでイベントを受信して在庫を減らす
3. 在庫不足の場合はInventoryShortageイベントを発行
4. 注文集約がInventoryShortageを受信して注文をキャンセルに変更
```

2段階の整合性により、各集約の自律性を保ちながら最終的な整合性を実現する。

## E1-33　集約の不変条件（Invariant）の設計（応用）

- 注文合計金額は0円より大きいこと
- キャンセル済みの注文には商品を追加できないこと

---

**模範解答**

```java
public class Order {
    private List<OrderItem> items;
    private OrderStatus status;
    
    // 不変条件1・2：コンストラクタで保証
    public Order(OrderId id, CustomerId customerId, List<OrderItem> items) {
        if (items == null || items.isEmpty()) {
            throw new DomainException("注文には商品が1件以上必要です");
        }
        if (items.stream().mapToInt(i -> i.amount()).sum() <= 0) {
            throw new DomainException("合計金額は0円より大きい必要があります");
        }
        this.items = new ArrayList<>(items);
        this.status = OrderStatus.DRAFT;
    }
    
    // 不変条件3：変更操作のたびにチェック
    public void addItem(OrderItem item) {
        if (this.status == OrderStatus.CANCELLED) {
            throw new DomainException("キャンセル済みの注文には商品を追加できません");
        }
        this.items.add(item);
        // 追加後も合計金額チェック
        assertValidTotal();
    }
    
    private void assertValidTotal() {
        if (this.items.stream().mapToInt(i -> i.amount()).sum() <= 0) {
            throw new DomainException("合計金額は0円より大きい必要があります");
        }
    }
}
```

不変条件はコンストラクタと各変更メソッドに分散して配置し、集約が常に有効な状態を保つ。

public class BacklogItemId {

}

## E1-35　集約の状態遷移を設計する（応用）

**問い**

---

**模範解答**

**状態遷移図**：

```
Draft → Confirmed → Shipped → Delivered
  ↓          ↓
Cancelled  Cancelled（出荷前のみ）
```

**実装**：

```java
public class Order {
    public void confirm() {
        if (status != DRAFT) throw new DomainException("下書き状態のみ確定できます");
        if (items.isEmpty()) throw new DomainException("商品が必要です");
        this.status = CONFIRMED;
    }
    
    public void ship() {
        if (status != CONFIRMED) throw new DomainException("確定済みのみ出荷できます");
        this.status = SHIPPED;
    }
    
    public void deliver() {
        if (status != SHIPPED) throw new DomainException("出荷済みのみ配達完了にできます");
        this.status = DELIVERED;
    }
    
    public void cancel() {
        if (status == SHIPPED || status == DELIVERED) {
            throw new DomainException("出荷後はキャンセルできません");
        }
        this.status = CANCELLED;
    }
}
```

各メソッドで有効な遷移かを検証し、不正な状態遷移をドメイン例外で防ぐ。

## E1-36　集約のロック戦略と競合対策（応用）

**問い**

ECサイトで複数ユーザーが同時に同じ注文集約を変更しようとした場合（例：注文明細の追加）の競合対策として、楽観的ロックと悲観的ロックの違いと、DDDにおける推奨アプローチを説明してください。

---

**模範解答**

**楽観的ロック（Optimistic Locking）**：

- 変更前にバージョン番号（version）を取得し、保存時に「バージョンが変わっていないこと」を確認する
- 変更されていた場合は`OptimisticLockException`を返し、ユーザーに再試行を促す
- ロックを保持しないため、競合が少ない場合に高いスループットを発揮する

```java
public class Order {
    private int version; // バージョン番号
    // JPA/@Version などで自動管理
}
```

**悲観的ロック（Pessimistic Locking）**：

- 取得時にDBレコードをロックし、他の変更を完全に排除する
- デッドロックリスクがあり、スループットが低下する

**DDDにおける推奨**：

楽観的ロックを推奨。集約を小さく設計することで競合が減り、楽観的ロックで十分に対処できるケースが多い。競合が多い場合は集約設計を見直す（集約が大きすぎる可能性）。

## E1-37　集約とファクトリの組み合わせ（応用）

**問い**

「新規プロジェクト（Project）集約」の生成が複雑で、デフォルトのマイルストーン（3か月後の「リリース」）を自動生成する必要があります。この生成ロジックをどこに実装すべきかを判断し、設計してください。

---

**模範解答**

**判断**：生成ロジックが複雑なため、コンストラクタではなくファクトリに実装する。

```java
// ファクトリ
public class ProjectFactory {
    public Project createNewProject(
            ProjectId projectId, String name, LocalDate startDate) {
        
        // デフォルトマイルストーンを生成
        LocalDate defaultReleaseDate = startDate.plusMonths(3);
        Milestone defaultMilestone = new Milestone(
            MilestoneId.generate(),
            "リリース",
            defaultReleaseDate
        );
        
        return new Project(projectId, name, startDate, List.of(defaultMilestone));
    }
}

// アプリケーションサービス
public Project create(String name, LocalDate startDate) {
    ProjectId id = ProjectId.generate();
    Project project = projectFactory.createNewProject(id, name, startDate);
    projectRepository.save(project);
    return project;
}
```

ファクトリが生成の複雑さを隠蔽し、`Project`集約は生成後の操作に集中できる。

## E1-38　集約内部のエンティティとルートエンティティの責任分担（応用）

**問い**

「注文（Order）集約ルート」と「注文明細（OrderItem）集約内エンティティ」の責任をどのように分担するかを設計してください。外部からの操作方法も含めて説明してください。

---

**模範解答**

**責任分担**：

| | Order（集約ルート） | OrderItem（集約内エンティティ） |
|---|---|---|
| **責任** | 注文全体の整合性（商品数制限・合計金額・ステータス管理） | 個別明細の整合性（数量・金額の妥当性） |
| **外部からのアクセス** | 直接アクセス可能（集約ルート） | Order経由でのみアクセス |

**実装**：

```java
public class Order {
    private List<OrderItem> items; // privateコレクション
    
    public void addItem(ProductId productId, int qty, Money unitPrice) {
        // Orderレベルのルール
        if (items.size() >= 50) throw new DomainException("明細は50件まで");
        
        // OrderItemを生成して追加（外部からnewさせない）
        items.add(new OrderItem(productId, qty, unitPrice));
    }
    
    public void changeItemQuantity(ProductId productId, int newQty) {
        OrderItem item = findItem(productId);
        item.changeQuantity(newQty); // OrderItem自身のメソッドを呼ぶ
    }
}

class OrderItem { // パッケージprivateなど外部から生成不可
    void changeQuantity(int newQty) {
        if (newQty <= 0) throw new DomainException("数量は1以上必要");
        this.quantity = newQty;
    }
}
```

## E1-39　集約設計の全体的な評価と改善（応用）

**問い**

以下の集約設計を評価し、問題点をすべて指摘して改善案を提案してください。

```java
public class ProjectAggregate {
    private String projectName;
    private List<Task> tasks;        // 数百件になる可能性あり
    private List<Member> members;    // 全メンバー
    private List<Milestone> milestones;
    private Customer customer;       // 他集約を直接参照
    private List<Invoice> invoices;  // 請求書（別ドメイン？）
    
    public void addTask(Task task) {
        tasks.add(task); // バリデーションなし
    }
}
```

---

**模範解答**

**問題点**：

1. **集約が大きすぎる**：タスク・メンバー・マイルストーン・請求書すべてを1集約に含めている。数百件のタスクをメモリに展開し、同時操作で競合多発

2. **他集約の直接参照**：`Customer customer`は他集約。`CustomerId`識別子のみを保持すべき

3. **関心が混在**：`Invoice`（請求書）はプロジェクト管理ではなく請求ドメインの概念。別BCに分けるべき

4. **バリデーションなし**：`addTask()`に業務ルールチェックがなく、不正なタスクが追加できてしまう

**改善案**：

| 集約（または別BC） | 集約ルート | 内部 |
|---|---|---|
| Project集約 | Project | ProjectName, CustomerId（識別子）, Status |
| Task集約 | Task | TaskName, Estimate, Status, ProjectId（識別子） |
| ProjectMember集約 | ProjectMember | Role, ProjectId（識別子） |
| Milestone集約 | Milestone | MilestoneName, Date, ProjectId（識別子） |
| 請求BC（Invoice集約） | Invoice | …（別BCとして独立） |

各集約は`ProjectId`識別子で参照し合い、イベントで結果整合性を保つ。
