## D1-01　エンティティと値オブジェクトの違い（知識確認）

**問い**

エンティティと値オブジェクトを「同一性の基準」「可変性」「主な用途」の観点で比較してください。

---

**模範解答**

| 比較軸 | エンティティ | 値オブジェクト |
|---|---|---|
| **同一性の基準** | 識別子（ID）が同じなら同一 | 属性値がすべて等しければ等値 |
| **可変性** | ミュータブル（状態変化あり） | イミュータブル（不変） |
| **主な用途** | ライフサイクルを持つ概念（ユーザー・注文など） | 値・属性のまとめ（金額・住所・優先度など） |

## D1-02　一意な識別子の4つの生成パターン（知識確認）

**問い**

一意な識別子の4つの生成パターンを列挙し、各パターンのメリットとデメリットを説明してください。

---

**模範解答**

| # | 生成者 | メリット | デメリット |
|---|---|---|---|
| 1 | **ユーザーが入力** | 識別子生成の仕組みが不要 | 重複のないよう適切な識別子を登録するワークフロー・チェックが必要 |
| 2 | **アプリケーションが生成**（UUID） | 軽量に一意な値を生成できる | 人間には読みにくい文字列になるため利用者には非表示にする必要がある |
| 3 | **DB（永続化メカニズム）が生成**（連番） | DBのシーケンス機能で簡単に連番生成できる | DB依存が高く、早期生成時のパフォーマンス課題がある |
| 4 | **他のBCから割り当て** | 識別子生成の仕組みを他のコンテキストに委ねられる | 他コンテキストとの識別子取得・照合・割り当ての仕組みが必要 |

## D1-03　値オブジェクトの6つの特性（知識確認）

**問い**

値オブジェクトが持つ6つの特性を列挙し、各特性を説明してください。

---

**模範解答**

| # | 特性 | 説明 |
|---|---|---|
| 1 | **計測/定量化/説明** | ドメイン内の物事を計測・定量化・説明する（金額・重量・品名など） |
| 2 | **不変性** | 状態を変えることができない。セッターを持たない |
| 3 | **概念的な統一体** | 関連する属性を一つの概念としてまとめる（例：金額 = 数値 + 通貨単位） |
| 4 | **交換可能性** | 値を変えたい場合は新しいインスタンスに丸ごと置き換える |
| 5 | **等値性** | 属性値がすべて等しければ同じとみなす（IDではなく属性で比較） |
| 6 | **副作用のない振る舞い** | メソッドは自身の状態を変えず、新しい値オブジェクトを返す |

## D1-05　代理識別子とは何か（知識確認）

**問い**

代理識別子とは何か、なぜ存在するのか、ビジネス上の識別子（一意な識別子）とどう違うかを説明してください。

---

**模範解答**

**代理識別子（Surrogate Identity）**：O/Rマッパー（HibernateやEntity Frameworkなど）がデータベースとのマッピングのために使うシステム的な識別子（DBの主キーなど）。

**なぜ存在するのか**：O/RマッパーがDB上でレコードを特定するために必要。DB連番（AUTO INCREMENT）などがその典型。

**ビジネス上の識別子との違い**

| | ビジネス上の識別子（一意な識別子） | 代理識別子 |
|---|---|---|
| **目的** | ドメイン上でエンティティを一意に識別する | O/RマッパーのDB連携のために使う |
| **公開** | ドメインモデルで使用・公開する | 外部に公開しない（隠蔽する） |
| **例** | テナントID（UUID）、ユーザー名 | DBの自動採番（1, 2, 3...） |

## D1-06　エンティティの同一性判定になぜIDが重要なのか（理解確認）

**問い**

エンティティの同一性判定において、属性値ではなく識別子（ID）を使うことがなぜ重要なのかを説明してください。

---

**模範解答**

エンティティは時間の経過とともに属性値が変化する。例えば、ユーザーの名前が変わっても「同じユーザー」として扱う必要がある。属性値で同一性を判断すると、属性が変化した瞬間に「別のエンティティ」になってしまう。

IDで同一性を判断することで、属性値が変わってもIDが同じである限り同一のエンティティとして追跡できる。これがエンティティの「ライフサイクル（生成・変更・削除）」を正しくモデル化するための基盤になる。

---

## D1-07　代理識別子をレイヤスーパータイプで隠蔽する理由（理解確認）

**問い**

代理識別子をレイヤスーパータイプ（Entityの親クラス）に配置し、`protected`で隠蔽する理由を説明してください。

---

**模範解答**

代理識別子はO/Rマッパーのシステム的な都合で使われるものであり、ビジネス上の意味を持たない。外部（ドメインの呼び出し元）からアクセスできてしまうと、ビジネス上の識別子と混同されてしまう。

`protected`で隠蔽し、`Entity`の親クラスで管理することで：

- 外部コードがビジネス上の識別子（一意な識別子）のみを使うよう強制できる
- ドメインモデルの意図が明確になる
- O/Rマッパーへの依存をエンティティの実装詳細に封じ込められる

## D1-08　不変性がもたらす利点（理解確認）

**問い**

値オブジェクトに不変性（Immutability）があることでスレッドセーフ・キャッシュ可能・共有可能になる理由を説明してください。

---

**模範解答**

**スレッドセーフ**：状態が変わらないため、複数スレッドから同時にアクセスしても競合状態（race condition）が起きない。ロックが不要。

**キャッシュ可能**：状態が変わらないため、一度計算した値をキャッシュして使い回しても安全。キャッシュが古くなる心配がない。

**共有可能**：同じ値を複数の場所で参照しても、ある場所での変更が他の場所に影響しない。コピーせずに同一インスタンスを安全に共有できる。

---

## D1-09　「概念的な統一体」として値オブジェクトを設計する利点（理解確認）

**問い**

「金額」という概念を「数値（int）と通貨単位（string）の2つのフィールド」として持つのではなく、`Money`値オブジェクトとして設計することの利点を説明してください。

---

**模範解答**

**バラバラのフィールドの問題**：「数値だけ」「通貨単位だけ」が渡されるミスを防げない。2つの値が「常に一緒に扱われる」というルールがコードに表現されない。

**`Money`値オブジェクトの利点**

- 金額に関するバリデーション・ルール（例：マイナス不可、通貨単位の検証）を`Money`クラスに集約できる
- `add(Money other)`など金額に意味のある振る舞いをメソッドとして持てる
- コンパイラが「金額」と「ただの数値」の混在を型でチェックしてくれる
- ユビキタス言語（「金額」）をコードで直接表現できる

## D1-10　値オブジェクトの等値性とエンティティの同一性の違い（理解確認）

**問い**

値オブジェクトの「等値性」とエンティティの「同一性」の違いを、コードの観点から説明してください。

---

**模範解答**

**エンティティの同一性**

```java
User a = new User(tenantId, "alice");
User b = new User(tenantId, "alice");
// IDが同じ → 同一エンティティ
a.equals(b); // tenantId と username が一致すれば true
```

IDで比較するため、属性が後で変わっても「同じユーザー」として追跡できる。

**値オブジェクトの等値性**

```java
Money a = new Money(100, "USD");
Money b = new Money(100, "USD");
// 属性値がすべて同じ → 等値
a.equals(b); // true（同じ値）
```

属性値で比較するため、別のインスタンスであっても値が同じなら等しいとみなす。識別子を持たない。

## D1-11　値オブジェクトの交換可能性とは何か（理解確認）

**問い**

値オブジェクトの交換可能性とは何か、そして不変性とどのように関係するかを説明してください。

---

**模範解答**

**交換可能性（Replaceability）**：値を変えたい場合にセッターで内部状態を書き換えるのではなく、新しいオブジェクトに丸ごと置き換える性質。

```java
// NG：セッターで状態を変える（不変性の違反）
money.setAmount(200);

// OK：新しいオブジェクトに置き換える（交換可能性）
money = new Money(200, "USD");
```

**不変性との関係**：値オブジェクトが不変（Immutable）だからこそ、状態を変える手段がなく、変えたければ新しいインスタンスを作るしかない。交換可能性は不変性の帰結として自然に生まれる設計パターン。

## D1-12　識別子生成のタイミング（早期生成 vs 遅延生成）の使い分け（理解確認）

**問い**

識別子の早期生成と遅延生成をそれぞれ説明し、どのような基準で使い分けるかを説明してください。

---

**模範解答**

| | 早期生成 | 遅延生成 |
|---|---|---|
| **タイミング** | オブジェクト生成時（new時）にIDを確定 | DB永続化時や後処理でIDを割り当て |
| **メリット** | 常にIDが存在するため整合性が高い | ID生成コストが高い場合のパフォーマンス問題を避けられる |
| **デメリット** | ID生成コストが高い場合のパフォーマンス問題 | ID未確定の状態でオブジェクトが存在する期間がある |

**使い分け**：UUIDのような軽量な生成コストのものは早期生成を推奨（オブジェクトが常にIDを持つため整合性が高い）。DBの連番（シーケンス）はDB接続が必要なため遅延生成になることが多い。

## D1-13　副作用のない振る舞いを値オブジェクトで実現する方法（理解確認）

**問い**

値オブジェクトの「副作用のない振る舞い」とは何か、`withメソッドパターン`を使った実装例で説明してください。

---

**模範解答**

**副作用のない振る舞い**：メソッドが自身の内部状態を変えず、計算結果を新しい値オブジェクトとして返す。純粋関数と同様の特性。

**withメソッドパターンの実装例**

```csharp
// NG：セッターで状態を変える（副作用あり）
priority.setRating(newRatings);

// OK：withメソッドで新しいインスタンスを返す（副作用なし）
BusinessPriority newPriority = priority.withRating(newRatings);
// priority は変わらない、newPriority が新しい値
```

`withRating`メソッドは新しい`BusinessPriority`インスタンスを返し、元の`priority`を変更しない。これにより呼び出し元での意図しない副作用が起きない。

## D1-14　標準型（タイプコード）をenumで実装する場合とクラスで実装する場合の違い（理解確認）

**問い**

区分・種類を表す「標準型（タイプコード）」をenumとクラス（値オブジェクト）で実装する場合の違いと、クラスで実装するメリットを説明してください。

---

**模範解答**

**enumの制限**：多くの言語でenumはドメイン固有の振る舞いを持てない（例：`isMember()`メソッドを持たせにくい）。振る舞いが増えると外部にロジックが散在する。

**クラス（値オブジェクト）のメリット**

```csharp
// クラスで実装した場合
public class GroupMemberType {
    public static readonly GroupMemberType Member = new GroupMemberType("Member");
    public static readonly GroupMemberType Moderator = new GroupMemberType("Moderator");
    
    public bool IsMember => _memberType.Equals("Member");
    public bool IsModerator => _memberType.Equals("Moderator");
    // ドメイン固有の振る舞いを追加できる
}
```

- ドメイン固有の振る舞いをクラスに持たせられる
- コンストラクタを`private`にすることで定義済みの値のみを使用させられる
- 将来の振る舞い追加に対して柔軟

## D1-15　ドメイン要件に基づいてエンティティを発見し設計する（応用）

**問い**

オンライン図書館システムの要件として「本の貸し出しを管理する。1冊の本には複数の貸し出し履歴がある。同じ本でも別々の利用者に貸し出した履歴は区別する必要がある」とあります。この要件からエンティティを発見し、識別子を設計してください。

---

**模範解答**

**エンティティの発見**

| エンティティ候補 | 理由 |
|---|---|
| **本（Book）** | 蔵書番号で同一の本を追跡する必要がある。属性（タイトル・著者）が変わっても「同じ本」として管理 |
| **貸し出し（Loan）** | 「誰に・いつ・何を」という組み合わせで区別する必要がある。履歴として追跡するライフサイクルを持つ |
| **利用者（Member）** | 会員番号で同一の利用者を追跡する必要がある |

**識別子の設計**

| エンティティ | 識別子 | 生成パターン |
|---|---|---|
| Book | ISBN（または蔵書番号） | ユーザー入力 or アプリ生成 |
| Loan | LoanId（UUID） | アプリケーションが生成 |
| Member | MemberId（UUID） | アプリケーションが生成 |

**Loan**のIDは「本ID + 利用者ID + 貸出日」のような複合キーも考えられるが、貸し出しに固有のUUIDを持たせることで追跡が容易になる。

## D1-16　ドメイン知識に基づいて値オブジェクトを発見し設計する（応用）

**問い**

物流システムの要件「配送先には住所（国・都道府県・市区町村・番地）と受取人名が含まれる」から値オブジェクトを発見し、設計してください。

---

**模範解答**

**値オブジェクトの発見**

| 候補 | エンティティ or 値オブジェクト | 理由 |
|---|---|---|
| **住所（Address）** | 値オブジェクト | 「東京都渋谷区」は属性値が等しければ同じ住所。ライフサイクルを持たない |
| **受取人名（RecipientName）** | 値オブジェクト | 名前は属性値で等値を判断する。変わったら新しい値として置き換える |
| **配送先（ShippingDestination）** | 値オブジェクト | `Address`と`RecipientName`をひとまとめにした概念的統一体 |

**設計**

```csharp
public class ShippingDestination {
    public Address Address { get; }
    public RecipientName RecipientName { get; }
    
    // コンストラクタのみで値をセット（不変）
    public ShippingDestination(Address address, RecipientName recipientName) {
        Address = address;
        RecipientName = recipientName;
    }
    // セッターなし、equals()で属性値を比較
}
```

## D1-17　設計上エンティティか値オブジェクトかを判断するシナリオ問題（応用）

**問い**

以下の各概念について、エンティティと値オブジェクトのどちらで設計すべきかを判断し、理由を説明してください。

- **A. 銀行口座（Bank Account）**：口座番号で識別され、残高が変化する
- **B. 通貨（Currency）**：ISO 4217コードで表される（USD、JPYなど）
- **C. 請求書（Invoice）**：請求書番号を持ち、ステータス（下書き/確定/支払い済み）が変化する
- **D. 住所（Address）**：国・都道府県・市区町村・番地の組み合わせ

---

**模範解答**

| 概念 | 分類 | 理由 |
|---|---|---|
| **A. 銀行口座** | **エンティティ** | 口座番号（ID）で追跡。残高が変化してもライフサイクルを持って同一口座として管理 |
| **B. 通貨** | **値オブジェクト** | USDとUSDは同じ。ライフサイクルを持たない。属性値（ISOコード）で等値を判断 |
| **C. 請求書** | **エンティティ** | 請求書番号（ID）で追跡。ステータス変化があっても同一の請求書として管理 |
| **D. 住所** | **値オブジェクト** | 「東京都渋谷区〇〇」という値で等値を判断。ライフサイクルを持たない。変わったら新しい住所に置き換える |

## D1-18　複数の属性をまとめた概念的統一体として値オブジェクトを設計する（応用）

**問い**

ECサイトの商品価格を「定価（int）」「割引率（double）」「税率（double）」の3つの独立したフィールドとして持つ設計の問題点を指摘し、値オブジェクトを使った改善案を設計してください。

---

**模範解答**

**問題点**

- 3つのフィールドが「常に一緒に使われる」という関係がコードに表現されない
- 価格計算ロジック（最終価格 = 定価 × (1 - 割引率) × (1 + 税率)）が呼び出し元に散在する
- 定価だけ渡して割引率を渡し忘れるミスをコンパイラが防げない

**値オブジェクトを使った改善案**

```csharp
public class ProductPrice {
    public int ListPrice { get; }
    public double DiscountRate { get; }
    public double TaxRate { get; }
    
    public ProductPrice(int listPrice, double discountRate, double taxRate) {
        // バリデーション（例：割引率は0以上1未満）
        ListPrice = listPrice;
        DiscountRate = discountRate;
        TaxRate = taxRate;
    }
    
    // 最終価格の計算ロジックをここに集約
    public int CalculateFinalPrice() =>
        (int)(ListPrice * (1 - DiscountRate) * (1 + TaxRate));
    
    // セッターなし
}
```

価格に関する振る舞いが`ProductPrice`に集まり、呼び出し元に散在しない。

## D1-19　コンテキスト間で使い回される値オブジェクトの設計と注意点（応用）

**問い**

「注文コンテキスト」と「請求コンテキスト」の両方で「金額（Money）」という値オブジェクトを使いたいとします。共有する場合の注意点と、推奨される設計方針を説明してください。

---

**模範解答**

**注意点**

- 各コンテキストで「金額」の意味が微妙に異なる可能性がある（注文コンテキストでは商品価格、請求コンテキストでは請求総額）
- 一方のコンテキストのニーズに合わせて`Money`を変更すると、他方のコンテキストに影響する

**推奨される設計方針**

**各コンテキストで独自に定義する**（コピー）：

- 各BCに`Money`クラスをそれぞれ定義し、BC内のユビキタス言語・ルールに合わせて独立して育てる
- コンテキスト間で共有しないことでBCの独立性を保つ

共有カーネルを採用する場合は、両チームの合意と変更管理が必要。変更頻度が低くシンプルな値オブジェクトに限定する。

## D1-20　バリデーションをどのレベルに実装するか判断するシナリオ問題（応用）

## D1-21　エンティティのライフサイクルを設計する（応用）

**問い**

「注文（Order）」エンティティのライフサイクル（生成・変更・削除）を設計してください。各段階でどのようなバリデーションを行うかも含めて説明してください。

---

**模範解答**

**生成**

- コンストラクタで`OrderId`（UUID早期生成）・`customerId`・`items`（最低1件）を必須引数として受け取る
- 引数が不正な場合はコンストラクタを失敗させる
- 初期ステータスは`Draft`（下書き）

**変更**

- `confirmOrder()`：ステータスを`Confirmed`に変更。商品が1件以上あることをpreconditionで確認
- `addItem(OrderItem item)`：`Draft`状態のみ商品を追加可能（`Confirmed`後は変更不可）
- `cancel()`：`Confirmed`状態のみキャンセル可能

**削除**

- 物理削除ではなくステータスを`Cancelled`に変更するソフトデリートを採用
- 支払い済み（`Paid`）の注文はキャンセル不可（invariant）

## D1-22　値オブジェクトのwithメソッドパターンの設計と適用場面（応用）

**問い**

`BusinessPriority`値オブジェクトの評価値（ratings）だけを変えた新しいオブジェクトを作りたい場合に、`withRating`メソッドを使ったパターンを設計し、このパターンが有効な場面を説明してください。

---

**模範解答**

**`withRating`メソッドの設計**

```csharp
public class BusinessPriority {
    private BusinessPriorityRatings _ratings;
    
    public BusinessPriority(BusinessPriorityRatings ratings) {
        _ratings = ratings;
    }
    
    // 新しいratingsを持つ新しいインスタンスを返す（自身は変わらない）
    public BusinessPriority WithRating(BusinessPriorityRatings newRatings) {
        return new BusinessPriority(newRatings);
    }
}

// 使用例
var original = new BusinessPriority(ratings1);
var updated = original.WithRating(ratings2);
// original は変わらない
```

**有効な場面**

- 値オブジェクトの一部属性だけ変えた「バリエーション」を作りたい場合
- 元のオブジェクトを保持しながら変更後のオブジェクトも必要な場合（undo/redo、差分比較など）
- 不変性を守りながら状態変化をシミュレートする場合

## D1-23　マルチテナント環境でのエンティティ識別子設計（応用）

**問い**

マルチテナントSaaSシステムで「ユーザーID（username）が各テナント内で一意だが、テナントをまたいだ場合は重複してよい」という要件があります。エンティティの識別子をどのように設計するかを説明してください。

---

**模範解答**

**問題**：usernameがテナント間で重複するため、usernameだけでは全システムで一意な識別子にならない。

**解決策：複合識別子の採用**

```java
public class UserId {
    private TenantId tenantId; // テナントを識別するUUID
    private String username;   // テナント内で一意
    
    public boolean equals(Object other) {
        UserId o = (UserId) other;
        return this.tenantId.equals(o.tenantId) && this.username.equals(o.username);
    }
}
```

`TenantId + username`の組み合わせで全システムで一意な識別子を構成する。`equals`・`hashCode`はこの複合識別子で実装する。

## D1-24　他コンテキストから割り当てられる識別子を安全に扱う設計（応用）

**問い**

注文コンテキストが外部の商品コンテキストから商品IDを受け取り、注文明細のエンティティで使用します。このとき、外部から受け取った識別子を安全に扱うための設計上の考慮点を説明してください。

---

**模範解答**

**考慮点**

1. **型による保護**：生の`string`や`int`で持つのではなく、`ProductId`値オブジェクトでラップする。型でビジネス上の意味を明確にし、誤った値の混在をコンパイラで防ぐ
2. **バリデーション**：外部から受け取った時点でフォーマット・存在確認（必要に応じて）のバリデーションを行う
3. **ACLで変換**：腐敗防止層（ACL）で外部コンテキストの識別子表現を自コンテキストの`ProductId`に変換して取り込む
4. **変更不可の保証**：`ProductId`を一度設定したら変更できないようにする（一意な識別子の不変性）

## D1-25　ドメインエキスパートの言葉から値オブジェクトの境界を見つける（応用）

**問い**

医療システムのドメインエキスパートが「患者の血圧は、上の血圧（収縮期）と下の血圧（拡張期）をセットで記録する。片方だけ記録することはない」と言っています。この発言から値オブジェクトの設計を導いてください。

---

**模範解答**

**発見した値オブジェクト**：`BloodPressure`（血圧）

**根拠**

- 「セットで記録する」「片方だけはない」→ 2つの属性が「概念的統一体」を形成する
- 血圧は計測値（数値）であり「計測/定量化」の特性に当てはまる
- 「〇〇番の患者の血圧」ではなく「120/80」という値そのものが意味を持つ → 値オブジェクト

**設計**

```java
public class BloodPressure {
    private int systolic;   // 収縮期（上）
    private int diastolic;  // 拡張期（下）
    
    public BloodPressure(int systolic, int diastolic) {
        // バリデーション（例：systolic > diastolic、正の値）
        this.systolic = systolic;
        this.diastolic = diastolic;
    }
    // セッターなし、equals()で属性値を比較
}
```

## D1-27　値オブジェクトを使ってドメインルールをコードに表現する（応用）

**問い**

ECサイトで「1回の注文で購入できる数量は1〜99個まで」というドメインルールがあります。このルールを`Quantity`値オブジェクトに表現してください。このアプローチの利点も説明してください。

---

**模範解答**

**`Quantity`値オブジェクトの設計**

```csharp
public class Quantity {
    private int _value;
    
    public Quantity(int value) {
        if (value < 1 || value > 99)
            throw new ArgumentException("数量は1以上99以下でなければなりません");
        _value = value;
    }
    
    public int Value => _value;
    
    public Quantity Add(Quantity other) {
        return new Quantity(_value + other._value); // ルール違反時は例外
    }
    
    // equals() で _value を比較
}

// 使用例
var qty = new Quantity(50);      // OK
var invalid = new Quantity(100); // 例外 → ルール違反をコンパイル後に即検出
```

**利点**

- ドメインルール（1〜99個）が`Quantity`クラスに集約され、呼び出し元に散在しない
- `int`の代わりに`Quantity`型を使うことで、型でビジネス上の意味を表現できる
- ルール変更（例：上限を99→999に変更）が1箇所の修正で済む
- テストがしやすい（`Quantity`単体でルールのテストができる）
