## E4-01【知識確認】DDDにおけるモジュールの定義

**問題**（用語問題）

DDDにおける「モジュール」とは何か定義せよ。また、JavaとC#ではそれぞれ何に対応するか答えよ。

**模範解答**

**定義**：プログラミング言語における名前空間（パッケージ）に相当するDDDのモデリング要素。ソースファイルの物理的な整理にとどまらず、**ドメインの概念をユビキタス言語で表現するための主要なモデリング要素**として活用する。

- **Java**：`package` に対応
- **C#**：`namespace` に対応

## E4-02【知識確認】モジュール設計の2原則

**問題**（用語問題）

DDDにおけるモジュール設計の2つの原則を答え、それぞれの意味を説明せよ。

**模範解答**

1. **高凝集（High Cohesion）**：モジュール内の要素が互いに強く関連し、単一の責任・概念にまとまっている状態。凝集度が高いほどシステムのメンテナンス性が向上する
2. **疎結合（Loose Coupling）**：モジュール間の依存関係が最小限に抑えられている状態。双方向の依存関係はできるかぎり避けることが推奨される

## E4-03【知識確認】モジュール命名の階層構造

**問題**（用語問題）

実践ドメイン駆動設計で示されるSaaSovationプロジェクトのモジュール命名規約を例示し、各階層の意味を説明せよ。

**模範解答**

```
SaaSovation.AgilePM.Domain.Model.Products
```

| 階層 | 例 | 意味 |
|------|----|------|
| 組織名 | `SaaSovation` | 会社・組織の名称（衝突防止） |
| コンテキスト名 | `AgilePM` | 境界づけられたコンテキスト |
| ドメイン | `Domain` | ドメイン関連要素の格納場所 |
| モデル | `Model` | 値オブジェクト・エンティティ・イベントなど |
| サブドメイン概念 | `Products` | ユビキタス言語に基づく概念名 |

## E4-05【知識確認】実践ドメイン駆動設計のモジュール設計ルール

**問題**（用語問題）

実践ドメイン駆動設計が示すモジュール設計ルール8項目のうち、特に重要な3つを挙げ、それぞれ説明せよ。

**模範解答**

1. **モジュールをドメイン上の概念に沿って設計する**（ルール1）：ドメインの構造がモジュール構造に反映されるため、ドメインへの理解がコードから読み取れる
2. **ユビキタス言語でモジュール名を命名する**（ルール7）：技術的な名前ではなくドメインの語彙を使うことで、コードとドメインエキスパートの会話を一致させる
3. **ドメインのコンセプトが変わったらモジュールもリファクタリングする**（ルール8）：モジュール名はユビキタス言語の現在の状態を反映すべきであり、陳腐化したモジュール名を放置しない

## E4-06【理解確認】コンテキストとモジュールの使い分け

**問題**（読解判断問題）

「モジュール」と「境界づけられたコンテキスト」は何が異なるか。どちらで分割すべきかを判断する基準を説明せよ。

**模範解答**

| 観点 | モジュール | 境界づけられたコンテキスト |
|------|-----------|------------------------|
| 性質 | プログラミング的な仕組み（パッケージ） | ユビキタス言語のビジネス的な境界線 |
| 規模 | 小〜中：クラス群のまとまり | 大：独立したチーム・デプロイ単位の可能性 |
| 目的 | コードの責任整理 | ドメインの概念的独立性の確保 |

**判断基準**：モデルが成長してきた段階で、ドメインエキスパートの言葉に従いどちらによる分割が最適かを検討する。最初は分割せず、ひとまとめのままにするほうが無難な場合もある。ビジネス的に独立した言語（ユビキタス言語）が存在するなら境界づけられたコンテキスト、コードの整理が目的ならモジュールを使う。

## E4-07【理解確認】双方向依存の問題

**問題**（読解判断問題）

SaaSovationの `Teams` モジュールは `Tenants` モジュールへの参照と `Products` モジュールからの参照の両方が存在する。この双方向の依存関係が問題になる理由を説明せよ。

**模範解答**

双方向の依存関係（循環的依存）があると：

1. **リファクタリングの連鎖**：`Teams` を変更すると `Products` に影響し、さらに `Teams` に影響が戻るという循環が起きる。一方を変えるには他方も修正が必要になり、変更コストが高くなる
2. **独立したビルド・テストの困難**：モジュールを独立してビルド・テストできない。どちらかが他方に依存しているため、切り離してテストすることが難しくなる
3. **メンテナンス性の低下**：依存関係が複雑になるほど、影響範囲の把握が難しくなり、予期しないバグが生まれやすくなる

## E4-08【応用】モジュール名の評価

**問題**（応用）

以下の2つのモジュール名を比較し、どちらがDDD的に適切かを選択し、その理由を述べよ。

- `（A）SaaSovation.AgilePM.Application.Services.Impl`
- `（B）SaaSovation.AgilePM.Application.Products`

**模範解答**

**（B）`SaaSovation.AgilePM.Application.Products` が適切**。

理由：
1. **ユビキタス言語**：`Products` はドメイン概念（プロダクト）を直接表す。`Services.Impl` は「サービスの実装」という技術的な名前であり、ドメインの語彙ではない
2. **責任の明確化**：`Products` モジュールとすることで、プロダクトに関するアプリケーションサービスを格納する場所だと一目でわかる
3. **（A）の問題**：`Impl` というサフィックスは実装の詳細を名前に含めており、将来の変更時にモジュール名と実態が乖離しやすい

## E4-09【応用】既存プロジェクトへのモジュール設計適用

**問題**（応用）

既存プロジェクトで以下の状態になっている。DDDのモジュール設計原則に照らして問題点を2つ指摘し、改善案を述べよ。

現状：
```
com.example.util         // StringUtils, DateUtils, MathUtils など
com.example.models       // User, Product, Order, Category など全てのエンティティ
com.example.services     // UserService, ProductService, OrderService など
```

**模範解答**

**問題点①：ドメイン概念でなく技術的な役割で分類されている**  
`models`・`services` はレイヤ（技術役割）でまとめており、ドメインの概念（ユビキタス言語）に沿った分割になっていない。実践ドメイン駆動設計ルール1・3・7に違反している。

**問題点②：高凝集が実現できていない**  
`User`・`Product`・`Order` が同じ `models` モジュールに混在しており、全く異なるドメイン概念が一か所に集まっている。ドメインの境界がコードに反映されていない。

**改善案**：
```
com.example.identity        // User, UserId, UserRepository など
com.example.catalog         // Product, Category, CatalogService など
com.example.ordering        // Order, OrderItem, OrderId, OrderRepository など
```
各モジュールはドメイン上の概念単位（ユビキタス言語）でまとめ、エンティティ・値オブジェクト・リポジトリ・ドメインサービスを同じモジュール内に格納する。
