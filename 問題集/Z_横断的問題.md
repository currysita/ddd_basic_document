## Z-01【知識確認】DDDの3層構造

**問題**（用語問題）

DDDにおける「問題領域」「解決領域」「実装」の3層構造を説明し、それぞれが対応するDDDの概念を答えよ。

**模範解答**

1. **問題領域（Problem Space）**：ビジネスが抱える問題・課題の空間。コアドメイン・支援サブドメイン・汎用サブドメインの区分で捉える
2. **解決領域（Solution Space）**：問題領域の解決策としてのシステム設計。境界づけられたコンテキストとコンテキストマップで表現する
3. **実装**：解決領域を実際のコードで表現したもの。エンティティ・値オブジェクト・集約・リポジトリ・ファクトリ・ドメインサービスなどのビルディングブロックで構成する

## Z-02【知識確認】ユビキタス言語とモデルの関係

**問題**（用語問題）

「ユビキタス言語（Ubiquitous Language）」がドメインモデルとコードに与える影響を説明せよ。

**模範解答**

ユビキタス言語はドメインエキスパートと開発者が共有する共通言語。この言語はコードのクラス名・メソッド名・変数名として直接反映される。例えば「フォーラムでディスカッションを開始する」という業務概念は `Forum.StartDiscussionFor()` というメソッドとして表現される。ユビキタス言語が変化した場合はコードもリファクタリングする。これによりビジネスの変化がモデルとコードに素直に反映される。

## Z-03【知識確認】DDDのビルディングブロック7種

**問題**（用語問題）

DDDの戦術的設計で使われるビルディングブロック（構成要素）を7種答えよ。

**模範解答**

1. **エンティティ（Entity）**：同一性（ID）によって区別されるオブジェクト
2. **値オブジェクト（Value Object）**：値の等値性で比較されるイミュータブルなオブジェクト
3. **集約（Aggregate）**：整合性境界を持つエンティティ・値オブジェクトのクラスター
4. **ドメインサービス（Domain Service）**：エンティティや値オブジェクトに自然に属さないドメインロジック
5. **ドメインイベント（Domain Event）**：ドメイン内で発生した重要な出来事を表すオブジェクト
6. **ファクトリ（Factory）**：集約の生成ロジックをカプセル化するオブジェクト
7. **リポジトリ（Repository）**：集約の永続化（追加・削除・取得）を担うオブジェクト

## Z-04【知識確認】コンテキストマップの統合パターン

**問題**（用語問題）

コンテキストマップで使われる5つの統合パターンを答えよ。

**模範解答**

1. **共有カーネル（Shared Kernel）**：2つのコンテキストが同じモデルの一部を共有する
2. **大きな泥団子（Big Ball of Mud）**：境界が不明確なレガシーシステムとの統合
3. **公開ホストサービス（OHS：Open Host Service）**：汎用プロトコルでサービスを公開する
4. **公開された言語（PL：Published Language）**：共有されたドキュメント化された交換言語を使う
5. **腐敗防止層（ACL：Anti-Corruption Layer）**：外部モデルを自コンテキストのモデルに変換する層

## Z-05【知識確認】集約の4設計ルール

**問題**（用語問題）

IDDDが示す集約設計の4つのルールを答えよ。

**模範解答**

1. **集約ルートを通じてのみ内部を変更する**：集約の外部から直接内部のエンティティを変更しない
2. **1トランザクションで変更できる集約は1つのみ**：集約をまたぐ変更は結果整合性で対応する
3. **IDで他の集約を参照する**：他集約へのオブジェクト参照ではなくIDで参照し、境界を明確にする
4. **小さな集約を設計する**：巨大集約はトランザクション競合・パフォーマンス問題・モデリングエラーを招く

## Z-06【知識確認】レイヤードアーキテクチャの4層

**問題**（用語問題）

レイヤードアーキテクチャの4つの層を上から順に答え、それぞれの役割を説明せよ。

**模範解答**

1. **UI層（プレゼンテーション層）**：ユーザーが操作するインターフェイスを提供。ビューとプレゼンテーションモデルで構成
2. **アプリケーション層**：ユースケースに応じた調整役。ドメインモデルを操作し、トランザクション境界を管理する
3. **ドメイン層**：ビジネスルールを表現するコア。エンティティ・値オブジェクト・集約・ドメインサービス・ドメインイベントが属する
4. **インフラストラクチャ層**：技術的な実装詳細（DB・外部API・メッセージング）。リポジトリ実装クラスが属する

## Z-07【知識確認】ドメインイベントの必須属性

**問題**（用語問題）

ドメインイベントが持つべき2つの必須属性を答えよ。

**模範解答**

1. **occurredOn（発生日時）**：イベントが発生した日時（タイムスタンプ）
2. **集約ルートのID（発生源の識別子）**：イベントを発生させた集約ルートのID（例：`forumId`、`orderId`）

これらにより、いつ・どの集約で何が起きたかを特定できる。

## Z-08【知識確認】値オブジェクトの6特性

**問題**（用語問題）

値オブジェクトが持つ6つの特性を答えよ。

**模範解答**

1. **計測・定量化・説明**：何かを計測・定量化・説明する目的で使われる
2. **不変性（Immutable）**：生成後に内部状態が変わらない
3. **概念的統一体**：関連する属性を一つの概念としてまとめる
4. **交換可能性（Replaceable）**：値を変えたい場合は新しいインスタンスで置き換える
5. **等値性（Equality）**：すべての属性値が同じなら等しいと判断する
6. **副作用なし振る舞い**：メソッドが内部状態を変えず、副作用がない

## Z-09【理解確認】エンティティと値オブジェクトの選択

**問題**（読解判断問題）

以下のドメイン概念をエンティティと値オブジェクトに分類し、理由を述べよ。

（A）注文（Order）  （B）住所（Address）  （C）お金（Money）  （D）ユーザーアカウント（UserAccount）  （E）注文明細行（OrderItem）

**模範解答**

- **（A）注文（Order）→ エンティティ**：注文番号というIDで区別される。状態が変化する（受注→出荷→完了）
- **（B）住所（Address）→ 値オブジェクト**：「東京都渋谷区〇〇1-2-3」という値で表現され、ID不要。不変でよく、同じ住所なら等しいと判断できる
- **（C）お金（Money）→ 値オブジェクト**：「100円」という値で表現。金額と通貨の組み合わせで等値性を判断できる
- **（D）ユーザーアカウント（UserAccount）→ エンティティ**：ユーザーIDで識別される。名前が変わっても同一ユーザー
- **（E）注文明細行（OrderItem）→ エンティティまたは値オブジェクト**：状況次第。同一注文内で同一商品の明細が区別できる必要があるならエンティティ、そうでなければ値オブジェクト

## Z-10【理解確認】ドメインサービスと貧血ドメインモデル

**問題**（読解判断問題）

「貧血ドメインモデル（Anemic Domain Model）」とは何か。ドメインサービスを使いすぎることがなぜ貧血ドメインモデルを招くか説明せよ。

**模範解答**

**貧血ドメインモデル**：エンティティや値オブジェクトがデータ（フィールド）のみを持ち、ビジネスロジックを持たないモデル。全てのロジックがサービス層に集まる。

**ドメインサービスの使いすぎが招く理由**：ドメインサービスはエンティティ・値オブジェクトに「自然に属さない」ロジックのみを置くべき場所。しかし、本来エンティティが持つべきロジック（例：`User.ChangePassword()`）まで `UserDomainService.ChangePassword(user, newPassword)` のようにサービスに置いてしまうと、エンティティが空洞化し貧血ドメインモデルになる。

## Z-11【理解確認】集約ルートを経由しなければならない理由

**問題**（読解判断問題）

「集約の外部から直接内部エンティティを変更してはいけない」というルールの理由を説明せよ。具体例で示すこと。

**模範解答**

**理由**：集約ルートが不変条件（ビジネスルール）を保証する門番だから。

**具体例**：`Order`集約に `OrderItem`が含まれているとする。
```
// 禁止：集約ルートを経由しない
order.Items[0].Quantity = 0;  // 不変条件チェックなし
// 許可：集約ルートを経由する
order.ChangeItemQuantity(itemId, newQuantity);  // Orderが「数量≥1」を保証
```
集約ルートを経由することで「注文明細の数量は1以上でなければならない」という不変条件が常に検証される。直接変更するとこのチェックをバイパスできてしまう。

## Z-12【理解確認】トランザクション整合性と結果整合性の使い分け

**問題**（読解判断問題）

「1トランザクションで変更できる集約は1つのみ」というルールがある。では、複数の集約を変更しなければならないビジネス要件はどう実現するか。

**模範解答**

**結果整合性（Eventual Consistency）**で実現する。

方法：
1. **ドメインイベント**：1つの集約を変更するトランザクションが完了時に、ドメインイベントを発行する
2. **非同期処理**：イベントを受け取った別の集約が、別のトランザクションで自身を更新する
3. **最終的に一貫性が達成される**：即時の一貫性は保証されないが、しばらく後には全体が一貫した状態になる

例：`Order` を確定するとき `OrderPlaced` イベントを発行→在庫コンテキストが `InventoryItem.Reserve()` を別トランザクションで実行する。

## Z-13【理解確認】依存性逆転の原則とDDD

**問題**（読解判断問題）

DDDにおいて依存性逆転の原則（DIP）がどのように活用されているか、2つの具体例を挙げて説明せよ。

**模範解答**

1. **リポジトリ**：リポジトリのインターフェイスはドメイン層に定義し、実装クラスはインフラ層に置く。ドメイン層がインフラ層のHibernateやLevelDBに依存せず、インフラ層がドメイン層のインターフェイスに依存する
2. **ドメインサービスのセパレートインターフェイス**：ドメインサービスがメールサービスなどのインフラを必要とする場合、`IEmailNotificationService` インターフェイスをドメイン層に定義し、実装はインフラ層に置く。これによりドメインは特定のメール送信技術に依存しない

どちらも「ドメイン層が具体的な技術（インフラ）に依存しない」ことを保証し、ドメインモデルをクリーンに保つ。

## Z-14【理解確認】DDDとマイクロサービスの関係

**問題**（読解判断問題）

境界づけられたコンテキストとマイクロサービスの関係を説明せよ。1つのBCが1つのマイクロサービスに対応するとは限らない理由も述べよ。

**模範解答**

**関係**：境界づけられたコンテキスト（BC）はマイクロサービスを分割する際の自然な候補単位。BCの境界はユビキタス言語の境界であり、ビジネス的に独立した責務を表すためサービス分割と相性がよい。

**1BC=1マイクロサービスとは限らない理由**：
- 小さなBCを個別サービスにすると運用コスト（デプロイ・監視・ネットワーク）が高くなりすぎる場合がある
- チームの規模・成熟度によっては1チームが複数BCを担当することもある
- 最初はモノリシックで複数BCを実装し、成長とともにサービス分割するアプローチ（モジュラーモノリス）もある
- BCは設計の単位であり、マイクロサービスはデプロイ・運用の単位として別の視点で判断する

## Z-15【理解確認】ドメインイベントとドメインサービスの使い分け

**問題**（読解判断問題）

「ユーザーが登録されたとき、ウェルカムメールを送信する」というビジネス要件を実現するのに、ドメインイベントを使う場合とドメインサービスを直接呼ぶ場合の違いを比較せよ。

**模範解答**

**ドメインサービス直接呼び出し**：
```
// アプリケーションサービス
user = userFactory.create(...);
userRepository.add(user);
emailService.sendWelcome(user);  // 直接呼び出し
```
- 問題：アプリケーションサービスが「メールを送る」という知識を持つ。ユーザー登録とメール送信が密結合。メール送信が増えるたびにアプリケーションサービスを変更しなければならない

**ドメインイベント使用**：
```
// アプリケーションサービス
user = userFactory.create(...);
userRepository.add(user);
// UserRegistered イベントが自動発行される
// 別のハンドラーがメール送信を担当する
```
- 利点：ユーザー登録の集約は「登録した」という事実を発行するだけ。メール送信・ポイント付与など後続処理は独立したハンドラーで対応でき、追加・変更が容易

## Z-16【応用】DDDの導入判断

**問題**（応用）

以下のプロジェクトについて、DDDを本格的に導入すべきかどうかを判断し、理由を述べよ。

プロジェクトA：社内の勤怠管理システム。ルールが明確で変化がほとんどない。開発チームは3人。  
プロジェクトB：複雑な金融取引プラットフォーム。ビジネスルールが複雑で頻繁に変化。ドメインエキスパートとの密な協働が必要。

**模範解答**

**プロジェクトA：DDDは不向き**  
ルールが単純・安定でドメインエキスパートとの継続的な協働も少ない場合、DDDの戦術的設計（集約・リポジトリ・ドメインサービスなど）のオーバーヘッドが実装コストを増やすだけになる。シンプルなCRUDアーキテクチャで十分。

**プロジェクトB：DDDが適している**  
複雑なビジネスルール・頻繁な変化・ドメインエキスパートとの協働が必要な場合こそDDDが価値を発揮する。ユビキタス言語により専門家とエンジニアのコミュニケーションが向上し、境界づけられたコンテキストにより複雑さを分割統治できる。

## Z-17【応用】コアドメインの識別

**問題**（応用）

ECサイト企業のシステムで、以下のドメインをコアドメイン・支援サブドメイン・汎用サブドメインに分類し、理由を述べよ。

（A）商品レコメンデーション（競合差別化の核）  
（B）注文管理  
（C）ユーザー認証  
（D）メール送信  
（E）在庫管理

**模範解答**

- **（A）商品レコメンデーション → コアドメイン**：競合他社との差別化を生む核心。最も投資すべき領域
- **（B）注文管理 → コアドメインまたは支援サブドメイン**：EC事業の中心的業務。独自のビジネスルールが多ければコアドメイン
- **（C）ユーザー認証 → 汎用サブドメイン**：どのWebサービスにも必要。Auth0やCognitoなど既製品で代替できる
- **（D）メール送信 → 汎用サブドメイン**：業界共通。SendGridなどSaaSで代替できる
- **（E）在庫管理 → 支援サブドメイン**：EC事業を支えるが差別化にはならない。SAP等の既製品で代替も検討できる

## Z-18【応用】レガシーシステムとのDDD統合

**問題**（応用）

ERPシステム（レガシー）と新規DDDシステムを統合する際に使うコンテキストマップのパターンを2つ選択し、その使い方を説明せよ。

**模範解答**

1. **腐敗防止層（ACL）**：新DDDシステム側にACLを設け、ERPの旧モデルを自コンテキストのユビキタス言語に変換する。`ERPProduct` → `CatalogItem` のような変換をACLが担い、ERP側の概念がドメインモデルに汚染されることを防ぐ

2. **大きな泥団子（Big Ball of Mud）**：ERPシステムを「大きな泥団子」として扱い、境界を明確に定義せず統合する。実用上やむを得ない場合に使う。ただし長期的にはACLを設けてリファクタリングする計画を持つべき

## Z-19【応用】アーキテクチャ選択とDDD

**問題**（応用）

以下の要件に対してCQRSとイベントソーシングの両方を採用すべきか検討し、採用する場合・しない場合それぞれの根拠を示せ。

要件：金融取引の監査ログが義務付けられており、すべての状態変化を後から再現できる必要がある。読み取りクエリは複雑で複数の集約情報を組み合わせる必要がある。

**模範解答**

**採用すべき**。

**イベントソーシング採用の根拠**：
- 「すべての状態変化を後から再現できる」という要件はイベントソーシングと完全に一致する。イベントストアに全ドメインイベントを蓄積することで任意の時点の状態を再現でき、監査ログが自然に実現できる

**CQRS採用の根拠**：
- 読み取りクエリが複雑で複数集約を組み合わせる必要がある場合、書き込み用の集約（イベントソーシング）から独立した読み取り専用モデル（リードモデル）を構築することで、クエリの複雑さと書き込みの複雑さを分離できる
- イベントを受け取ってリードモデルを更新する（プロジェクション）ことで、UIに最適化したデータを効率的に提供できる

## Z-20【応用】ユビキタス言語の具体化

**問題**（応用）

病院の予約システムを開発する。以下のエンジニア用語を、ドメインエキスパート（医師・受付スタッフ）との会話で使われるユビキタス言語に置き換えよ。

- `Appointment.setStatus("CANCELLED")`
- `Patient.updateRecord(newData)`
- `Schedule.findAvailableSlot(date)`

**模範解答**

- `Appointment.setStatus("CANCELLED")` → `Appointment.Cancel(CancellationReason reason)` または `Appointment.CancelByPatient()` / `Appointment.CancelByClinic()`
  - 理由：「キャンセルする」という業務行為と、誰がキャンセルするかが明確になる

- `Patient.updateRecord(newData)` → `Patient.UpdateMedicalHistory(diagnosis, treatedAt)` または `Patient.RegisterAllergy(allergyInfo)`
  - 理由：「記録を更新する」は抽象的すぎる。何の情報をどう更新するかをドメイン語彙で表現する

- `Schedule.findAvailableSlot(date)` → `Schedule.FindAvailableAppointmentSlotFor(Specialty specialty, DateRange preferredPeriod)`
  - 理由：「何の専門科の予約枠を探すか」というドメイン知識を引数に反映する

## Z-21【応用】集約の設計選択

**問題**（応用）

SNSアプリで「投稿（Post）」と「コメント（Comment）」の設計について、以下の2案を比較し、DDDの観点でどちらが適切か判断せよ。

案A：`Post` 集約が `Comment` のリストを持つ（`Post → List<Comment>`）  
案B：`Post` と `Comment` を別々の集約にし、`Comment` が `PostId` を持つ

**模範解答**

**案Bが適切**。

理由：
1. **パフォーマンス**：案Aでは `Post` を取得するたびに全コメントをロードする必要がある。コメント数が数千に達すると深刻なパフォーマンス問題になる（巨大集約の問題）
2. **トランザクション競合**：案Aでは、コメントの投稿・削除のたびに `Post` 集約全体がロックされる。多数のユーザーが同時にコメントするとトランザクション競合が激化する
3. **独立した不変条件**：コメントの不変条件（例：本文は1〜280文字）は `Post` の不変条件と独立している。別集約にすることでそれぞれが独立して管理できる
4. **結果整合性で十分**：投稿とコメントが同一トランザクションで整合する必要はない。`Comment.PostId` で関連を保ちつつ、別トランザクションで管理できる

## Z-23【応用】コンテキストマップの読み方

**問題**（応用）

コンテキストマップ上で「認証・アクセスコンテキスト（上流）→ OHS/PL → コラボレーションコンテキスト（下流・ACL）」という関係がある。この関係で各パターンが果たす役割を説明せよ。

**模範解答**

- **OHS（公開ホストサービス）**：認証・アクセスコンテキストが複数のコンテキストから利用されるため、汎用的なREST APIとして認証情報を公開している。特定のクライアントに合わせた特注APIではない

- **ACL（腐敗防止層）**：コラボレーションコンテキスト側に設けられ、認証コンテキストの `User` モデルをコラボレーションコンテキストの `Author` `Creator` `Moderator` などの値オブジェクトに変換する。認証コンテキストのモデル変更がコラボレーションのドメインモデルに波及することを防ぐ

## Z-25【応用】DDDの適用範囲の見極め

**問題**（応用）

（A）商品レコメンデーション（AIモデルによる推薦）  
（B）注文管理（複雑なビジネスルールあり）  
（C）ログ収集・分析基盤

- **（A）商品レコメンデーション → 限定的適用**：AIモデルの推薦は機械学習パイプラインであり、DDDのビルディングブロックよりもデータパイプライン・特徴量エンジニアリングの設計が主体。入出力インターフェイスにDDDのコンセプト（`Recommendation` 値オブジェクトなど）を適用するに留める

- **（C）ログ収集・分析基盤 → 適用不要**：汎用的なインフラ機能。Elasticsearch・Fluentdなどの既製品で実現すべき。DDDを適用するほど複雑なビジネスルールがなく、汎用サブドメインとして外部サービスを活用する

## Z-26【応用】ファクトリ・集約・リポジトリの協調

**問題**（応用）

「新規ユーザーが登録される」というユースケースで、ファクトリ・集約ルート・リポジトリがどのように協調するか、処理フローを説明せよ。

**模範解答**

```
処理フロー：
1. アプリケーションサービスが RegisterUserCommand を受け取る
2. アプリケーションサービスが Tenant 集約をリポジトリから取得する
   → tenantRepository.TenantOfId(tenantId)
3. Tenant 集約のファクトリメソッドを呼んで User 集約を生成する
   → tenant.RegisterUser(invitationId, username, ...)
     ← 内部でInvitationの有効性確認（不変条件）
     ← UserRegistered ドメインイベントが発行される
4. 生成した User をリポジトリに追加する
   → userRepository.Add(user)
5. リポジトリがデータベースに User を保存する（トランザクションコミット）
6. ドメインイベント UserRegistered を購読するハンドラーがウェルカムメールを送信する

役割分担：
- ファクトリメソッド (Tenant.RegisterUser): User生成の不変条件を保証
- 集約ルート (Tenant): Invitation の有効性という不変条件を検証
- リポジトリ (UserRepository): 永続化の詳細を隠蔽
- アプリケーションサービス: 全体の調整・トランザクション管理
```

## Z-30【応用】DDDのアンチパターン診断

**問題**（応用）

以下のコードに潜むDDDのアンチパターンを2つ特定し、改善案を示せ。

```csharp
public class OrderService {
    public void ProcessOrder(int orderId) {
        var order = _db.Orders.Find(orderId);  // ①
        if (order.Status == "PENDING") {        // ②
            order.Status = "PROCESSING";
            order.TotalAmount = order.Items.Sum(i => i.Price * i.Quantity);  // ③
            _db.SaveChanges();
        }
    }
}
```

**模範解答**

**アンチパターン①：ドメインロジックがサービスに漏れている（貧血ドメインモデル）**  
注文のステータス変更・合計金額計算は `Order` 集約が持つべきドメインロジック。`OrderService` がこれを担うと、ドメイン知識がサービス層に散在し、再利用もテストも困難になる。

**改善**：
```csharp
// Order集約がドメインロジックを持つ
public class Order {
    public void StartProcessing() {
        if (this.Status != OrderStatus.Pending)
            throw new InvalidOperationException("注文は保留状態でなければ処理できない");
        this.Status = OrderStatus.Processing;
        this.TotalAmount = this.Items.Sum(i => i.Price * i.Quantity);
        DomainEventPublisher.Publish(new OrderProcessingStarted(this.OrderId));
    }
}
```

**アンチパターン②：文字列でステータスを管理している**  
`"PENDING"` `"PROCESSING"` のような文字列リテラルはタイポのリスクがあり、型安全性がない。

**改善**：`OrderStatus` 列挙型（または値オブジェクト）を使う。

## Z-31【応用】システム全体のDDD設計レビュー

**問題**（応用）

以下の設計図を読み、DDDの観点で2つの問題点を指摘し、改善案を示せ。

設計内容：
- `UserController` が `UserRepository` を直接 `new UserRepository()` している
- `UserRepository` がメール送信処理も含んでいる（ユーザー登録完了後にメールを送信する）

**模範解答**

**問題①：UserController が UserRepository を直接インスタンス化している**  
依存性逆転の原則（DIP）違反。コントローラーがリポジトリの具体実装クラスに直接依存するため、テスト時にモックに差し替えられず、リポジトリ実装の変更がコントローラーに波及する。

**改善**：`IUserRepository` インターフェイスをコントローラーに注入（DI）する。実装クラスはDIコンテナが解決する。コントローラーはインターフェイスのみに依存する。

**問題②：UserRepository がメール送信処理を含んでいる**  
単一責任原則（SRP）違反。リポジトリの責務は「集約の永続化（追加・削除・取得）」のみ。メール送信はリポジトリの責務外。

**改善**：メール送信はドメインイベント（`UserRegistered`）を発行し、別のイベントハンドラー（`WelcomeEmailSender`）がメール送信を担当する。またはアプリケーションサービスが `IEmailNotificationService` を呼ぶ形にする。
