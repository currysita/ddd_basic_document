## D2-01　ドメインサービスとアプリケーションサービスの違い（知識確認）

**問い**

ドメインサービスとアプリケーションサービスをそれぞれ定義し、どの層に位置するかとともに違いを説明してください。

---

**模範解答**

| | ドメインサービス | アプリケーションサービス |
|---|---|---|
| **定義** | ドメインモデルの一部として、単一のエンティティ/値オブジェクトには自然に割り当てられないドメイン固有のビジネスロジックを担う | ドメインモデルを利用するクライアント。ユーザーとのやり取りを調整・制御し、トランザクション管理やユースケースを担う |
| **所属する層** | ドメイン層 | アプリケーション層 |
| **ドメイン知識** | 持つ | 持たない（ドメイン層に委譲する） |
| **主な役割** | 複数のドメインオブジェクトを組み合わせた計算・変換・操作 | ユースケースの調整、トランザクション管理 |

## D2-02　貧血ドメインモデルの定義と問題点（知識確認）

**問い**

貧血ドメインモデルとは何か、なぜ問題になるのかを説明してください。

---

**模範解答**

**貧血ドメインモデル（Anemic Domain Model）**：エンティティや値オブジェクトがビジネスロジックを持たず、ただデータを保持するだけの状態になってしまうアンチパターン。

**なぜ問題か**

- ビジネスロジックがドメインモデルの外（アプリケーション層・ドメインサービスなど）に散在する
- エンティティがデータコンテナと化し、DDDが意図するモデル本位の設計にならない
- ドメインエキスパートと共有できるモデルが育たず、ユビキタス言語を体現できなくなる
- 結果として、DDDではなくトランザクションスクリプトと同等の設計になる

## D2-03　セパレートインターフェイスパターンの目的（知識確認）

**問い**

セパレートインターフェイスパターンとは何か、DDDでドメインサービスに適用する目的を説明してください。

---

**模範解答**

**セパレートインターフェイス**：Martin Fowler が提唱したパターン。インターフェイスと実装を別のパッケージ（層）に分離する手法。

**DDDでの適用目的**

ドメイン層にインターフェイスを定義し、実装をインフラストラクチャ層に置くことで、ドメイン層がインフラストラクチャ技術に依存しない設計を実現する。

例：外部サービスとの通信が必要なドメインサービスを、インターフェイスをドメイン層に定義・実装をインフラ層に分離することで、ドメイン層のピュアさを保てる。

なお、インターフェイスを分離する必要がない場合は実装クラスをそのままドメイン層に置くこともある。

## D2-04　ミニレイヤとは何か（知識確認）

**問い**

「ミニレイヤ」とは何か、なぜ避けるべきかを説明してください。

---

**模範解答**

**ミニレイヤ**：ドメインサービスが誤った形で使われることで生まれる設計上の問題。ビジネスロジックをドメインサービスに集中させすぎることで、エンティティや値オブジェクトがただのデータコンテナ（貧血ドメインモデル）と化す。

**なぜ避けるべきか**

- ドメインモデルが薄くなり、DDDの本来の目的（モデルにビジネスロジックを集約する）が達成できない
- 100ものトランザクションスクリプトをドメインサービスとして登録するような状態になると、ドメインサービスがただの手続き呼び出しの集合になる
- ユビキタス言語がビジネスロジックを記述しきれていない場合に起きることが多い

## D2-05　なぜ複数オブジェクトを扱うロジックをドメインサービスに実装するのか（理解確認）

**問い**

複数のエンティティや値オブジェクトを組み合わせたビジネスロジックをドメインサービスに実装する理由を説明してください。エンティティのメソッドに実装する場合の問題点と比較して述べてください。

---

**模範解答**

**エンティティのメソッドに実装した場合の問題**

複数のエンティティや値オブジェクトを組み合わせた処理をエンティティの一つに実装すると、そのエンティティが他のエンティティ（リポジトリを含む）に強く依存してしまい、設計が不自然になる。

例えば「複数のBacklogItemの優先度合計」を`Product`エンティティのメソッドで実装するには、`Product`が`BacklogItemRepository`を知る必要がある。これは`Product`の責務を超えている。

**ドメインサービスに実装する理由**

- 「複数オブジェクトを組み合わせる処理」という責務を独立したドメインサービスに持たせることで、各エンティティの責務をシンプルに保てる
- 処理の凝集度が高まり、計算ロジックが一箇所に集まる
- Evans氏も「エンティティや値オブジェクトに自然に割り当てられないビジネスロジックはドメインサービスに移すべき」と説明している

---

## D2-06　ドメインサービスの多用が「ミニレイヤ」や「貧血ドメインモデル」につながる理由（理解確認）

**問い**

ドメインサービスを多用・誤用すると、なぜ「ミニレイヤ」や「貧血ドメインモデル」につながるのかを説明してください。

---

**模範解答**

ドメインサービスが本来エンティティや値オブジェクトに属すべきビジネスロジックも担いはじめると、エンティティ・値オブジェクトはデータを持つだけのコンテナになる（貧血ドメインモデル）。

ビジネスロジックがドメインサービスに集中しすぎると、ドメインサービスが事実上の「手続きの集合」（トランザクションスクリプト）になる。これが「ミニレイヤ」と呼ばれる問題。

**正しい使い方との違い**

| 正しい使い方 | 誤った使い方（ミニレイヤ） |
|---|---|
| 「複数オブジェクト間の計算・変換」のみドメインサービスに | 単一エンティティの操作もドメインサービスに書く |
| エンティティが自身のビジネスロジックを持つ | エンティティはデータだけ持ち、ロジックはサービスに |

## D2-07　セパレートインターフェイスがドメイン層の独立性を確保する仕組み（理解確認）

**問い**

セパレートインターフェイスを適用することで、ドメイン層がインフラストラクチャ層に依存しなくなる仕組みを説明してください。

---

**模範解答**

セパレートインターフェイスなしでは、ドメイン層のサービスが外部技術（DB・メッセージング・外部APIなど）を直接使う実装クラスに依存してしまう。

**セパレートインターフェイスを適用した仕組み**

```
ドメイン層：IMemberInvitationsService（インターフェイス）を定義
インフラ層：MemberInvitationsServiceImpl（実装クラス）がインターフェイスを実装
```

ドメイン層はインターフェイスにのみ依存する。実装（インフラ技術）はインフラ層に封じ込められる。DIコンテナが実行時に実装クラスを注入するため、ドメイン層はどの実装が使われるかを知らなくてよい。

## D2-08　ドメインサービスとアプリケーションサービスの責任境界が曖昧になるケースの分析（理解確認）

**問い**

「ユーザー登録時に、メールアドレスが既存ユーザーと重複していないかを確認する」という処理について、ドメインサービスとアプリケーションサービスのどちらに実装すべきかを判断し、理由を説明してください。

---

**模範解答**

**ドメインサービスに実装すべき**

理由：「メールアドレスが重複しているかどうか」はドメインルール（ビジネスルール）であり、ドメイン層に属する。リポジトリを使って他のユーザーエンティティを検索する処理であり、複数のドメインオブジェクトを組み合わせた判断になるため、ドメインサービスが適切。

```java
// ドメインサービス
public class UserRegistrationService {
    private UserRepository userRepository;
    
    public boolean isEmailAvailable(Email email) {
        return !userRepository.existsByEmail(email); // ドメインルールをここに
    }
}
```

アプリケーションサービスに実装してしまうと、ドメインルールがアプリケーション層に流出し、貧血ドメインモデルにつながる。

## D2-09　DIコンテナを使ったドメインサービスのインスタンス生成設計（理解確認）

**問い**

ドメインサービスのインスタンス生成に、DIコンテナを使う方法とコンストラクタで直接渡す方法を比較し、それぞれの適用場面を説明してください。

---

**模範解答**

**コンストラクタ（またはメソッド引数）で直接渡す方法**

- シンプルな依存関係のとき
- テストコードでモックを注入しやすい
- 依存が少ない場合や一時的な利用に適する

```java
// 呼び出し元がインスタンスを作って渡す
BusinessPriorityCalculator service = new BusinessPriorityCalculator(repository);
service.businessPriorityTotals(tenantId, productId);
```

**DIコンテナを使う方法**

- 依存関係が多いか、複数の場所から使われる場合
- 環境（テスト/本番）によって実装を切り替えたい場合
- Spring（Java）やUnity Container（.NET）が代表例
- 推奨される方法。依存関係の管理が自動化される

```java
// DIコンテナが自動的に依存を解決して注入する
@Autowired
BusinessPriorityCalculator calculator;
```

## D2-10　エンティティ/値オブジェクトに実装するかドメインサービスに実装するかの判断（応用）

**問い**

以下のビジネスロジックについて、エンティティ/値オブジェクトのメソッドに実装すべきか、ドメインサービスに実装すべきかを判断し、理由を説明してください。

- **A. ユーザーのパスワードを変更する（旧パスワードを確認して新パスワードに変更）**
- **B. 複数の商品注文の合計金額を計算する**
- **C. 注文をキャンセルする（在庫を戻す処理を含む）**

---

**模範解答**

| ロジック | 実装場所 | 理由 |
|---|---|---|
| **A. パスワード変更** | **Userエンティティのメソッド** | 自身のパスワードを変更する操作は`User`エンティティ内で完結する。他のオブジェクトに依存しない |
| **B. 合計金額計算** | **ドメインサービス** | 複数の`OrderItem`を組み合わせた計算。1つのエンティティに持たせると不自然。`OrderSummaryCalculator`などのドメインサービスが適切 |
| **C. 注文キャンセル（在庫戻し含む）** | **ドメインサービス** | `Order`エンティティと`Inventory`エンティティの両方を操作する。単一エンティティの責務を超えており、ドメインサービスが自然 |

## D2-11　ドメインサービスとアプリケーションサービスの責任分離を設計する（応用）

**問い**

「新規注文確定」ユースケースで「注文が有効かチェック → 在庫引き当て → 決済実行 → 確認メール送信」という処理があります。どの処理をドメインサービスに、どの処理をアプリケーションサービスに割り当てるかを設計してください。

---

**模範解答**

**ドメインサービスに割り当てる処理**

- **注文が有効かチェック**：注文に含まれる商品・数量・金額の整合性チェックはドメインルール。`OrderValidationService`が担当
- **在庫引き当て**：`Order`と`Inventory`の両方を操作するドメイン処理。`InventoryAllocationService`が担当

**アプリケーションサービスに割り当てる処理**

- **決済実行**：外部決済サービスとの連携はアプリケーション層の責務。技術的な関心事
- **確認メール送信**：インフラ的な通知処理。アプリケーション層からインフラ層（メール送信サービス）を呼び出す
- **トランザクション管理**：ユースケース全体のトランザクション境界はアプリケーションサービスが管理

```
アプリケーションサービス：
  1. orderValidationService.validate(order)  // DSに委譲
  2. inventoryAllocationService.allocate(order)  // DSに委譲
  3. paymentService.execute(order)  // 外部サービス呼び出し
  4. emailService.sendConfirmation(order)  // インフラ呼び出し
```

## D2-12　既存コードの貧血ドメインモデルを検出してリファクタリング方針を立てる（応用）

**問い**

以下のコードを見て、貧血ドメインモデルの問題を指摘し、リファクタリング方針を説明してください。

```java
// アプリケーションサービス
public class OrderApplicationService {
    public void confirmOrder(String orderId) {
        Order order = orderRepository.findById(orderId);
        
        // 注文確定のビジネスロジックがアプリケーション層にある
        if (order.getStatus() != OrderStatus.DRAFT) {
            throw new Exception("下書き状態の注文のみ確定できます");
        }
        if (order.getItems().isEmpty()) {
            throw new Exception("商品が1件以上必要です");
        }
        order.setStatus(OrderStatus.CONFIRMED);
        orderRepository.save(order);
    }
}
```

---

**模範解答**

**問題点**

- 「下書き状態のみ確定可能」「商品が1件以上必要」というドメインルールがアプリケーション層に書かれている（ドメインロジックの流出）
- `Order`エンティティがデータコンテナになっており、ビジネスルールを持っていない（貧血ドメインモデル）

**リファクタリング方針**

```java
// Order エンティティにビジネスロジックを移す
public class Order {
    public void confirm() {
        if (this.status != OrderStatus.DRAFT) {
            throw new DomainException("下書き状態の注文のみ確定できます");
        }
        if (this.items.isEmpty()) {
            throw new DomainException("商品が1件以上必要です");
        }
        this.status = OrderStatus.CONFIRMED;
    }
}

// アプリケーションサービスは薄くなる
public class OrderApplicationService {
    public void confirmOrder(String orderId) {
        Order order = orderRepository.findById(orderId);
        order.confirm(); // ドメインに委譲
        orderRepository.save(order);
    }
}
```

## D2-13　複数の集約にまたがる計算ロジックをドメインサービスとして設計する（応用）

**問い**

プロジェクト管理システムで「バックログアイテムのビジネス優先度合計を計算する」処理を、`BusinessPriorityCalculator`ドメインサービスとして設計してください。リポジトリの使い方も含めて説明してください。

---

**模範解答**

```java
// ドメインサービス
public class BusinessPriorityCalculator {
    private BacklogItemRepository backlogItemRepository;
    
    public BusinessPriorityCalculator(BacklogItemRepository repository) {
        this.backlogItemRepository = repository;
    }
    
    public BusinessPriorityTotals businessPriorityTotals(
            TenantId tenantId, ProductId productId) {
        
        // リポジトリでバックログアイテムを取得
        Collection<BacklogItem> items =
            backlogItemRepository.allOutstandingProductBacklogItems(tenantId, productId);
        
        int totalBenefit = 0, totalPenalty = 0, totalCost = 0, totalRisk = 0;
        
        for (BacklogItem item : items) {
            if (item.hasBusinessPriority()) {
                BusinessPriorityRatings r = item.businessPriority().ratings();
                totalBenefit += r.benefit();
                totalPenalty += r.penalty();
                totalCost += r.cost();
                totalRisk += r.risk();
            }
        }
        
        return new BusinessPriorityTotals(
            totalBenefit, totalPenalty,
            totalBenefit + totalPenalty,
            totalCost, totalRisk);
    }
}
```

**設計ポイント**：リポジトリをコンストラクタ注入で受け取り、複数の`BacklogItem`の値を集計して`BusinessPriorityTotals`値オブジェクトを返す。単一のエンティティに持たせると不自然なロジックをドメインサービスに集約した形。

## D2-14　ドメインサービスのインターフェイスをドメイン層に配置する設計判断（応用）

**問い**

「招待メールの送信」という処理をドメインサービスとして設計します。この処理には外部メールサービスとの通信が必要です。セパレートインターフェイスパターンを使った設計を説明してください。

---

**模範解答**

**セパレートインターフェイスを使った設計**

```
ドメイン層
  IMemberInvitationsService（インターフェイス）
  → sendInvitation(tenantId, invitee) メソッドを定義

インフラストラクチャ層
  EmailMemberInvitationsService（実装クラス）
  → IMemberInvitationsService を implements
  → 実際のメール送信ロジックを実装（外部メールAPIを呼ぶ）
```

**DIコンテナで注入**

```java
// ドメイン層はインターフェイスにのみ依存
public class SomeApplicationService {
    private IMemberInvitationsService invitationsService; // インターフェイス
    
    public SomeApplicationService(IMemberInvitationsService service) {
        this.invitationsService = service;
    }
}
// DIコンテナがEmailMemberInvitationsServiceを注入する
```

**利点**：ドメイン層が外部メールAPIに直接依存しない。テスト時はモック実装を注入できる。

## D2-15　ドメインサービスが肥大化した場合の分割方針（応用）

**問い**

注文管理システムの`OrderService`が以下の責務を持つほど肥大化しています。単一責任原則（SRP）に基づいて分割方針を提案してください。

- 注文の有効性チェック
- 在庫引き当て計算
- 注文合計金額計算
- 割引適用ルールの判定

---

**模範解答**

**分割方針**：各責務を独立したドメインサービスに分ける。

| 責務 | 新しいドメインサービス |
|---|---|
| 注文の有効性チェック | `OrderValidationService` |
| 在庫引き当て計算 | `InventoryAllocationService` |
| 注文合計金額計算 | `OrderAmountCalculator` |
| 割引適用ルールの判定 | `DiscountRuleService` |

**SRPの適用理由**：「クラスを変更する理由は1つだけであるべき」。在庫引き当てのロジックが変わったときに、割引ロジックには無関係なはずの変更が同じクラスに触れることになるのは問題。独立したクラスにすることで、各サービスの変更が他のサービスに影響しない。

## D2-16　外部サービスとの連携をドメインサービスとして抽象化する設計（応用）

**問い**

「与信チェックを外部クレジットサービスに委ねる」という要件があります。ドメインサービスとしてどのように設計するかを説明してください。セパレートインターフェイスの活用を含めること。

---

**模範解答**

**設計**

```
ドメイン層
  ICreditCheckService（インターフェイス）
  → checkCredit(customerId, amount): CreditCheckResult

インフラストラクチャ層
  ExternalCreditCheckService（実装クラス）
  → ICreditCheckService を実装
  → 実際の外部クレジットAPIを呼び出す
  → レスポンスをCreditCheckResult（ドメインの値オブジェクト）に変換
```

**ポイント**

- ドメイン層のインターフェイスに登場する型（`CreditCheckResult`）はドメインの値オブジェクト。外部APIのレスポンスをそのまま持ち込まない
- インフラ層が外部APIのレスポンスを`CreditCheckResult`に変換（ACLの役割も兼ねる）
- ドメインロジックがどの外部サービスを使うかを知らなくて済む

## D2-17　ドメインサービスのテスタビリティを高める設計（応用）

**問い**

`BusinessPriorityCalculator`ドメインサービスのテストを書くとき、リポジトリ（DB）に依存しないでテストするにはどうすればよいかを説明してください。

---

**模範解答**

**アプローチ：リポジトリをモック（またはスタブ）で置き換える**

コンストラクタ注入でリポジトリを受け取る設計にすることで、テスト時はモックリポジトリを渡せる。

```java
// テストコード
@Test
public void testBusinessPriorityTotals() {
    // モックリポジトリを用意
    BacklogItemRepository mockRepo = new InMemoryBacklogItemRepository();
    mockRepo.add(new BacklogItem(...)); // テスト用データを追加

    BusinessPriorityCalculator calculator =
```

## D2-18　アプリケーションサービスにドメインロジックが流出しているコードの改善（応用）

**問い**

以下のアプリケーションサービスコードにドメインロジックが流出しています。問題を指摘し、改善してください。

```java
public class ProductApplicationService {
    public void applyDiscount(String productId, double discountRate) {
        Product product = productRepository.findById(productId);
        
        // ドメインロジックがアプリケーション層に流出
        if (discountRate < 0 || discountRate > 0.5) {
            throw new Exception("割引率は0〜50%の範囲です");
        }
        double newPrice = product.getPrice() * (1 - discountRate);
        product.setPrice(newPrice);
        productRepository.save(product);
    }
}
```

---

**模範解答**

**問題点**：「割引率は0〜50%の範囲」「最終価格の計算」というドメインルールがアプリケーション層に書かれている。

**改善策**

```java
// Product エンティティにビジネスロジックを移す
public class Product {
    public void applyDiscount(double discountRate) {
        if (discountRate < 0 || discountRate > 0.5) {
            throw new DomainException("割引率は0〜50%の範囲です");
        }
        this.price = this.price * (1 - discountRate);
    }
}

// アプリケーションサービスは薄くなる
public class ProductApplicationService {
    public void applyDiscount(String productId, double discountRate) {
        Product product = productRepository.findById(productId);
        product.applyDiscount(discountRate); // ドメインに委譲
        productRepository.save(product);
    }
}
```

## D2-19　単一責任原則に基づいたドメインサービスの粒度設計（応用）

**問い**

「ユーザー管理ドメインサービス」が「ユーザー認証」「ユーザー登録」「ユーザー無効化」「メール重複チェック」の4つの処理を1つのクラスで持っています。SRPに基づいて粒度を設計し直してください。

---

**模範解答**

**変更理由の分析**

| 処理 | 変更理由 |
|---|---|
| ユーザー認証 | 認証方式（パスワード/OAuth）の変更 |
| ユーザー登録 | 登録フローやバリデーションルールの変更 |
| ユーザー無効化 | 無効化ポリシー（理由の種類など）の変更 |
| メール重複チェック | 重複チェックのロジック・スコープの変更 |

各処理の変更理由が異なるため、それぞれ独立したドメインサービスにする。

**分割後の設計**

| ドメインサービス | 責務 |
|---|---|
| `UserAuthenticationService` | ユーザー認証 |
| `UserRegistrationService` | ユーザー登録（メール重複チェックを内包してよい） |
| `UserDeactivationService` | ユーザー無効化 |

メール重複チェックは登録フローの一部なので`UserRegistrationService`内に含めるか、共通の`UserUniquenessPolicy`値オブジェクトとして切り出す選択もある。

## D2-20　ドメインサービスとファクトリの責務を区別する判断（応用）

**問い**

「新規プロジェクトを作成する際に、デフォルトのタスクテンプレートを設定する」という処理をドメインサービスとファクトリのどちらで実装するかを判断し、理由を説明してください。

---

**模範解答**

**ファクトリに実装すべき**

理由：「新規プロジェクトを作成する」という生成の責務はファクトリが担う。デフォルトのタスクテンプレートの設定は生成時の初期化ロジックであり、生成を担うファクトリに自然に収まる。

```java
// ファクトリ
public class ProjectFactory {
    private TaskTemplateRepository templateRepository;
    
    public Project createProject(ProjectId id, String name) {
        TaskTemplate defaultTemplate = templateRepository.findDefault();
        return new Project(id, name, defaultTemplate); // 生成時に設定
    }
}
```

ドメインサービスは「複数のオブジェクトを組み合わせた計算・変換・操作」を担う。単純なオブジェクト生成はファクトリ（第11章）の役割。

## D2-21　ドメインサービスが必要かエンティティのメソッドで十分かを判断するシナリオ問題（応用）

**問い**

以下の各処理について、エンティティのメソッドに実装するか、ドメインサービスに実装するかを判断してください。

- **A. `Order.cancel()`：注文をキャンセルする（ステータス変更のみ）**
- **B. キャンセルされた注文に対して在庫を元に戻す処理（`Order` + `Inventory` の両方を操作）**
- **C. `User.changeEmail(newEmail)`：自分のメールアドレスを変更する**
- **D. 同一ユーザーが複数注文の合計額が10万円以上なら優良顧客とする判定**

---

**模範解答**

| 処理 | 実装場所 | 理由 |
|---|---|---|
| **A. `Order.cancel()`** | **`Order`エンティティのメソッド** | `Order`自身の状態変更。他のオブジェクトに依存しない |
| **B. キャンセル時の在庫戻し** | **ドメインサービス** | `Order`と`Inventory`の両方を操作する。単一エンティティの責務を超えている |
| **C. `User.changeEmail()`** | **`User`エンティティのメソッド** | `User`自身のメール変更。自分自身の操作 |
| **D. 優良顧客判定（複数注文の合計）** | **ドメインサービス** | 複数の`Order`を取得して合計する処理。`User`エンティティが`OrderRepository`を持つのは不自然 |

**判断軸**：「単一エンティティが持つ情報だけで完結する処理か」「複数のエンティティ・リポジトリが必要か」。
