## F2-01【知識確認】アプリケーション層の定義と役割

**問題**（用語問題）

DDDにおける「アプリケーション層（アプリケーションサービス）」を定義し、主な3つのタスクを答えよ。

**模範解答**

**定義**：ドメインモデルの調整役として機能するクライアント層。ドメインロジック自体は持たず、UI層とドメイン層を仲介する。

**主な3つのタスク**：
1. ユーザーの行動（ユースケース）に応じた調整役を実施する
2. リポジトリから集約を取り出し、ドメインのコマンドを実行する（またはドメインサービスを利用する）
3. データベーストランザクションの境界を管理し、最後にリポジトリに集約を保存する

---

## F2-02【知識確認】UI層の種類

**問題**（用語問題）

DDDにおける「UI層」に含まれるソフトウェアの種類を4つ答えよ。

**模範解答**

1. サーバー側で処理するWebシステム
2. ブラウザ側で描画するWeb APIサービス
3. WinFormsやSWTやSwingといったネイティブGUIクライアント
4. スマートフォンアプリケーション

## F2-03【知識確認】ドメインモデルをUI層へ渡す3つの方法

**問題**（用語問題）

ドメイン層の情報をUI層へ渡す3つの方法を答えよ。

**模範解答**

1. **複数の集約情報をDTOに詰め替え**：プレゼンテーション専用のデータ構造（DTO）に変換してUIに渡す。ドメインモデルの内部構造をUIに漏らさない（推奨）
2. **集約IDのみを送信**：UIがドメインオブジェクトを直接操作できる場合に、DTOに集約IDのみを含める
3. **ユースケースに最適化したクエリによる取得**：CQRSのクエリサイドを用いてUIが必要なデータを直接取得する。ドメインモデルを介さない読み取り専用データモデル

## F2-04【知識確認】DTO（Data Transfer Object）

**問題**（用語問題）

「DTO（Data Transfer Object）」とは何か。なぜドメインモデルを直接UIに公開せずDTOを使うのか理由を述べよ。

**模範解答**

**定義**：ドメインモデルの情報をUI層に渡すためのデータ転送用オブジェクト。プレゼンテーション専用のデータ構造であり、ドメインモデルの内部構造を持たない。

**DTOを使う理由**：
- ドメインモデルの内部構造をUIに漏らさないため（関心の分離）
- セキュリティ：不要なフィールドを含まない最小限のデータのみ公開する
- ドメインモデルを変更してもUIへの影響を局所化できる（変更の波及を防ぐ）
- バリデーションをDTO側で別途行える

## F2-05【知識確認】コマンドパターン

**問題**（用語問題）

アプリケーションサービスで使われる「コマンドパターン」を説明し、メリットを答えよ。

**模範解答**

**説明**：アプリケーションサービスへの入力を `XxxCommand` クラス（例：`ProvisionTenantCommand`）として表現するパターン。ユースケースに必要最小限のデータをカプセル化する。

**メリット**：
1. メソッドの引数が増えすぎることを防ぐ（パラメータをコマンドクラスに集約）
2. コマンドオブジェクトが何のユースケースに対応するかを明示的に表現する
3. バリデーションをコマンドクラスに閉じ込められる
4. コマンドのシリアライズ・デシリアライズが容易（メッセージング統合でも活用できる）

## F2-06【知識確認】プレゼンテーションモデルとビューの違い

**問題**（用語問題）

「ビュー（View）」と「プレゼンテーションモデル（Presentation Model）」の違いを説明せよ。

**模範解答**

| 観点 | ビュー | プレゼンテーションモデル |
|------|--------|----------------------|
| 役割 | UIの表示（描画）を担当 | UIの状態とUI固有ロジックを持つ |
| 例 | HTMLテンプレート、画面描画コード | MVVMのViewModel相当 |
| 関係 | プレゼンテーションモデルの状態に基づいてレンダリング | ビューに渡すデータを管理する |

## F2-07【知識確認】ドメインモデルをUIに公開する際のトレードオフ

**問題**（用語問題）

ドメインモデルをUI層に直接公開した場合と、DTOを使って隠蔽した場合のトレードオフを4つの観点で比較せよ。

**模範解答**

| 観点 | 公開あり（直接公開） | 公開なし（DTO使用） |
|------|-----------------|----------------|
| 分離度 | 低い（ドメインとUIが密結合） | 高い（独立して変更可能） |
| 実装コスト | 小さい（変換不要） | 大きい（DTO・アセンブラが必要） |
| セキュリティ | 漏洩リスクあり | 必要な情報のみ公開できる |
| テスト容易性 | やや困難 | 高い（変換ロジックを独立してテスト可能） |
| バリデーション | ドメイン側で対応 | DTOで別途対応できる |
| ドメイン変更の影響 | UIへの波及大 | UIへの波及小 |

## F2-08【理解確認】アプリケーションサービスとドメインサービスの責務分担

**問題**（読解判断問題）

以下のコードを見て、どのロジックがアプリケーションサービスにあり、どのロジックがドメイン層にあるべきかを判断せよ。

```java
@Transactional
public void provisionTenant(ProvisionTenantCommand aCommand) {
    return this.tenantProvisioningService().provisionTenant(
        aCommand.getTenantName(),
        aCommand.getTenantDescription(),
        new FullName(aCommand.getAdministratorFirstName(), aCommand.getAdministratorLastName()),
        new EmailAddress(aCommand.getAdministratorEmailAddress()),
        ...
    );
}
```

**模範解答**

**アプリケーションサービス（このコードが担うもの）**：
- コマンドオブジェクト（`ProvisionTenantCommand`）からパラメータを取り出してドメインサービスに渡す調整役
- `@Transactional` によるトランザクション境界の管理

**ドメイン層（`tenantProvisioningService` が担うもの）**：
- テナントのプロビジョニングに関するビジネスルール（何が有効なテナントか、どの集約を生成すべきかなど）
- `FullName`・`EmailAddress` などの値オブジェクト生成はアプリケーション層で行っているが、これらの値オブジェクトの不変条件検証はドメインのコンストラクタが担う

→ アプリケーションサービスは「何を」「どの順に」呼ぶかの調整のみ行い、「どのように」は常にドメイン層に委ねる。

## F2-09【理解確認】レイヤードアーキテクチャとヘキサゴナルアーキテクチャの役割

**問題**（読解判断問題）

第14章では「レイヤードアーキテクチャ」と「ヘキサゴナルアーキテクチャ」の2つが紹介されている。いずれのアーキテクチャもドメイン層を守るという点で共通している。それぞれがどのようにドメイン層を守るか説明せよ。

**模範解答**

**レイヤードアーキテクチャ**：UI層→アプリケーション層→ドメイン層→インフラストラクチャ層という階層構造により、各層は下位層にのみ依存する。ドメイン層はUI層・アプリケーション層から直接参照されるが、インフラ層には依存しない。

**ヘキサゴナルアーキテクチャ（ポート&アダプタ）**：ドメインを中心に置き、外部（UI、DB、メッセージングなど）とポート（インターフェイス）で接続する。ドメインは外部の技術（DBの種類、UIフレームワーク）を一切知らない。アダプターが外部の技術とポートを繋ぐ。

どちらも「ドメイン層がインフラや技術の詳細に依存しない」ことを保証し、ドメインモデルをクリーンに保つ。

## F2-10【理解確認】複数コンテキストにまたがる場合の2パターン

**問題**（読解判断問題）

1つのUI操作が複数の境界づけられたコンテキストに影響を与える場合、IDDDが示す2つの統合パターンとそれぞれのトレードオフを説明せよ。

**模範解答**

**パターン1（直接呼び出し）**：1つのアプリケーションサービスが複数コンテキストに対して直接コマンドを実行する。

- 利点：シンプルで理解しやすい
- 欠点：アプリケーションサービスが複数コンテキストに依存する密結合。一方のコンテキストが障害の場合、全体が止まる

**パターン2（イベント駆動）**：コンテキスト側が結果をドメインイベントとして発行し、他コンテキストがイベントを購読して連携する。

- 利点：疎結合。各コンテキストが独立して動作し、片方の障害が他に波及しない。トランザクション管理は各コンテキスト側に委ねられる
- 欠点：非同期のため即時一貫性は保証されない（結果整合性）

## F2-11【理解確認】DTOアセンブラ（Transformer）の役割

**問題**（読解判断問題）

`CalendarWebDTOTransformer` のようなDTOアセンブラクラスがなぜ必要か。複数クライアント対応の観点から説明せよ。

**模範解答**

集約からDTOへの変換ロジックをアセンブラクラスに集約することで：

1. **クライアント別の変換を切り替えられる**：Web（JSON）用の `CalendarWebDTOTransformer` とネイティブクライアント用の `CalendarNativeDTOTransformer` を別々に実装し、クライアントの種類に応じて切り替える
2. **ドメインモデルを汚染しない**：集約自体に「JSONに変換するメソッド」などを持たせると、ドメインモデルがUI/フォーマットに依存してしまう。アセンブラを別クラスにすることで関心を分離する
3. **テストが書きやすい**：変換ロジックを独立したクラスにすることで、単体テストが書きやすくなる

## F2-12【理解確認】CQRSの「クエリサイド」とドメインモデルの関係

**問題**（読解判断問題）

「ユースケースに最適化したクエリによる取得」（CQRSのクエリサイド）では、ドメインモデルを介さない。これが適切である理由を説明せよ。

**模範解答**

**適切な理由**：

1. **読み取りはビジネスルールを必要としない**：データを取得してUIに表示するだけの処理では、集約の不変条件やドメインサービスは不要。ドメインモデルを経由するとオーバーヘッドになる
2. **UIに最適化したデータ構造**：ドメインモデルは「書き込み」用に設計されており、複数集約の情報を1画面用に組み合わせるには向いていない。クエリサイドではUIが必要な形に直接クエリできる
3. **パフォーマンス**：ドメインオブジェクトのロードを省略し、SQLやNoSQLクエリを直接実行することでレスポンスを高速化できる

ただし、更新（コマンド）操作は必ずドメインモデルを経由してビジネスルールを保護する。読み取りと書き込みでモデルを分ける（CQRS）ことで双方の要件を満たす。

## F2-13【応用】アプリケーションサービスの設計評価

**問題**（応用）

以下のアプリケーションサービスコードにDDD的な問題点が2つある。指摘せよ。

```java
@Transactional
public void updateUserRole(String userId, String newRole) {
    User user = userRepository.findById(userId);
    // ビジネスルール：管理者は削除できない
    if (user.getRole().equals("ADMIN") && newRole.equals("")) {
        throw new IllegalStateException("管理者は削除できません");
    }
    user.setRole(newRole);
    userRepository.save(user);
}
```

**模範解答**

**問題点①：ビジネスルールがアプリケーション層に漏れている**  
「管理者は削除できない」というビジネスルールは `User` エンティティまたはドメインサービスが持つべき。アプリケーションサービスはユースケースの調整役であり、ドメインルールを直接記述すべきではない。改善：`user.changeRole(newRole)` というメソッドを `User` エンティティに持たせ、エンティティ内でルールを検証する。

**問題点②：`setRole` という直接セッターの使用**  
`user.setRole(newRole)` はTell-Don't-Ask原則に違反する。エンティティの内部状態を直接変更するのではなく、ドメインの意図を表すメソッド名（`user.changeRole(role)` や `user.revokeAdminPrivileges()`）を使うべき。

## F2-14【応用】コマンドパターンの実装設計

**問題**（応用）

「ユーザーのロールを変更する」ユースケースのためのコマンドクラスとアプリケーションサービスメソッドを設計せよ（C#またはJavaで）。

**模範解答**

```csharp
// コマンドクラス
public class ChangeUserRoleCommand {
    public string TenantId { get; }
    public string UserId { get; }
    public string NewRole { get; }

    public ChangeUserRoleCommand(string tenantId, string userId, string newRole) {
        TenantId = tenantId;
        UserId = userId;
        NewRole = newRole;
    }
}

// アプリケーションサービス
public class UserApplicationService {
    private readonly IUserRepository _userRepository;

    public UserApplicationService(IUserRepository userRepository) {
        _userRepository = userRepository;
    }

    [Transactional]
    public void ChangeUserRole(ChangeUserRoleCommand command) {
        var tenantId = new TenantId(command.TenantId);
        var userId = new UserId(command.UserId);
        var user = _userRepository.UserWithId(tenantId, userId);
        // ドメインロジックはエンティティに委譲
        user.ChangeRole(new Role(command.NewRole));
        _userRepository.Save(user);
    }
}
```

ポイント：コマンドクラスには最小限のプリミティブ型データのみ持たせ、値オブジェクトへの変換はアプリケーションサービス内で行う。

## F2-15【応用】DTOとドメインモデルの分離設計

**問題**（応用）

「プロジェクト一覧画面」で表示したい情報（プロジェクト名、メンバー数、最新タスク名）が複数の集約にまたがっている場合、どのアプローチで取得してUIに渡すか設計せよ。

**模範解答**

**推奨アプローチ：ユースケース最適化クエリ（CQRSクエリサイド）**

```
// クエリ専用のDTO
public class ProjectSummaryDto {
    public string ProjectId { get; set; }
    public string ProjectName { get; set; }
    public int MemberCount { get; set; }
    public string LatestTaskName { get; set; }
}

// クエリサービス（読み取り専用・ドメインモデルを介さない）
public class ProjectQueryService {
    public List<ProjectSummaryDto> GetProjectSummaries(string tenantId) {
        // SQLで必要な情報を1クエリで取得（JOIN/サブクエリ）
        return _db.Query<ProjectSummaryDto>(
            "SELECT p.Id, p.Name, COUNT(m.Id) as MemberCount, " +
            "  (SELECT TOP 1 Name FROM Tasks WHERE ProjectId = p.Id ORDER BY CreatedAt DESC) " +
            "FROM Projects p JOIN Members m ON p.Id = m.ProjectId " +
            "WHERE p.TenantId = @tenantId GROUP BY p.Id, p.Name",
            new { tenantId });
    }
}
```

ドメインモデルを経由せず直接クエリすることで、複数集約にまたがる一覧表示のパフォーマンスとシンプルさを両立する。更新操作は必ずドメインモデルを経由する（CQRS）。

## F2-16【応用】アプリケーション層の全体設計レビュー

**問題**（応用）

以下のアーキテクチャ設計をレビューし、DDD・クリーンアーキテクチャの観点から問題点を指摘し、改善案を示せ。

現状：
- UI層がリポジトリを直接呼び出している
- アプリケーションサービスがドメインオブジェクトを直接JSONにシリアライズしてUIに返している
- トランザクション管理がリポジトリ内に記述されている

**模範解答**

**問題点①：UI層がリポジトリを直接呼び出している**  
UI層はアプリケーション層（アプリケーションサービス）を経由してドメインモデルにアクセスすべき。UIが直接リポジトリを操作するとビジネスルールがバイパスされ、トランザクション管理も不明確になる。

**改善**：UIはアプリケーションサービスのメソッドのみを呼ぶ。リポジトリはアプリケーションサービス（またはドメインサービス）経由でのみアクセスする。

**問題点②：ドメインオブジェクトをそのままJSONシリアライズしている**  
ドメインモデルの内部構造がAPIレスポンスに露出し、ドメインモデルの変更がAPI仕様の破壊的変更になる。セキュリティ上も不要なフィールドが漏れるリスクがある。

**改善**：アプリケーションサービスがドメインオブジェクトをDTOに変換し（DTOアセンブラを使用）、DTOをUIに返す。

**問題点③：トランザクション管理がリポジトリ内にある**  
トランザクション境界はアプリケーション層（アプリケーションサービス）が管理すべき。リポジトリはストレージアクセスのみを担う。

**改善**：アプリケーションサービスに `@Transactional` または Unit of Work パターンを適用し、リポジトリからトランザクション管理を取り除く。
