## E3-01【知識確認】リポジトリの定義

**問題**（用語問題）

DDDにおける「リポジトリ（Repository）」を定義し、ファーストクラスオブジェクトとして扱う理由を述べよ。

**模範解答**

**定義**：集約の永続化（追加・削除・取得）を担うオブジェクト。クライアント（アプリケーションサービスなど）からデータストアの種類（RDB・NoSQLなど）や保存の詳細を完全に隠蔽し、コレクションを操作するかのようなAPIを提供する。

**ファーストクラスオブジェクトとして扱う理由**：単なるユーティリティクラスではなく、集約の永続化という責務を明確に持つ独立したドメインオブジェクトとして設計することで、ドメインモデルとインフラ層を明確に分離できるため。

## E3-02【知識確認】リポジトリの3つの基本操作

**問題**（用語問題）

リポジトリが提供する3つの基本操作（メソッド）を答えよ。

**模範解答**

1. **追加（add）**：新しい集約をリポジトリに追加する
2. **削除（remove）**：既存の集約をリポジトリから削除する
3. **検索（find系）**：条件に合った集約をリポジトリから取得する（例：`userWithUsername(TenantId, String)`）

## E3-03【知識確認】リポジトリの配置ルール

**問題**（用語問題）

DDDのレイヤ構造において、リポジトリのインターフェイスと実装クラスはそれぞれどの層に配置されるか答えよ。また、その理由（設計原則）も説明せよ。

**模範解答**

- **インターフェイス**：ドメイン層に配置
- **実装クラス**：インフラストラクチャ層に配置

**理由**：依存性逆転の原則（DIP）。ドメイン層がインフラ層に依存するのではなく、インフラ層がドメイン層のインターフェイスに依存する形にすることで、ドメインモデルをデータストアの詳細から切り離せる。テストでもモック実装に差し替え可能になる。

## E3-04【知識確認】トランザクション管理の責務

**問題**（用語問題）

リポジトリはトランザクションを管理するか？正しい責務の分担を説明せよ。

**模範解答**

**リポジトリ自体はトランザクションを管理しない。** トランザクション境界はアプリケーション層（アプリケーションサービス）が管理する。JavaのSpring Frameworkであれば `@Transactional` アノテーションを使い、Spring がトランザクションの開始・コミット・ロールバックを自動管理する。リポジトリはストレージへのアクセスのみを責務とする。

## E3-05【知識確認】コレクション指向リポジトリ

**問題**（用語問題）

「コレクション指向リポジトリ（Collection-Oriented Repository）」の特徴を3点答えよ。

**模範解答**

1. **対象ストレージ**：RDBなどO/Rマッパーを使えるデータストアに適する
2. **saveが不要**：O/Rマッパーの変更追跡（ダーティチェッキング）により、明示的な `save` メソッドなしに変更が自動反映される
3. **addメソッドを使う**：新規追加には `add`、削除には `remove` メソッドを使い、コレクション（リスト）を操作するようなAPIを提供する

---

## E3-06【知識確認】永続指向リポジトリ

**問題**（用語問題）

「永続指向リポジトリ（Persistence-Oriented Repository）」の特徴を3点答えよ。またコレクション指向との最大の違いを述べよ。

**模範解答**

**特徴**：
1. **対象ストレージ**：NoSQLなどのキーバリューストアに適する
2. **集約全体を1レコードで保存**：集約をテーブルに分割せず、1つのドキュメントとしてまとめて保存する
3. **saveメソッドを使う**：変更追跡の仕組みがないため、明示的に `save` メソッドを呼んで保存する

**最大の違い**：コレクション指向は `add`（追加のみ）で変更も自動反映されるが、永続指向は `save`（明示保存）が必要な点。

---

## E3-07【知識確認】ダーティチェッキング

**問題**（用語問題）

「ダーティチェッキング（Dirty Checking）」とは何か。コレクション指向リポジトリにおける役割を説明せよ。

**模範解答**

O/Rマッパー（Hibernateなど）が持つ変更追跡の仕組み。リポジトリから取得した集約に変更があった場合、トランザクション終了時に自動的に変更をデータベースに反映する。これによりコレクション指向リポジトリでは `save` を明示的に呼ばなくても変更が反映されるため、クライアントコードがシンプルになる。

---

## E3-08【知識確認】リポジトリとDAOの違い

**問題**（用語問題）

リポジトリとDAO（Data Access Object）の違いを表形式で説明せよ。

**模範解答**

| 観点 | リポジトリ | DAO |
|------|-----------|-----|
| 責務の主体 | ドメインモデル側（集約単位） | データベース側（テーブル単位） |
| 抽象化の対象 | 集約のコレクション操作 | テーブル・ストレージ操作 |
| O/Rマッパー等 | 利用可能（実装詳細を隠蔽） | 利用可能（O/Rマッパー自体がDAO的） |
| ドメイン知識 | ある（集約の概念を理解） | ない（集約の概念がない） |
| DDD適合性 | 高い | 低い（DDDに不適合） |

## E3-09【理解確認】リポジトリを使った集約操作フロー

**問題**（読解判断問題）

新しい集約をデータベースに登録する場合の操作フロー（4ステップ）を答えよ。

**模範解答**

1. クライアント（アプリケーションサービス）がファクトリに集約の生成を依頼する
2. ファクトリが新しい集約を生成して返す
3. クライアントがリポジトリに集約の追加（`add`）を依頼する
4. リポジトリがデータベースに集約を保存する

※クライアントはどこにデータが格納されるかを意識しない。リポジトリが詳細を隠蔽している。

## E3-10【理解確認】コレクション指向インターフェイスの分析

**問題**（読解判断問題）

以下のインターフェイスを見て、これがコレクション指向リポジトリであることを示す特徴を2点挙げよ。

```java
public interface UserRepository {
    public void add(User aUser);
    public void remove(User aUser);
    public User userWithUsername(TenantId aTenantId, String aUsername);
}
```

**模範解答**

1. **`add` メソッドの使用**：`save` ではなく `add` というメソッド名を使っており、コレクション（リスト）に要素を追加する操作に対応している。O/Rマッパーのダーティチェッキングを前提とした設計
2. **`save` メソッドがない**：更新用の `save` メソッドが存在しない。O/Rマッパーが変更を自動追跡するため、明示的な更新操作が不要であることを前提としている

## E3-11【理解確認】永続指向インターフェイスの分析

**問題**（読解判断問題）

以下のインターフェイスを見て、これが永続指向リポジトリである根拠を説明せよ。

```java
public interface ProductRepository {
    public Product productOfId(TenantId aTenantId, ProductId aProductId);
    public void save(Product aProduct);
    public void remove(Product aProduct);
}
```

**模範解答**

`save` メソッドが存在する点が永続指向の特徴。NoSQLなどのキーバリューストアにはO/Rマッパーのような変更追跡機能がないため、集約を更新する際に明示的に `save` を呼ぶ必要がある。また `add` ではなく `save` という名前は、集約全体を1レコードとして上書き保存する操作を表している。

## E3-12【理解確認】アプリケーションサービスとリポジトリの関係

**問題**（読解判断問題）

以下のJavaコードを読み、アプリケーションサービスがリポジトリを通じて集約を操作するパターンの利点を2点述べよ。

```java
@Transactional
public User registerUser(RegisterUserCommand aCommand) {
    Tenant tenant = this.existingTenant(aCommand.getTenantId());
    User user = tenant.registerUser(
        aCommand.getInvitationIdentifier(),
        aCommand.getUsername(), ...);
    this.userRepository().add(user);
    return user;
}
```

**模範解答**

1. **データストアの詳細隠蔽**：アプリケーションサービスはデータベースの種類（RDB・NoSQLなど）や保存の実装詳細を知らなくてよい。`add(user)` というドメイン的な操作だけを呼べばよい
2. **テスト容易性**：`userRepository` をインターフェイスとして注入（DI）しているため、テスト時にはインメモリのモック実装に差し替えてユニットテストが書ける

## E3-13【理解確認】LevelDBのキー設計

**問題**（読解判断問題）

永続指向リポジトリでNoSQL（LevelDB）を使う場合、IDDDが推奨するキーの構成要素を答えよ。また、その設計の意図を説明せよ。

**模範解答**

**推奨するキー構成要素**：
1. 境界づけられたコンテキストの短縮名（DBの分離）
2. 集約を示す定数（例：`PRODUCT#PK`）
3. 一意な識別子（例：テナントID＋プロダクトID）

**設計の意図**：コンテキストごとにDBを分離し、集約の種類と一意識別子の組み合わせで一意なキーを保証することで、スキーマのない NoSQL 上でも集約を確実に取得・区別できるようにする。

## E3-14【理解確認】リポジトリが「コレクション」を意識する理由

**問題**（読解判断問題）

リポジトリがコレクション（リスト・集合）を操作するかのようなAPIを提供することが推奨される理由を説明せよ。

**模範解答**

アプリケーションサービスからすると、データがどこ（RDB、NoSQL、メモリなど）に保存されているかを意識せず、まるでメモリ上のコレクションを操作するように集約を扱えるべきだという設計思想から。これにより：
- データストアの変更（RDB→NoSQL）の影響をリポジトリ実装クラスに閉じ込められる
- アプリケーションサービスのコードが永続化の詳細から独立し、ドメインロジックに集中できる
- コレクション指向のAPIはドメインモデルの語彙に近く、ユビキタス言語的な表現が可能

---

## E3-15【応用】リポジトリのモックを使ったテスト設計

**問題**（応用）

以下の `UserRepository` インターフェイスを使ったアプリケーションサービスのユニットテストを設計する際、どのようなアプローチでテストを書くか説明せよ。

```java
public interface UserRepository {
    void add(User aUser);
    User userWithUsername(TenantId aTenantId, String aUsername);
}
```

**模範解答**

インメモリ実装（インメモリリポジトリ）またはモックを使う：

1. **インメモリ実装**：`HashMap` などを使った `InMemoryUserRepository` クラスを作り、テスト用に `UserRepository` インターフェイスを実装する。永続化なしでアプリケーションサービスの動作を検証できる
2. **モックフレームワーク**：Mockito（Java）などを使い `UserRepository` をモックし、`add()` が呼ばれたことの検証や `userWithUsername()` の戻り値を指定する
3. **インターフェイス注入（DI）**：アプリケーションサービスのコンストラクタや `@Autowired` でリポジトリを注入することで、テスト時に実装クラスを差し替えられる構造にする

## E3-16【応用】リポジトリ設計の誤り検出

**問題**（応用）

以下のリポジトリ設計にDDD的な問題点が2つある。それぞれ指摘せよ。

```java
public class UserRepository {  // ① クラスとして定義
    public void beginTransaction() { ... }  // ② トランザクション開始
    public void commitTransaction() { ... }  // ② トランザクションコミット
    public void add(User user) { ... }
    public void save(User user) { ... }
    public List<User> getAllUsers() { ... }
}
```

**模範解答**

**問題点①：クラスとして定義されている**  
リポジトリはインターフェイスで定義し、実装クラスをインフラ層に配置すべき（依存性逆転の原則）。クラスで直接定義するとインフラの詳細がドメイン層に漏れ、実装の差し替えが困難になる。

**問題点②：トランザクション管理をリポジトリが持っている**  
トランザクション境界はアプリケーション層が管理すべき。リポジトリはストレージへのアクセスのみを責務とし、`beginTransaction` / `commitTransaction` はリポジトリの責務外。

## E3-17【応用】コレクション指向と永続指向の選択

**問題**（応用）

新しいプロジェクトでシステムを構築する。以下の条件のとき、どちらのリポジトリスタイルを選ぶべきか答え、理由を述べよ。

条件：集約の要素が多く複雑なリレーションがある、チームはHibernateの経験が豊富、スケールアウトの要件はない

**模範解答**

**コレクション指向リポジトリ**を選ぶ。

理由：
- Hibernateを使ったRDB構成に最適であり、チームの既存スキルを活かせる
- 複雑なリレーションはRDBのJOINや正規化が得意とするところであり、1レコードにまとめるNoSQLよりも適している
- ダーティチェッキングにより `save` を明示的に呼ぶ手間がなく、コードがシンプルになる
- スケールアウト要件がなければ、NoSQLの分散処理の複雑さを取り込む必要がない

## E3-18【応用】リポジトリと集約の1対1対応

**問題**（応用）

「リポジトリは集約ルートに1つだけ作成する」という設計原則がある。この原則に違反した場合に何が起きるか、具体例を使って説明せよ。

**模範解答**

**違反例**：`Order`（注文）集約の中に `OrderItem`（注文明細）が含まれているのに、`OrderItemRepository` を別に作る。

**問題点**：
1. **整合性の破壊**：`OrderItemRepository` を使って `OrderItem` を直接操作すると、`Order` 集約ルートを経由せずに内部状態が変更される。集約の不変条件（例：注文金額の整合性）が保証されなくなる
2. **トランザクション境界の曖昧化**：`Order` と `OrderItem` で別々のリポジトリを使うと、1つのビジネス操作が複数トランザクションに分割されるリスクがある
3. **ドメインルールのバイパス**：集約ルートを通さない操作は、ドメインサービスやエンティティが持つビジネスロジックを迂回してしまう

---

## E3-19【応用】リポジトリの検索メソッド設計

**問題**（応用）

以下のリポジトリに `findByEmail` メソッドを追加したい。DDDの観点から適切なメソッド名・シグネチャを選択し、理由を述べよ。

選択肢：
- `（A）User findByEmail(String email)`
- `（B）User userWithEmailAddress(TenantId tenantId, EmailAddress email)`
- `（C）List<User> searchUsers(Map<String, Object> criteria)`

**模範解答**

**（B）`User userWithEmailAddress(TenantId tenantId, EmailAddress email)` が適切**。

理由：
1. **ユビキタス言語**：`userWith～` という命名は「〜を持つユーザー」という自然なドメイン言語に対応している。`findBy～` は技術的な命名
2. **値オブジェクトの使用**：`EmailAddress` を値オブジェクトとして受け取ることで型安全性が高く、メールアドレスの正規化・検証が保証された状態で検索できる
3. **マルチテナント対応**：`TenantId` を必ず含めることで、テナント間のデータ混入を防ぐ設計になっている
4. **（C）の問題**：`Map<String, Object>` での汎用検索はドメイン語彙を失い、型安全性もない。リポジトリのAPIは具体的な意図を表すべき
