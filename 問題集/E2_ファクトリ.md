## E2-01【知識確認】ファクトリの目的

**問題**（用語問題）

DDDにおける「ファクトリ（Factory）」の主な目的を2点答えよ。

**模範解答**

1. **アトミックな生成**：集約の生成を単一の操作として完結させ、不完全な状態のオブジェクトが外部に漏れないようにする
2. **実装詳細の隠蔽**：クライアント（アプリケーションサービスなど）から集約の生成に関する実装詳細を隠し、ドメインモデルをシンプルに保つ

---

## E2-02【知識確認】ファクトリの4パターン

**問題**（用語問題）

実践ドメイン駆動設計で紹介されているファクトリの4つの実装パターンを列挙せよ。

**模範解答**

1. **コンストラクタ（Constructor）**：単純な生成はコンストラクタで直接行う
2. **集約ルートのファクトリメソッド（Factory Method on Aggregate Root）**：集約ルートが別の集約を生成するメソッドを持つ
3. **ドメインサービス上のファクトリ（Factory on Domain Service）**：複数の依存物を組み合わせる場合にドメインサービスがファクトリ機能を担う
4. **複数コンテキスト間サービスファクトリ（Factory for cross-context）**：他コンテキストのオブジェクトを変換・生成する際に使用するファクトリ

---

## E2-03【知識確認】優れたファクトリの条件

**問題**（用語問題）

Evans（DDDの著者）が示す「優れたファクトリ」が満たすべき2つの条件を答えよ。

**模範解答**

1. **アトミックな生成**：生成するオブジェクトまたは集約の不変条件をすべて強制し、不完全な状態を一切外部に出さない
2. **実装詳細の隠蔽（抽象化）**：生成されたオブジェクトの具体的な型よりも抽象的なインターフェイスで生成物を返し、クライアントから実装詳細を隠す

---

## E2-04【知識確認】ファクトリ関連デザインパターン

**問題**（用語問題）

ファクトリの実装で参照されるGoFデザインパターンを4つ答え、それぞれの特徴を簡潔に述べよ。

**模範解答**

1. **ファクトリメソッド（Factory Method）**：生成するオブジェクトのクラスをサブクラスで決定するパターン。集約ルートのファクトリメソッドに対応する
2. **アブストラクトファクトリ（Abstract Factory）**：関連するオブジェクト群を一括生成するインターフェイスを提供するパターン
3. **ビルダー（Builder）**：複雑な生成ステップを段階的に組み立てるパターン。複数の任意パラメータを持つ集約の生成に適する
4. **アダプター（Adapter）**：異なるインターフェースや異なるコンテキスト間のオブジェクト変換を担うパターン。複数の境界づけられたコンテキストにまたがる変換型ファクトリに使用される

## E2-05【理解確認】コンストラクタで十分な場合とファクトリが必要な場合

**問題**（読解判断問題）

次の2つの状況について、コンストラクタで直接生成すべきか、ファクトリを使うべきかを選択し、理由を述べよ。

（A）`Money`値オブジェクトを `new Money(100, "JPY")` で生成する  
（B）`Forum` 集約の生成時に、`ForumId` の採番（`forumIdService.getNextForumId()`）と不変条件の検証が必要である

**模範解答**

- **（A）コンストラクタで十分**：単純な値オブジェクトで、依存するサービスが不要で不変条件もコンストラクタ内で完結できるため
- **（B）ファクトリが必要**：IDの採番に外部サービス（`ForumIdentityService`）が必要であり、複数の依存物を組み合わせてアトミックに生成する必要があるため。集約ルートのファクトリメソッドパターンが適切

## E2-06【理解確認】集約ルートのファクトリメソッドが不変条件を検証できる理由

**問題**（読解判断問題）

以下の `Forum.StartDiscussionFor()` メソッドを見て、このメソッドがファクトリメソッドとして優れている理由を2点説明せよ。

```csharp
public Discussion StartDiscussionFor(ForumIdentityService forumIdService,
        Author author, string subject, string exclusiveOwner = null) {
    AssertOpen();   // Forumが開いている状態でなければ例外
    return new Discussion(this.tenantId, this.forumId,
        forumIdService.GetNextDiscussionId(), author, subject, exclusiveOwner);
}
```

**模範解答**

1. **不変条件の強制**：`AssertOpen()` によってフォーラムがオープン状態でなければ `Discussion` を生成しない。不完全な状態の `Discussion` が外部に漏れることを防ぐ
2. **ドメイン知識の表現**：「フォーラムがオープンのときにのみディスカッションを開始できる」というドメインルールが `Forum` 集約のメソッド名 `StartDiscussionFor` として自然に表現されており、ユビキタス言語になっている

## E2-07【理解確認】ドメインサービスファクトリが適切な場面

**問題**（読解判断問題）

以下の `ICollaboratorService` インターフェイスはドメインサービス上のファクトリとして機能する。このパターンがコンテキスト間変換に適している理由を説明せよ。

```csharp
public interface ICollaboratorService {
    Author GetAuthorFrom(Tenant tenant, string identity);
    Creator GetCreatorFrom(Tenant tenant, string identity);
    Moderator GetModeratorFrom(Tenant tenant, string identity);
    Owner GetOwnerFrom(Tenant tenant, string identity);
    Participant GetParticipantFrom(Tenant tenant, string identity);
}
```

**模範解答**

他コンテキスト（認証・アクセスコンテキストなど）からユーザー情報を取得し、コラボレーションコンテキストの値オブジェクト（`Author`、`Creator` など）へ変換する処理が必要なため。この変換ロジックは単一の集約では担えず、複数の依存物（テナント情報、外部ユーザー情報）を組み合わせる必要があるため、ドメインサービスにファクトリ機能を持たせる。クライアントはどのコンテキストからどう変換されるかを知らずに `Author` などを取得できる。

## E2-08【理解確認】ユビキタス言語とファクトリメソッド名

**問題**（読解判断問題）

以下の2つのメソッドを比較し、どちらがDDD的に適切か答え、その理由を述べよ。

- `（A）Forum.createDiscussion(ForumIdentityService svc, String subject)`
- `（B）Forum.startDiscussionFor(ForumIdentityService svc, Author author, String subject)`

**模範解答**

**（B）が適切**。理由：

1. `startDiscussionFor` という名前はドメインエキスパートの言葉（ユビキタス言語）に近く、「誰が」「何のために」ディスカッションを始めるかが表現されている
2. `Author` パラメータを明示することで、ディスカッションは特定の著者が開始するものというドメインの意図が明確になる
3. `create` という技術的な言葉より、`startDiscussionFor` という業務的な言葉の方がドメインモデルとして自然

## E2-09【応用】ファクトリパターンの選択

**問題**（応用）

ECサイトにおいて、次のシナリオそれぞれで最適なファクトリパターンを選択し、理由を述べよ。

（A）`Order`（注文）集約が `OrderItem`（注文明細）を追加するとき、在庫確認と価格計算が必要  
（B）他の「商品カタログコンテキスト」の商品情報を「注文コンテキスト」の `Product` 値オブジェクトに変換する

**模範解答**

- **（A）集約ルートのファクトリメソッド**：`Order.AddItem(product, quantity, inventoryService)` のように `Order` 集約がファクトリメソッドを持つ。在庫確認という不変条件を `Order` が保証できるため
- **（B）複数コンテキスト間サービスファクトリ または ドメインサービスファクトリ**：コンテキスト境界を越えた変換は単一集約では担えない。`ProductTranslationService.ToOrderProduct(catalogProduct)` のようなドメインサービスが変換を担う。ACL（腐敗防止層）としての役割も果たす

## E2-10【応用】ファクトリとコンストラクタの責任分離

**問題**（応用）

次のコードにはDDD的な問題がある。問題点を指摘し、ファクトリパターンを使った改善案を示せ。

```csharp
// アプリケーションサービス
public class OrderApplicationService {
    public Order PlaceOrder(string customerId, List<OrderItemDto> items) {
        var customer = customerRepo.FindById(customerId);
        var priceService = new PriceCalculationService();
        var order = new Order(
            Guid.NewGuid(),
            customer,
            items.Select(i => new OrderItem(i.ProductId, i.Quantity, priceService.Calculate(i))).ToList()
        );
        orderRepo.Add(order);
        return order;
    }
}
```

**模範解答**

**問題点**：
1. 注文の生成ロジック（価格計算、`OrderItem` 生成）がアプリケーションサービスに漏れており、ドメイン知識がドメイン層に集まっていない（貧血ドメインモデルの兆候）
2. `new PriceCalculationService()` をアプリケーションサービスが直接生成しており、依存関係が密結合

**改善案（集約ルートファクトリメソッド）**：
```csharp
// Order集約にファクトリメソッドを追加
public class Order {
    public static Order Place(CustomerId customerId, List<OrderItemDto> items,
            IPriceCalculationService priceService) {
        AssertValidCustomer(customerId);
        var orderItems = items.Select(i => OrderItem.Create(i, priceService)).ToList();
        return new Order(OrderId.Next(), customerId, orderItems);
    }
}
// アプリケーションサービスはファクトリメソッドを呼ぶだけ
var order = Order.Place(customerId, items, priceService);
```

## E2-11【応用】EventSourcedRootEntityとファクトリ

**問題**（応用）

イベントソーシングを採用した集約（`EventSourcedRootEntity` を継承する集約）では、ファクトリメソッドはどのように機能するか。通常のファクトリとの違いを説明せよ。

**模範解答**

通常のファクトリは集約の現在状態をメモリ上のオブジェクトとして直接生成する。一方、イベントソーシング集約のファクトリメソッドは：

1. **ドメインイベントを発行する**：`Discussion` を生成する代わりに `DiscussionStarted` イベントを発行する
2. **状態はイベント適用で構築される**：集約の状態は発行されたドメインイベントを適用（`apply`）することで決まる
3. **再構築との一貫性**：ファクトリによる新規生成も、イベントストアからの再構築も同じイベント適用メカニズムを使うため一貫性が保たれる

## E2-12【応用】ファクトリとリポジトリの役割分担

**問題**（応用）

次の説明を読み、ファクトリとリポジトリの役割の違いを明確にせよ。また、集約の「新規作成」と「再構築」でどちらを使うべきかを説明せよ。

> 「ファクトリはオブジェクトの生成を担い、リポジトリはオブジェクトの永続化を担う」

**模範解答**

| 観点 | ファクトリ | リポジトリ |
|------|-----------|-----------|
| 責務 | 集約の新規生成・不変条件の保証 | 集約の永続化・取得・削除 |
| タイミング | 初めて集約を作るとき | 保存済み集約を取得・保存するとき |
| ドメイン知識 | 生成に必要な不変条件・業務ルール | データストアへのアクセス詳細（隠蔽） |

- **新規作成**：ファクトリ（またはファクトリメソッド）が集約を生成し、その後リポジトリの `add()` で永続化する
- **再構築**：リポジトリが永続化されたデータから集約を取得する（リポジトリが内部でファクトリを使うこともある）
- 混同してはいけない：リポジトリの取得メソッドはファクトリではなく「再構築」であり、ドメインイベントからの再構築とは区別される
