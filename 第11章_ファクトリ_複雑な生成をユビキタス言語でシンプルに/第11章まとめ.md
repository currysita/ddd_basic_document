# 第11章「ファクトリ」〜複雑な生成をユビキタス言語でシンプルに〜

## 11.1 ファクトリとは

問題領域における一般的なオブジェクト生成の責務について整理した節。

問題領域における一般的なオブジェクト生成の責務として、以下の2点を述べている：

1. 定義済みの集約に、別のオブジェクト（主に集約）を生成させる
2. ユースケースに合わせた複雑な生成をドメインサービスやアプリケーションサービスで行う代わりに、ロールのある「複雑なインスタンスを生成する」役割のファクトリオブジェクトに任せる

上記のアプローチにより、生成のロジックが明確に分離されてコードがシンプルになる。

---

## 11.2 ファクトリを使う理由

ファクトリを使う理由として、実践ドメイン駆動設計では DDD 本での Evans 氏の次の説明を引用している：

> 複雑なオブジェクトと集約のインスタンスを生成する責務を、別のオブジェクトに移すこと。その別のオブジェクトは、それ自体ではドメインモデルにおいて何の責務も負っていないかもしれないが、それでもドメイン設計の一部であることに変わりはない。（DDD 本 136 ページより）

Evans 氏は「車のエンジン」や「銀行口座」を例にファクトリを利用する理由を紹介している。エンジンの場合「複数部品を使ってエンジンを組み立てる処理」と「回転する処理」を分けている。銀行口座の場合「口座の開設処理」と「口座の取引処理」を分けている。このように複雑な生成処理を分離することで、全体を理解しやすくできることがファクトリのメリットと紹介している。

### 実践ドメイン駆動設計 におけるファクトリの例

実践ドメイン駆動設計においては、別の集約にファクトリメソッドを用意する方法を主に紹介している。つまり「ファクトリ」は「集約の振る舞い（メソッド）」となる。

---

## 11.3 ファクトリの設計

ここにはオブジェクトの生成のパターンについて示している。どのようなオブジェクトを生成する場合にも共通して、以下のパターンが適用できる。

| # | 生成に使われる方法 | コンストラクタの役割 | ファクトリの役割 | 用途 |
|---|----------------|------------------|--------------|------|
| (1) | コンストラクタ | オブジェクトの生成 | — | シンプルな生成 |
| (2) | 集約上のファクトリ | — | オブジェクトの生成 | 複数のオブジェクトを作る場合 |
| (3) | ドメインサービス上のファクトリメソッド | — | オブジェクトの生成 | ドメイン知識が必要な場合 |
| (4) | サービス上のファクトリ | — | オブジェクトの生成 | 複数のコンテキストにまたがる場合 |

これらの特性について詳しく説明している。

### （1）コンストラクタで十分な場合

生成処理がシンプルな場合には、コンストラクタを利用してファクトリアイテムを使用しないほうがよい場合もある。コンストラクタは複数のオブジェクトをコンポジションする場合に、複雑になりがちである。ただし、それらのオブジェクトが少数（1 〜 2 個程度）の場合はコンストラクタで十分といえるかもしれない。

### （2）集約ルートに実装するファクトリ

既に先述したとおりに、集約上でのファクトリを生成する場合のケースについてまとめる。オブジェクトの生成に複数のオブジェクトを組み合わせる場合、あるいはビジネスロジックとしてファクトリとして取り込みたい場合に有効である。ファクトリの振る舞いをした方がよい場合に適する。このようなオブジェクトを複数作る場合に適している。

#### 集約上のファクトリのコード例

SaaSOvation のコードを見てみよう。集約である `Forum` クラスの `StartDiscussionFor` というファクトリが呼び出されると、戻り値として `Discussion` オブジェクトを戻す。これは「投稿者はフォーラム上でディスカッションを開始する」というユビキタス言語を表現していることがわかる。

**リスト 11.1　集約「Forum」で「ディスカッションを開始する」ファクトリ（C#）**

```csharp
namespace SaaSOvation.Collaboration.Domain.Model.Forums
{
    public class Forum : EventSourcedRootEntity
    {
        // ディスカッションを開始する（ファクトリ）
        public Discussion StartDiscussionFor(ForumIdentityService forumIdService,
                Author author, string subject, string exclusiveOwner = null)
        {
            AssertOpen();
            return new Discussion(
                    this.tenantId,
                    this.forumId,
                    forumIdService.GetNextDiscussionId(),
                    author,
                    subject,
                    exclusiveOwner);
        }

        // 不変条件の確認（フォーラムがオープンしているか）
        void AssertOpen()
        {
            if (this.closed)
                throw new InvalidOperationException("Forum is closed.");
        }
    }
}
```

#### 不変条件を満たすファクトリ

集約に基づいてファクトリを使用することにより、ファクトリが他のドメインオブジェクトを正しく生成するよう制御できる。また、Evans 氏は 実践ドメイン駆動設計においてこの不変条件を満たすことがファクトリにとって重要であると述べており、集約ルートが生成処理の中で自身の不変条件を確認（`AssertOpen()` など）することもできる。

### （3）サービスに実装するファクトリ

次に、サービスに実装するファクトリについて示す。なお、ファクトリを実装するサービスには、ドメインサービスとアプリケーションサービスの2種類がある。

**リスト 11.2　ファクトリを呼び出すアプリケーションサービス（C#）**

```csharp
namespace SaaSOvation.Collaboration.Application.Forums
{
    public class ForumApplicationService
    {
        // フォーラムにモデレーターをアサインする
        public void AssignModeratorToForum(string tenantId,
                string forumId, string moderatorId)
        {
            var tenant = new Tenant(tenantId);
            var forum = this.forumRepository.Get(tenantId, forumId);

            // モデレーターを生成するファクトリ
            var moderator = this.collaboratorService.GetModeratorFrom(
                    tenant, moderatorId);
            forum.AssignModerator(moderator);
            this.forumRepository.Save(forum);
        }
    }
}
```

このコードでは、`ForumApplicationService` クラスの `AssignModeratorToForum` の中で、`collaboratorService` の `GetModeratorFrom` を呼び出している。`collaboratorService` のファクトリで `Moderator` オブジェクトを生成していることがわかる。

#### ドメインサービス上のファクトリのコード例

**リスト 11.3　コラボレーションコンテキストのオブジェクトを生成するファクトリ（C#）**

```csharp
namespace SaaSOvation.Collaboration.Domain.Model.Collaborators
{
    // 変換用のドメインサービス
    public interface ICollaboratorService
    {
        // 投稿者を生成して取得
        Author GetAuthorFrom(Tenant tenant, string identity);

        // 作成者を生成して取得
        Creator GetCreatorFrom(Tenant tenant, string identity);

        // モデレーターを生成して取得
        Moderator GetModeratorFrom(Tenant tenant, string identity);

        // 所有者を生成して取得
        Owner GetOwnerFrom(Tenant tenant, string identity);

        // 参加者を生成して取得
        Participant GetParticipantFrom(Tenant tenant, string identity);
    }
}
```

6 章の「値オブジェクト」でも紹介した通り、「人」という概念でも、認証・アクセスコンテキストでは「ユーザー／ロール」というモデルになったり、コラボレーションコンテキストでは「投稿者／作成者／モデレーター／所有者／参加者」という別のモデル（ユビキタス言語）になったりする。

【図 11.2】実践ドメイン駆動設計 におけるサービスの「ファクトリ」メソッドの例

複数の境界づけられたコンテキストの間の変換情報を管理するオブジェクトが「ファクトリ」メソッドを持っている。認証・アクセスコンテキストのユーザー（`User`）情報をもとに、コラボレーションコンテキストの各ロール（`Author`、`Creator`、`Moderator`、`Owner`、`Participant`）を生成するファクトリが `ICollaboratorService` として定義されている。

このファクトリではサービス上のファクトリを使って、複数の境界づけられたコンテキストの間で変換が可能となる。

## 11.4 優れたファクトリとデザインパターン

優れたファクトリの設計指針について、Evans 氏は DDD 本において次の2点を述べている：

1. 各生成メソッドはアトミックであること。生成したオブジェクトや集約のインスタンスを生成する際に、不変条件を検証する。
2. クライアントから実装の詳細を隠す。

さらに次のように述べている。ファクトリは「生成ルールはドメインオブジェクトではなくファクトリが持つべきである」と主張しているが、それはクライアント側から実装の詳細を隠すとも言い換えられる。実践ドメイン駆動設計における「ファクトリ」は、「生成を別のオブジェクトへ移すことで、クライアントから実装の詳細を隠す」という考え方と一致している。

これらを満たすファクトリの実装には、Eric Gamma 氏の書籍『オブジェクト指向の再利用のためのデザインパターン \*7』（SB クリエイティブ、1999 年）の「ファクトリメソッド \*6」「アブストラクトファクトリ \*5」「ビルダー \*4」といったプログラミングパターンがよく使われる。また、変換を行う場合には「アダプター \*3」パターンがよく使用される。これらのデザインパターンに興味がある方は、書籍や脚注のページをご覧ください。
