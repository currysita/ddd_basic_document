# 用語集

> 五十音順。英数字で始まる用語は末尾にまとめています。

---

## あ行

### 依存関係逆転の原則

**略語：** DIP（Dependency Inversion Principle）

Robert C. Martin が提唱した SOLID 原則の「D」。上位モジュールが下位モジュールに依存するのではなく、両者が抽象（インターフェイス）に依存するよう設計する原則。リポジトリのインターフェイスをドメイン層に定義し実装をインフラストラクチャ層に置くことで、ドメイン層がインフラ層に依存しない構造を実現できる（参照：リポジトリ）。

### アジャイル開発

DDDが前提とする開発手法。DDDとアジャイルは相性がよく、アジャイル経験者のほうがDDD導入がスムーズになる。

### アジャイルプロジェクト管理ドメイン

SaaSOvation のコアドメイン。スプリント・バックログアイテム・リリース・タスクなどアジャイル開発の管理機能を提供する。競合他社に対する差別化ポイントを担う。

### アダプター（Adapter）

異なるインターフェースや異なるコンテキスト間のオブジェクト変換を担うデザインパターン。複数のコンテキストにまたがる変換型ファクトリによく使用される。腐敗防止層の実装にも用いられる。

### アプリケーション層（Application Layer）

ドメインモデルの調整役として機能するクライアント層。ユースケースに応じてリポジトリから集約を取り出し、ドメインのコマンドを実行し、トランザクション境界を管理し、集約を保存する。ドメインロジック自体は持たない。「アプリケーションサービス」とも呼ばれる。

### アトミックな生成（Atomic Creation）

ファクトリメソッドは生成操作をひとまとまり（アトミック）に行い、中途半端な状態のオブジェクトが外部に漏れないようにする設計指針。Evans 氏が優れたファクトリの条件として挙げている。

### アブストラクトファクトリ（Abstract Factory）

複数の関連オブジェクト群を生成するインターフェースを提供するデザインパターン。ファクトリ実装の選択肢の一つ。

### アプリケーションサービス（Application Service）

ドメインモデルを利用するクライアントとしてのサービス。ユーザーとのやり取りを調整・制御し、トランザクション管理やユースケースを担う。アプリケーション層に属し、ドメインサービスとは明確に区別される。リポジトリを利用してファクトリで生成した集約を永続化する役割も担う（`@Transactional` 等のトランザクション宣言もここで管理）。ユースケースごとのメソッドを持ち、Commandパターンで入力を受け取ることが多い。

### アプリケーションレイヤのモジュール（Application Layer Module）

`[組織名].[コンテキスト名].Application` の階層に配置されるモジュール。アプリケーションサービス・コマンド・通知などが格納される。ドメインモデルを操作するクライアントとしての役割を持つ。

### イベントソーシング（Event Sourcing）

集約の現在の状態を直接保存する代わりに、発生したすべてのドメインイベントを記録し、それを再生することで集約の状態を再構築するパターン。イベントストアと組み合わせて使用される。過去の任意の時点でシステムの状態を再構築（リプレイ）できるため、監査ログやコンプライアンス対応にも有効。

### イベントストア（Event Store）

発生したドメインイベントをすべてストレージに記録するコンポーネント。メッセージング基盤のキュー、監査ログ、分析データ、イベントソーシングによる集約の再構築などに活用できる。ストレージには RDB や NoSQL を使用する。

### イベントストーミング（Event Storming）

ドメインイベントを起点にシステムの振る舞いを設計・発見するワークショップ手法。付箋などを用いてドメインエキスパートと開発者が協力してイベントを洗い出す。

### イベント駆動アーキテクチャ（Event-Driven Architecture）

システム内の状態変化を「イベント」として発行し、それを受け取ったサブスクライバが処理を行うアーキテクチャスタイル。非同期処理や疎結合を実現しやすい。

### 一意な識別子（Unique Identity）

エンティティ・集約を一意に識別するための値。ビジネス上のキー（ユーザー名、テナントIDなど）を指す。生成パターンは「ユーザー入力・アプリ生成（UUID）・DB生成（連番）・他コンテキストから割り当て」の4種類がある。他集約への参照は直接オブジェクト参照ではなくこの識別子（ID値オブジェクト）を通じて行う。

### 上流（Upstream）

**略語：** U

コンテキストマップにおける情報・サービスを提供する側のコンテキスト。

### 大きな泥団子（Big Ball of Mud）

統合パターンの1つ。境界づけられたコンテキストが明確に定義されていないレガシーシステムの状態。対応策として腐敗防止層を設けて自コンテキストのモデルを守ることが推奨される。

### オープンホストサービス（Open Host Service）

**略語：** OHS

統合パターンの1つ。境界づけられたコンテキストの内部へアクセスするための公開サービスをプロトコルとして定義するパターン。XML や JSON などの標準的なフォーマットを利用することが多い。公表された言語と組み合わせて使用されることが多い。

## か行

### 解決領域

ドメインのうち、問題領域の課題をソフトウェアで解決する側。境界づけられたコンテキストによって構成される。

### 概念的な統一体（Conceptual Whole）

関連する複数の属性をひとつの概念としてまとめた値オブジェクトの特性。例：「金額」は「数値」と「通貨単位」の組み合わせであり、分離せずに一つの値オブジェクトとして扱う。

### 隔離されたコア（Segregated Core）

コアドメインで本質的でない部分をサブドメインに移していくリファクタリングパターン。

### 集約（Aggregate）

関連するオブジェクト群（エンティティ・値オブジェクト）をトランザクション整合性の観点でひとまとめにしたグループ。集約内のオブジェクトは常に一貫した状態を保つ。集約の外からはルートエンティティ（集約ルート）経由でのみ操作できる。内部オブジェクトへの参照は識別子（ID）経由で行い、直接オブジェクト参照は避ける。1つのトランザクションで変更できるのは原則1つの集約のみ。

### 集約ルート（Aggregate Root）

集約の代表となるエンティティ。外部からの操作は必ず集約ルートを通じて行われ、集約内部オブジェクトへの直接アクセスは禁止される。集約のトランザクション整合性を守る唯一の入口。一意な識別子を持ち、外部集約はこの識別子のみを参照できる。「ルートエンティティ」とも呼ばれる。

### 顧客/供給者（Customer/Supplier）

組織的パターンの1つ。上流側が供給者となり、下流側が顧客となる関係。上流側が下流側のサポートを確約している。

### 下流（Downstream）

**略語：** D

コンテキストマップにおける情報・サービスを受け取る側のコンテキスト。

### 汎用サブドメイン

どの企業でも共通して持つような汎用的な機能を提供するサブドメイン。認証・認可、ERPなどが該当する。既製品や外部サービスで対応可能な場合が多い。

### 技術的負債

開発費用を「コスト」として扱い、品質を落とした機能を素早くリリースし続けた結果として蓄積される問題。技術的負債が増えると開発速度が低下し、さらにコストが膨らむ負の連鎖に陥る。DDDを採用して開発費用を「事業投資」と捉えることで、この負の連鎖を断ち切ることを目指す。

### 境界づけられたコンテキスト

**略語：** BC（Bounded Context）

ドメインモデルが適用される明確な境界。その境界内でユビキタス言語が一貫した意味を持ち、モデルの整合性が保たれる。1つの境界づけられたコンテキストには1つのモデルを持たせる。チームの責任範囲とも連動する。

### 共有カーネル（Shared Kernel）

統合パターンの1つ。2つのコンテキストが共有のドメインモデルをソースコードレベルで共有するパターン。両チームの合意が必要。変更時に互いに影響を受ける可能性がある。

### 凝集度（Cohesion）

モジュールの責任がどれほど一貫しているかの尺度。高いほど良い。関数型凝集（functional cohesion）が最も高い凝集度で、モジュール内のすべての要素が単一のタスクの実行に向けて協力している状態。凝集度が高いほどシステムのメンテナンス性が向上する。

### 交換可能性（Replaceability）

値オブジェクトの値を「変更したい」場合、セッターで内部状態を書き換えるのではなく、新しいインスタンスに丸ごと置き換える性質。不変性の帰結として生まれる設計パターン。

### 公表された言語（Published Language）

**略語：** PL

統合パターンの1つ。2つの境界づけられたコンテキスト間でモデルを変換する際に共通の言語（仕様）を用いるパターン。オープンホストサービスと組み合わせて使用されることが多い。

### 高凝集（High Cohesion）

モジュール内の要素が互いに強く関連し、単一の責任・概念にまとまっている状態。DDDにおけるモジュール設計の原則の一つ。

### コアドメイン

ビジネス上で最も重要な関心事を担うドメイン。競合他社に対して優位性をもたらす領域であり、DDDにおいて最も注力すべき対象。

### コピーコンストラクタ（Copy Constructor）

値オブジェクト自身を引数として受け取り、全属性をコピーした新しいインスタンスを生成するコンストラクタ。不変な値オブジェクトで「一部の属性だけ変えた新しいオブジェクト」を作る際や、不変性のテストを記述する際によく使われる。

### コマンド（CQRS）

CQRSにおけるコマンドは、ドメインオブジェクトの状態を変更する命令。戻り値を返さない。コマンドハンドラ（またはコマンドプロセッサ）が受け取り、対応するドメインモデルに処理を委譲する。

### コマンドクエリ責務分離

**略語：** CQRS（Command Query Responsibility Segregation）

コマンド（状態変更）とクエリ（状態参照）を別々のモデルに分離するアーキテクチャパターン。読み取りモデルを非正規化することで、複雑なクエリを高速に処理できる。

### コマンドクエリ分離

**略語：** CQS（Command Query Separation）

Bertrand Meyer が提唱した設計原則。メソッドは状態を変更する「コマンド」か値を返す「クエリ」のどちらか一方であるべき、という原則。CQRSはこれをアーキテクチャレベルに拡張したもの。

### コラボレーションドメイン

SaaSOvation の支援サブドメイン。フォーラム・ディスカッション・カレンダー・共有コンテンツなどのコラボレーション機能を提供する。チームの置かれた状況によってはコアドメインになり得る。

### コラボレーターサービス（Collaborator Service）

認証・アクセスコンテキストのユーザー情報を、コラボレーションコンテキストのロールオブジェクト（投稿者・作成者・モデレーター・所有者・参加者）に変換するドメインサービス。複数の境界づけられたコンテキスト間の変換型ファクトリとして機能する。

### カスタムメディアタイプ（Custom Media Type）

REST API のレスポンスで使用する独自の MIME タイプ。`application/vnd.<vendor>.<type>+json` の形式で定義され、ベンダーツリー（`vnd.`）プレフィックスを持つ。コンテキスト間の「公表された言語」として機能する。SaaSOvation の例では `application/vnd.saasovation.projectovation+json` など。

### コレクション指向リポジトリ（Collection-Oriented Repository）

対象ストレージが RDB など、O/R マッパーを使えるデータストアに適したリポジトリの設計スタイル。O/R マッパーが変更追跡（ダーティチェッキング）を行うため、追加時は `add`、削除時は `remove` を使い、明示的な `save` は不要。コレクション（リスト・集合）を操作するような API を持つことが特徴。

### コンストラクタ（Constructor）

オブジェクト生成時に呼び出されるメソッド。エンティティのコンストラクタでは必須属性（識別子を含む）をすべて引数として受け取り、欠けていた場合は生成を失敗させることで、オブジェクトの完全性を保証する。

### コンテキストマップ

**略語：** CM（Context Map）

複数の「境界づけられたコンテキスト」間の関係を俯瞰する地図。システムの全体像や相互関係を把握するために使用する。チームの全メンバーが随時閲覧できる場所に置き、オープンな会話の基盤とする。

## さ行

### サービス指向アーキテクチャ

**略語：** SOA（Service Oriented Architecture）

ソフトウェアをサービスの集合として設計するアーキテクチャスタイル。

### サブドメイン

コアドメインを補助・支援する役割を持つドメインの総称。支援サブドメインと汎用サブドメインに分類される。

### サブモジュール（Submodule）

上位モジュール配下に階層的に作られるモジュール。Domain.Model 配下に値オブジェクト・エンティティ・ドメインイベントなどをまとめたものがサブモジュールとなる。

### 実践ドメイン駆動設計

**略語：** IDDD本（Implementing Domain-Driven Design）

ヴォーン・ヴァーノン著の書籍。本ドキュメントはこのIDDD本の構成に沿って解説している。

### 自立型サービス（Autonomous Service）

他のサービスに依存せず独立して動作するサービス。ドメインイベントを活用して疎結合に連携することで実現される。障害時の影響範囲を最小化し、個別のスケーリングが可能になる。

### 支援サブドメイン

コアドメインを補助する役割を持つサブドメイン。独自の機能を持つが、コアドメインほどの重要性はない。

### 順応者（Conformist）

組織的パターンの1つ。上流側が下流側の要望を考慮する必要がない状態。下流側が上流側のモデルに従わざるを得ず、腐敗防止層の使用を甘受して受け入れる。

### 疎結合（Loose Coupling）

モジュール間の依存関係が最小限に抑えられている状態。DDDにおけるモジュール設計の原則の一つ。双方向の依存関係はできるかぎり避けることが推奨される。

### 組織的パターン

コンテキストマップにおいて、チーム間の関係を示すパターン群。パートナーシップ・別々の道・順応者・顧客/供給者の4つのパターンがある。

### セカンダリなドメイン

コアドメイン以外のドメイン。コアドメインほど競争力に直結しない業務領域。DDDを採用することで、コアドメインとセカンダリなドメインを区別し、リソースを適切に配分できるようになる。

### セパレートインターフェイス（Separate Interface）

Martin Fowler 氏が提唱したパターン（P of EAA）。インターフェイスと実装を別のパッケージ（層）に分離する手法。ドメイン層にインターフェイスを定義し、実装をインフラストラクチャ層に置くことで、ドメイン層がインフラストラクチャに依存しないクリーンな設計を実現できる。リポジトリはこのパターンを体現する典型的な例。

### ストアドイベント（Stored Domain Event）

ドメインイベントをイベントストアに永続化するためのデータクラス。EventId やイベントクラス名などのメタ情報とイベントのシリアライズデータを保持する。

## た行

### 代理識別子（Surrogate Identity）

O/Rマッパー（Hibernate や Entity Framework など）がデータベースとのマッピングのために使用するシステム的な識別子（DBの主キーなど）。ビジネス上の識別子（一意な識別子）とは別物であり、外部に公開しないよう隠蔽するのがベストプラクティス。

### 単一責任原則

**略語：** SRP（Single Responsibility Principle）

「クラスを変更する理由は1つだけであるべき」という設計原則。Robert C. Martin 氏が提唱。ドメインサービスに適用する場合、1つのドメインサービスは1つのことに集中すべきであり、責任が増えた場合は分割を検討する。

### データ転送オブジェクト

**略語：** DTO（Data Transfer Object）

層間でデータを受け渡すための単純なデータ構造。ユーザーインターフェイス層への入力をDTOとして受け取り、アプリケーションサービスやドメインオブジェクトへ渡す。

### データアクセスオブジェクト

**略語：** DAO（Data Access Object）

データアクセスを担うオブジェクト。リポジトリと似ているが設計の意図が異なる。DAO はデータストアのテーブル単位での操作を意識しており、集約という単位の概念がない。対してリポジトリは集約単位で操作するため、DDD 的に適切な抽象化レベルを持つ。

### データファブリック（グリッドコンピューティング）

Data Fabric。イベントソーシングを基盤として、すべてのオブジェクトの状態変化をイベントとして記録し、分散環境に展開するアーキテクチャ。Pivotal GemFire、Oracle In-Memory Coherence などが実装例。キャッシュ、レプリカ提供、結果整合性のサポートなどに使われる。

### テストファースト

DDDでは設計段階からモデルをテストで検証する手法。新しいモデルの概念が出てきた際に、チームで検討→テストコード作成→実装→リファクタリング→ドメインエキスパートによる確認、というサイクルで進める。従来の TDD と同様の考え方だが、ドメインエキスパートが設計時点から検証に参加できる点が異なる。

### テナントID（TenantId）

SaaSOvation においてテナントを一意に識別するための値オブジェクト。アプリケーション側で UUID を生成して使用する。エンティティの一意な識別子の一例。

### テナントモジュール（Tenants Module）

SaaSOvation の AgilePM コンテキストにおける `Domain.Model.Tenants` サブモジュール。TenantId（値オブジェクト）を含み、他のほとんどのモジュールから参照される。シンプルな片方向の依存関係を持つ。

### タイムアウトトラッカー（TimeoutTracker）

長期プロセスにおいて、一定時間以内に応答がない場合にイベントの再送や補償処理をトリガーする監視コンポーネント。長期プロセスの信頼性・耐障害性を高めるために導入される。

### 統合パターン

コンテキストマップにおいて、データとプログラムの連携方法を示すパターン群。共有カーネル・大きな泥団子・オープンホストサービス・公表された言語・腐敗防止層の5つのパターンがある。

### 同一性（Identity）

エンティティが「同じものであるかどうか」を判断するための概念。エンティティの同一性は一意な識別子（ID）によって決まる。値オブジェクトの同一性は属性値の等価性によって決まる。

### 等値性（Equality）

値オブジェクトが同一かどうかを判断する基準。エンティティはIDで同一性を判断するのに対し、値オブジェクトは属性の値がすべて等しければ等しいとみなす。equals() メソッドをオーバーライドして実装する。

### トランザクションスクリプト

DDDのアンチパターンのひとつ。ビジネスロジックが画面・処理ごとに分散・重複してしまう構造。複雑化するにつれて保守が難しくなる。ドメインモデルと対比される。

### トランザクション整合性（Transaction Consistency）

1つのトランザクション内で集約の状態が一貫していることを保証すること。集約の境界はトランザクションの境界と一致する。1つのトランザクションで変更できるのは原則1つの集約のみ。異なる集約間の整合性は結果整合性で保つ。

### トランザクション管理

リポジトリのメソッド自体はトランザクション処理を担わない。トランザクション境界はアプリケーションサービス層で管理する（Java/Spring の `@Transactional` アノテーション等）。リポジトリはストレージへのアクセスのみを責務とする。

### ドメイン

組織が行う事業や活動の範囲全体。チームが向き合う対象領域。広義のドメインは「問題領域」と「解決領域」に分けられる。

### ドメイン駆動設計

**略語：** DDD（Domain-Driven Design）

2003年にエリック・エヴァンスが提唱した、ドメイン（業務領域）の知識をソフトウェア設計の中心に置く開発手法。ドメインエキスパートと開発者が協力してモデルを作り上げることを重視する。

### ドメインエキスパート

役職名ではなく、業務のシステム上で問題を最もよく知る人物。プログラマや経営者、組織、部門、特定のロールの場合もある。1人ではなく複数人にわたることが多い。

### ドメインイベント（Domain Event）

ドメインの専門家が気にかける、ドメイン内で起きた「出来事」を表すオブジェクト。「注文された」「在庫が切れた」「決済が完了した」などが例として挙げられる。Eric Evans の DDD 本では「ドメインの専門家が気にかけるドメイン内で起きたあること」と定義される。不変（イミュータブル）なオブジェクトとして設計される。

### ドメインイベントサブスクライバー（Domain Event Subscriber）

パブリッシャーが発行したドメインイベントを受信（購読）して処理する役割を担うクラス。受信後の処理には同期処理（トランザクション実行）と非同期処理（キューへの格納）の 2 種類がある。

### ドメインイベントパブリッシャー（Domain Event Publisher）

ドメインイベントを発行（出版）する役割を担うクラス。集約がコマンドを処理した結果としてイベントを生成し、パブリッシャー経由で外部（サブスクライバー）に通知する。

### ドメインサービス（Domain Service）

ドメインモデルの一部として、単一のエンティティや値オブジェクトには自然に割り当てられないドメイン固有のビジネスロジックを担うサービス。複数のドメインオブジェクトを組み合わせた操作・変換・計算処理を実装する場所。ドメイン層に属する。

### ドメインモデル

問題の対象ドメインの概念を表現したソフトウェアのモデル。業務の「モノ」とその振る舞いを中心にモデリングしたもの。システムの設計を理解するための地図となる。トランザクションスクリプトと対比される。

### ドメインモデルモジュール（Domain Model Module）

`[組織名].[コンテキスト名].Domain.Model` の階層に配置されるモジュール。値オブジェクト、エンティティ、ドメインイベント、リポジトリ、ファクトリなどのクラスやインターフェイスが定義される。

## な行

### 認証・アクセスドメイン

SaaSOvation の汎用サブドメイン。ユーザー・グループ・ロール・パーミッション・テナントなどを管理する。他の境界づけられたコンテキストに依存せず、また差し替え可能なため汎用サブドメインに分類される。

### 命名規約（Naming Convention）

モジュールの名前をユビキタス言語に基づいて階層的に設定するルール。SaaSOvation の例では以下の階層構造を採用する：

```
[組織名].[境界づけられたコンテキスト名].Domain.Model.[サブドメイン概念名]
例：SaaSOvation.AgilePM.Domain.Model.Products
```

## は行

### パートナーシップ（Partnership）

組織的パターンの1つ。2つのチームが協力的な関係を持ち、計画から進捗試験まで共同でマネジメントを行う状態。両者の利害が一致しており、インターフェイスをともに検討・発展させる。

### パイプ＆フィルター

Pipe and Filter パターン。データをフィルター（処理ステップ）に通してパイプ（チャンネル）で次へ渡す処理パターン。各フィルターは独立して動作し、並列化やテストが容易。

### バリデーション（Validation）

エンティティの状態が正しいかを検証する処理。3つのレベルがある：
- 属性単位（precondition/postcondition）：セッターや操作メソッドで属性ごとに検証
- オブジェクト全体：バリデーション専用クラスを分離してオブジェクト全体の整合性を検証
- 複数オブジェクト間：ドメインサービスを使って複数エンティティ間の整合性を検証

### 貧血ドメインモデル（Anemic Domain Model）

エンティティや値オブジェクトがビジネスロジックを持たず、ただデータを保持するだけの状態になってしまうアンチパターン。ドメインサービスの多用・誤用や、アプリケーション層へのドメインロジックの流出によって引き起こされる。

### ファーストクラスオブジェクト

DDD においてリポジトリは、単なるユーティリティクラスではなく、ドメインモデルの中の一等市民（ファーストクラスオブジェクト）として扱われる。集約の永続化という責務を明確に持つ独立したオブジェクト。

### ファクトリ（Factory）

複雑なオブジェクト生成のロジックをカプセル化するオブジェクトまたはメソッド。エンティティや集約の生成が複雑な場合はコンストラクタを直接呼ぶ代わりにファクトリを使い、生成の責務を分離する。ドメインモデルにおいて直接ビジネス責務を持たない場合でも、ドメイン設計の一部として位置づけられる。生成時に不変条件を検証する責務も担う。

### ファクトリメソッド（Factory Method）

集約ルートや集約に定義された、別のオブジェクトを生成して返すメソッド。IDDD 本では「集約の振る舞い（メソッド）」としてファクトリを実装することが主に紹介されている。例：`Forum.StartDiscussionFor()` → `Discussion` を生成して返す。

### 副作用のない振る舞い（Side-Effect-Free Behavior）

値オブジェクトのメソッドは自身の内部状態を変えず、計算結果を新しい値オブジェクトとして返す。純粋関数と同様の性質であり、テストが容易になる。

### 不変（Immutable）

一度生成されたオブジェクトの状態が変更されない設計。ドメインイベントクラスはすべてのフィールドを final として宣言し、setter を提供しないことで不変性を保つ。

### 不変性（Immutability）

値オブジェクトの中心的な特性。一度生成されたオブジェクトの状態は変更できない。値を「変えたい」場合は新しいオブジェクトに丸ごと置き換える（交換可能性）。不変にすることでスレッドセーフ・キャッシュ可能・共有可能になる。

### 不変性（Immutability of Identity）

一意な識別子は一度設定したら変更しないという原則。識別子が変更されてしまうと同一性の保証が崩れるため、セッターにアサーションを記載するなどして変更を防ぐ。

### 不変条件（invariant）

オブジェクトや集約が常に満たしていなければならない整合性の条件。ファクトリは生成時に不変条件を検証する責務を持つ（例：`Forum.AssertOpen()` でフォーラムがオープン状態であることを確認）。集約ルートが不変条件の保証を担う。

### 腐敗防止層（Anti-Corruption Layer）

**略語：** ACL（Anti-Corruption Layer）

統合パターンの1つ。他のコンテキストのモデルを自コンテキストのモデルに変換する翻訳層。アダプタパターンを用いて実装することが多い。外部コンテキストの影響（腐敗）から自コンテキストのモデルを守り、クリーンに保つ。「大きな泥団子」のシステムと連携する場合に特に重要。

### 分散コンピューティングの誤謬（Fallacies of Distributed Computing）

ピーター・ドイチュ氏らが提唱した、分散システム設計における8つの誤った前提。「ネットワークは信頼できる」「遅延はゼロ」「帯域幅は無限」などが含まれる。境界づけられたコンテキストを分散配置する際にこれらの誤謬を意識した設計が重要。

### 別々の道（Separate Ways）

組織的パターンの1つ。2つのコンテキスト間で統合を行わないことを明確に宣言した状態。連携するメリットがない場合に採用する。

### べき等性（Idempotency）

同じ操作を何度繰り返しても結果が変わらない性質。メッセージング基盤では NW 障害などによりメッセージが重複配送される可能性があるため、受信側がべき等性を保って重複処理を防ぐ必要がある。

### ヘキサゴナルアーキテクチャ

Alistair Cockburn が提唱したアーキテクチャ。「ポート＆アダプター」アーキテクチャとも呼ばれる。ドメインをシステムの中心に配置し、外部との接続を「ポート」とそれに対応する「アダプター」によって実現する。テスト容易性と変更への柔軟性が高い。

### プロセスマネージャー（Process Manager）

長期プロセスの状態（どのステップまで完了したか）を管理し、複数コンテキスト間のイベント連携全体をオーケストレーションするコンポーネント。「サガ（Saga）」とほぼ同義で用いられることが多い。

### プレゼンテーションモデル（ビューモデル）

UI層においてUIの状態とUI固有のロジックを持つコンポーネント。MVVMパターンのViewModelに相当する場合もある。ドメインモデルとは独立して存在し、画面表示に特化したデータと振る舞いを持つ。ドメインオブジェクトをUIに直接渡すのではなく、UI表示に特化したデータ構造として変換したもの。ドメイン層とプレゼンテーション層の関心を分離する。

### 永続指向リポジトリ（Persistence-Oriented Repository）

対象ストレージが NoSQL などのキーバリューストアに適したリポジトリの設計スタイル。集約の情報全体を 1 つのレコード（ドキュメント）としてまとめて保存する。変更追跡の仕組みがないため、`add` ではなく `save` メソッドを使って明示的に保存する。

### ビュー（View）

UI層においてUIの表示（描画）を担当するコンポーネント。プレゼンテーションモデルの状態に基づいて画面をレンダリングする。

### 標準型 / タイプコード（Type Code）

ドメインモデルの区分や種類を表す値オブジェクト。enum（列挙型）で表現することもできるが、クラスとして実装することでドメイン固有の振る舞いを持たせることができる。コンストラクタを private にして定義済みの値のみを使用させる設計が一般的。

### ポート＆アダプター

ヘキサゴナルアーキテクチャを構成する概念。「ポート」はドメインが外部と通信するためのインターフェイス（契約）。「アダプター」はポートを実装し、具体的な技術（REST、メッセージング、DBなど）と繋ぐ部品。

### 問題領域

ドメインのうち、ビジネス上の課題・関心事を定義する側。コアドメインやサブドメインによって構成される。

## ま行

### 命じる、たずねるな（Tell-Don't-Ask）

オブジェクトに状態を問い合わせてから呼び出し側で判断するのではなく、オブジェクト自身に行動を命じるべきという設計指針。カプセル化を高め、ロジックをオブジェクト内に留める。集約設計において特に重要で、集約ルートに振る舞いを集約することと密接に関連する。

### ミニマルモデル

自コンテキストが他コンテキストのデータを参照する際に、必要最小限の情報のみを保持するモデル設計のアプローチ。外部依存を減らし、自コンテキストの自律性・シンプルさを保つことを目的とする。

### ミニレイヤ（Mini-Layer）

ドメインサービスが誤った形で使われることで生まれる、ドメインモデルを薄くする設計上の問題。ビジネスロジックをドメインサービスに集中させすぎると、エンティティや値オブジェクトがただのデータコンテナ（貧血ドメインモデル）と化す。避けるべきアンチパターン。

### モジュール（Module）

プログラミング言語における「名前空間（パッケージ）」に相当するDDDのモデリング要素。Java の `package`、C# の `namespace` を指す。DDDではソースファイルの物理的な整理にとどまらず、ドメインの概念をユビキタス言語で表現するための主要なモデリング要素として活用する。境界づけられたコンテキストとの違いとして、モジュールはクラス群をまとめるプログラミング的な仕組みであり、境界づけられたコンテキストはビジネス的な境界線（より大きな概念的分割）を指す。

### メッセージング統合

RabbitMQ などのメッセージング基盤を使ってドメインイベントを配信・購読する統合パターン。`ExchangeListener` と `NotificationReader` を組み合わせてイベントを受信する。RESTful 統合より疎結合で非同期処理に向く。べき等性の確保のために ChangeTracker を併用する。

### モデリング

業務の「モノ」とその振る舞いを中心に、不要なものを省いてドメインモデルを作り上げる作業。ドメインエキスパートと開発者が協力して行い、ユビキタス言語を使って共通の理解を形成しながら進める。

### ユビキタス言語

**略語：** UL（Ubiquitous Language）、別名：共通言語

境界づけられたコンテキスト内でチームが共有する共通言語。開発者・ドメインエキスパートが同じ言葉で話せるよう策定される。コンテキストをまたぐと同じ用語でも意味が変わるため、言語はコンテキスト内でのみ有効。コードの命名・ファクトリメソッド名にもこの言語を使う。

## ら行

### 軽量なオブザーバー（Lightweight Observer）

パブリッシャーとサブスクライバが同じプロセス／スレッドで動作し、イベントを発行した同じトランザクション内で処理されるシンプルなイベント送受信方式。IDDD 本が推奨する基本的な実装パターン。

### 結合度（Coupling）

コンポーネント間の依存の強さを示す尺度。低いほど良い。データ結合（data coupling）が最も低い結合度で、コンポーネントは引数を介してのみ通信する状態。結合度が低いほどシステムのメンテナンス性が向上する。

### 結果整合性（Eventual Consistency）

複数の境界づけられたコンテキスト・集約をまたぐ操作において、即時の整合性ではなく「最終的には整合性が取れる」ことを許容する設計方針。ドメインイベントを他コンテキストへ非同期送信することで実現される。トランザクション整合性（1集約1トランザクション）と対になる概念。

### リポジトリ（Repository）

集約の永続化（追加・削除・取得）を担うオブジェクト。クライアント（アプリケーションサービスなど）からはコレクション（リスト）を操作するかのようなAPIを提供し、データストアの種類や保存の詳細を完全に隠蔽する。インターフェイスはドメイン層に、実装はインフラストラクチャ層に置く（セパレートインターフェイス）。

### レイヤ化アーキテクチャ

ソフトウェアを「ユーザーインターフェイス層」「アプリケーション層」「ドメイン層」「インフラストラクチャ層」の4つの層に分割して構築するアーキテクチャ。上位の層から下位の層への依存のみを許容する。DDDで最初に採用されることが多いシンプルなアーキテクチャ。

### レイヤスーパータイプ（Layer Supertype）

レイヤ共通の処理をまとめる抽象クラス。エンティティの文脈では、代理識別子（id）を protected で保持する親クラス Entity を定義し、サブクラスから直接アクセスできるが外部からは隠蔽するパターンとして使われる。

## 英数字

### Advanced Message Queuing Protocol

**略語：** AMQP

メッセージングキューのための標準プロトコル。RabbitMQ が採用しており、P2P のメッセージング通信を実現する。

### BacklogItem（バックログアイテム）

SaaSOvation サンプルにおける集約ルートの例。スクラム開発における課題（バックログ）を表すエンティティ。`BacklogItemId`・`TenantId`・`ProductId` などの識別子値オブジェクトを持つ。

### BusinessPriority（値オブジェクト）

本書のサンプルドメイン（プロジェクト管理）における「ビジネス優先度」を表す値オブジェクト。BusinessPriorityRatings（benefit・penalty・cost・risk の4つの数値）を内包し、優先度の計算を行う。

### BusinessPriorityCalculator

SaaSOvation のサンプルコードに登場するドメインサービスの具体例。BacklogItem のコレクションをリポジトリから取得し、ビジネス優先度（benefit・penalty・cost・risk）の合計値を計算して、BusinessPriorityTotals という値オブジェクトとして返す。

### BusinessPriorityRatings（値オブジェクト）

BusinessPriority が内包する値オブジェクト。benefit・penalty・cost・risk の4属性を保持。コンストラクタでのみ値をセットし、セッターを持たない（不変）。

### BusinessPriorityTotals（値オブジェクト）

ビジネス優先度の集計値を保持する値オブジェクト。totalBenefit・totalPenalty・totalCost・totalRisk・totalValue・totalPercent を保持。全属性をコンストラクタで受け取り、セッターを持たない（不変）。

### CollabOvation

SaaSOvation のコラボレーション境界づけられたコンテキストサービス。チャット・フォーラム・カレンダー・共有資料などのコラボレーション機能を提供する。

### DomainEvent インターフェース

すべてのドメインイベントクラスが実装する基底インターフェース。occurredOn() メソッドを定義し、イベント発生日時を返す。

### DomainEventPublisher

イベントをサブスクライバに通知するシステム共通サービス。IDDD 本のサンプルコードで Java / C# 版が提供されている。パブリッシャーはサブスクライバの登録と、対応するイベント種別への配信を担う。

### DomainEventSubscriber

イベント受信時の処理を定義するシステム共通インターフェース。メールの送信やイベントストアへの保存など、処理別にインターフェイスを実装する。

### EntityWithCompositeId

IDDD サンプルコードにおける基底クラス。複合IDを持つエンティティの親クラス。`BacklogItem` はこれを継承する。

### エンタープライズサービスバス

**略語：** ESB（Enterprise Service Bus）

サービス間のメッセージングを仲介するミドルウェア。SOAにおいてサービスを疎結合に統合するために使用される。

### エンティティ（Entity）

一意な識別子（ID）を持ち、その識別子によって同一性を判断するオブジェクト。属性値が変化してもIDが同じであれば同一のエンティティとみなす。ライフサイクル（生成・変更・消滅）を持つ概念のモデリングに使われる。

### EventSourcedRootEntity

SaaSOvation における集約ルートの基底クラス。イベントソーシングをサポートする。`Forum` クラスはこれを継承し、`StartDiscussionFor` ファクトリメソッドを持つ。

### equals / hashCode

Javaにおいてオブジェクトの同一性を比較するメソッド。エンティティでは一意な識別子の値を使って equals を実装する。同じIDを持つオブジェクトは同一エンティティとして扱われる。

### HATEOAS

Hypermedia As The Engine of Application State。REST の特性の一つ。レスポンスに関連リソースへのリンクを含めることで、クライアントがサーバーの知識なしにリソース間を遷移できるようにする。

### ICollaboratorService

コラボレーションコンテキストにおける変換用ドメインサービスのインターフェース。`GetAuthorFrom`・`GetCreatorFrom`・`GetModeratorFrom`・`GetOwnerFrom`・`GetParticipantFrom` の各ファクトリメソッドを定義する。

### IdOvation

SaaSOvation における認証・認可を担う汎用サブドメインの境界づけられたコンテキスト。CollabOvation・ProjectOvation の両方から参照される。

### IoC/DIコンテナ

**略語：** IoC/DI（Inversion of Control / Dependency Injection）

サービスのインスタンス生成と依存関係の注入を自動的に管理するコンテナ。Spring（Java）や Unity Container（.NET）が代表例。セパレートインターフェイスを使ったドメインサービスの生成手段として推奨される方法の1つ。

### キーバリューストア（Key-Value Store）

NoSQL データベースの一種で、キーと値の組み合わせでデータを管理するストレージ（例：LevelDB、MongoDB）。永続指向リポジトリの対象ストレージとして利用される。IDDD 本では「境界づけられたコンテキストの短縮名＋集約の名前＋一意な識別子」の組み合わせでキーを構成することを推奨している。

### デメテルの法則（Law of Demeter）

オブジェクトは直接知っているオブジェクトのメソッドのみを呼び出すべきという原則。「友人にのみ話しかけ、見知らぬ人には話しかけない」。連鎖参照（`a.getB().getC().doSomething()`）を避け、疎結合なコードを実現する。集約設計において、集約内部オブジェクトへの連鎖参照を禁止する根拠の一つ。

### LevelDB

Google が開発したキーバリューストア型の組み込み NoSQL データベース。永続指向リポジトリの実装例として IDDD 本で紹介されている。`LevelDBUnitOfWork` を使ってデータの保存・取得を行う。

### occurredOn

ドメインイベントが発生した日時を保持するプロパティ。DomainEvent インターフェースが持つ共通プロパティの一つ。コンストラクタ内で new Date() によりセットされる。

### O/Rマッパー（Object-Relational Mapper）

オブジェクトとリレーショナルデータベースのテーブルの対応付け（マッピング）を自動化するツール・ライブラリ（例：Hibernate）。コレクション指向リポジトリの実装で活用される。変更追跡（ダーティチェッキング）により、明示的な `save` を呼ばなくても変更をDBに反映できる。

### ProjectOvation

SaaSOvation のスクラム管理境界づけられたコンテキストサービス。バックログアイテム・スプリント・タスク等を管理する。CollabOvation に依存する形で統合されている。

### チームモジュール（Teams Module）

SaaSOvation の AgilePM コンテキストにおける `Domain.Model.Teams` サブモジュール。ProductOwner・Team・TeamMember の 3 つの集約を含む。Tenants モジュールへの参照と Products モジュールからの参照（双方向）が存在する。

### 長期プロセス（サーガ）

Long Running Process / Saga。複数の境界づけられたコンテキストをまたぐ、複数ステップにわたる非同期の処理フロー。単一のトランザクションでは完結できない複雑なビジネスプロセスを表現する。各サブプロセスをドメインイベントで連鎖させ、プロセスマネージャーが全体の状態を管理する。分散トランザクションの代替として使用される。「サガ（Saga）」とも呼ばれる。

### RabbitMQ

AMQP（Advanced Message Queuing Protocol）を使用したオープンソースのメッセージングミドルウェア。P2P のキューイングモデルでメッセージング通信を実現する。IDDD 本でイベント送受信の実装例として紹介されている。

### REST

Representational State Transfer。Roy T. Fielding の博士論文に登場したアーキテクチャスタイル。以下4つの特性を持つ：
1. リソースを重要な URI で識別可能
2. ステートレスな通信
3. HTTP メソッド（GET/POST/PUT/DELETE）でリソース操作を表現
4. ハイパーメディア（HATEOAS）によるリソース間ナビゲーション

### RESTful

REST の特性に従ったシステムやサービスの呼び方。

### RESTful 統合（リアルタイム連携）

ユーザーアクションのたびに REST サービスへリクエストを発行し、リモートコンテキストのリソースをリアルタイムに参照する統合パターン。即時性が高いが呼び出しコストも高い。

### RESTful 統合（バッチ / タイマー連携）

定期的に REST リソースをポーリングし、変更があった場合のみ自コンテキストのモデルを更新する統合パターン。`Notification` リソース（`/notifications`）を公開し、受信側が最終取得 ID 以降のイベントをバッチ取得する。リアルタイム連携よりも疎結合。

### SaaSOvation

IDDD 本のサンプルプロジェクト。以下の境界づけられたコンテキストを持つ：
- CollabOvation：コラボレーションツール（チャット・フォーラム・カレンダー等）
- ProjectOvation：スクラムプロジェクト管理ツール
- IdOvation：認証・認可の汎用サブドメイン

### 事後条件（postcondition）

メソッド呼び出し後に満たされるべき条件。操作の結果として状態が正しいかを検証する。

### 事前条件（precondition）

メソッド呼び出し前に満たされるべき条件。エンティティの操作が呼び出される前に確認する。

### プロダクトモジュール（Products Module）

SaaSOvation の AgilePM コンテキストにおける `Domain.Model.Products` サブモジュール。Sprint・BacklogItem・Release を含む複数のサブモジュールで構成されており、依存関係が複雑になりやすい。

### クエリ（CQRS）

CQRSにおけるクエリは、データを参照して返す命令。状態を変更しない。読み取りに最適化された非正規化データストアから直接データを取得することが多い。

### UUID / GUID

Universally Unique Identifier / Globally Unique Identifier の略。アプリケーション側で一意な識別子を生成する際に使用するアルゴリズム。人間が読みにくい文字列になるため通常は利用者に非表示とする。

### 値オブジェクト（Value Object）

DDDにおけるモデリングの基本構成要素の一つ。属性と振る舞いを持つが、エンティティとは異なり一意な識別子（ID）を持たない。属性の値によって等値性を判断する不変オブジェクト。特性：計測／定量化／説明、不変性、概念的な統一体、交換可能性、等値性、副作用のない振る舞い。集約内のオブジェクトは可能な限り値オブジェクトとして設計することが推奨される。

### ベンダーツリー（Vendor Tree）

MIME タイプの `vnd.` プレフィックスで示されるカテゴリ。公開されている製品に関連するメディアタイプとして用いられる。カスタムメディアタイプを「公表された言語」として定義する際に使用する。

### ビルダー（Builder）

複雑なオブジェクトの生成手順を段階的に組み立てるデザインパターン。ファクトリ実装の選択肢の一つ。

### ChangeTracker（追跡トラッカー）

メッセージング統合において、処理済みのイベント ID を記録し、未受信イベントのみを処理するためのコンポーネント。「最終受信イベント ID」を保持することで、べき等性を確保し重複処理を防ぐ。

### Commandパターン（Command Pattern）

アプリケーションサービスへの入力を `XxxCommand` クラスとして表現するパターン。必要最小限のデータをカプセル化し、ユースケースを明示的に表現する。例：`ProvisionTenantCommand`。

### DPO（Data Presentation Object）

DTO と同様にドメインモデルの情報を UI 層に渡すためのオブジェクト。特にプレゼンテーション目的を強調する文脈で使われる。

### DTOアセンブラ／Transformer

集約などのドメインオブジェクトを DTO に変換するクラス。クライアントの種類（Web/ネイティブ等）ごとに実装を切り替えることができる。例：`CalendarWebDTOTransformer`。

### Notification（通知）

ドメインイベントを外部の境界づけられたコンテキストに連携するための仕組み。`Notification` クラスはイベント（`event`）とその識別情報（`notificationId`・`occurredOn`・`typeName`）をまとめて格納する。RESTful サービスまたはメッセージング基盤（RabbitMQ など）経由で配信される。

### NotificationReader

`Notification` からイベント情報を読み取るクラス。XPath 形式のパス（例：`/notification/occurredOn`）を使ってイベントのプロパティにアクセスする。イベントの受信側コンテキストが使用する。

### UI層（User Interface Layer）

利用者が操作するインターフェイスを提供する層。Webシステム・Web APIサービス・GUIクライアント・スマートフォンアプリなどが含まれる。ドメインモデルの情報を表示・更新の形で描画する役割を担う。プレゼンテーションモデルや DTO を介してドメイン層と接続する。
