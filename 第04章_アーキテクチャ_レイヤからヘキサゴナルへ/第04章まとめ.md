# 第4章「アーキテクチャ」〜レイヤからヘキサゴナルへ〜

（page_0050〜page_0069）

## 4.1 本章で紹介するアーキテクチャの種類

DDDではアーキテクチャに関する制約はない。DDD の中心に置かれるのは「ドメインモデル」であり、アーキテクチャはそれを実現するための手段にすぎない。DDD が採用できるアーキテクチャは以下の通り。

1. レイヤ化アーキテクチャ
2. DIP に基づいたレイヤ化アーキテクチャ
3. ヘキサゴナルアーキテクチャ（ポート＆アダプター）
4. SOA（サービス指向アーキテクチャ）
5. REST
6. CQRS（コマンドクエリ責務分離）
7. イベント駆動アーキテクチャ / パイプ＆フィルター
8. イベントソーシング

---

## 4.2 特定のアーキテクチャに依存しない DDD

DDD は特定のアーキテクチャに縛られない。SaaSovation チームの CIO が各フェーズでアーキテクチャを段階的に進化させた例が示されている。

## 4.3 SaaSovation の CIO に聞くアーキテクチャの変遷

**● 表 4.1　DDDシステムのアーキテクチャの変遷（SaaSovation）**

| 順番 | 背景 | 選定アーキテクチャ | 詳細 |
|-----|------|----------------|------|
| 1 | デスクトップアプリケーションとして初期開発 | レイヤ化アーキテクチャ | C/SシステムのためクライアントのアプリケーションとDB層で構築 |
| 2 | SaaSモデルへの移行に加え、アドオン機能の複雑化 | レイヤ化アーキテクチャに「依存関係逆転の原則（DIP）」を導入 | DIとユニットテストを導入して品質向上。ドメインに集中し、後にUI層や永続化層を入れ替え |
| 3 | スマホへの対応、認証やセキュリティの一元管理、管理ツール、BIツールなど多数の要望 | ヘキサゴナルアーキテクチャへの移行／RESTの導入 | ポート＆アダプターの手法によりNoSQLやメッセージングなどに対応し、新機能への要望に対応 |
| 4 | レガシーシステムのユーザーへのサービスを提供する際の統合 | SOAの導入 | アプリケーション統合フレームワーク（ESB Mule）を用いてデータをサービスで集約 |
| 5 | ダッシュボードの機能拡張によるユーザー別情報の表示増加 | CQRSの導入 | コマンド（入力）とクエリ（描画イベント）を分解して複雑なダッシュボードの表示が可能 |
| 6 | 時間のかかる処理のユーザーストレスやタイムアウトへの対応 | イベント駆動アーキテクチャの導入 | 「パイプ＆フィルター」パターンを導入 |
| 7 | 30億ドルで買収され、新ユーザー増加とシステム統合によりデータが爆発的に増加 | 分散変更処理への対応 | 「パイプ＆フィルター」を分散処理対応するため、「長期プロセス（サーガ）」を導入しインフラを改善 |
| 8 | 政府より法的な監査の要求 | イベントソーシングの導入 | ドメインモデルでのすべての変更内容を追跡することでコンプライアンス問題に対応 |

この表では、DDDを採用していたからこそ柔軟にアーキテクチャを拡張できるというストーリーを理解できる。

## 4.4 レイヤから DIP、そしてヘキサゴナルへ

大きな変化としては、(1) レイヤ化アーキテクチャ → (2) DIP に基づいたレイヤ化アーキテクチャ → (3) ヘキサゴナルアーキテクチャ、の変遷がある。

【図 4.1】DDDでのアーキテクチャの変遷（図）

- レイヤ化アーキテクチャ → DIPを基づいたレイヤ化アーキテクチャ → ヘキサゴナルアーキテクチャ（ポート＆アダプター）
- 右へ進むほどドメイン層の独立性が高まる。

---

## 4.5 レイヤ化アーキテクチャとは

ソフトウェアをいくつかの層（レイヤ）に分割して構築する方式。DDDの観点では「ユーザーインターフェイス層」「アプリケーション層」「ドメイン層」「インフラストラクチャ層」の4層が一般的。

### レイヤ化アーキテクチャの DDD

DDDのレイヤ化アーキテクチャで最も重要なのはドメイン層の独立性。上位の層から下位の層への依存のみを許容する。

#### ユーザーインターフェイス層

ドメインオブジェクトを直接UIに渡す場合と、渡さない場合がある。後者では「プレゼンテーションモデル（ビューモデル）」を作成する。

ユーザーインターフェイス層への入力はDTOで受け取り、コントローラーはそれをアプリケーションサービスまたはドメインオブジェクトへ直接渡すことで処理する。

#### ドメイン層

すべてのビジネスロジックとビジネスルールを含む。エンティティ、値オブジェクト、ドメインサービス、イベント等がここに置かれる。

#### インフラストラクチャ層

「リポジトリ」を使用してデータの永続化を担当する。「インフラストラクチャ層」にはメッセージングやメールなども含まれる。

---

## 4.6 依存関係逆転の原則（DIP）を用いたレイヤ化アーキテクチャ

DIPとは Robert C. Martin が提唱したもので（SOLID原則のD）、これを使うとドメイン層からインフラストラクチャ層への依存をなくすことができる。プラグインのように依存性を注入することで、インフラストラクチャ層をドメイン層から切り離す。

---

## 4.7 ヘキサゴナルアーキテクチャ

DIPの問題をドメインの側面から解決したのがヘキサゴナルアーキテクチャ。Alistair Cockburn によって提唱された。ヘキサゴン（六角形）は複数のポートを持ち、それぞれにアダプターをとりつける。

### ヘキサゴナルを構成する「ポート＆アダプター」

ポート＆アダプターにより、複数の口（ポート）が用意され、初期化処理もテストも同じインターフェイスで実施できるようになる。

ヘキサゴナルアーキテクチャ：ドメインは中心に配置され、さまざまなアダプター（REST API、メッセージング、UI等）から切り離される。

### ヘキサゴナルアーキテクチャのメリット

- アーキテクチャ全体を通じてドメインオブジェクトがピュアに保たれる
- テストがしやすくなる（インフラストラクチャに依存しない）
- 複数の入出力ポートを自由に追加できる

## 4.9 ヘキサゴナルで使用される REST

DDDで Web サービスを構築するときによく利用される技術が REST。SaaSovation チームは、モバイル対応、認証管理、BIレポートの対応において REST を採用した。

### RESTful なシステムとは

REST（Representational State Transfer）は、Roy T. Fielding 氏が書いた博士論文にて登場したアーキテクチャスタイル。REST に従ったサービスを「RESTful」と呼ぶ。

【図 4.3】 RESTの特性と DDD

RESTの特性：

(1) **リソースを重要な URI で識別可能** — リソースごとに一意な URL（URI）を割り当て、識別する（例：`/user/{id}`）。
(2) **ステートレスな通信** — REST は HTTP メッセージのやり取りだけでシステムとコミュニケートし、セッション等の状態管理を持たない。各 HTTP メッセージには処理に必要な情報をすべて含める。
(3) **GET / POST などの命令に特化したリソース操作** — HTTP メソッド（GET、POST、PUT、DELETE）で操作を表現する。
(4) **関連リソースへのナビゲーションを担う「ハイパーメディア」** — リソースのレスポンスに他リソースへのリンクを含め、クライアントがリンクをたどることで遷移できるようにする（HATEOAS：Hypermedia As The Engine of Application State）。

### DDD で REST を活用する

REST は標準規格を使うため、スケーラビリティやキャッシュのような機能が利用できる。また、REST のコントロールとリソースへのアクセスはヘキサゴナルアーキテクチャの一部として実装できる。

## 4.10 コマンドクエリ責務分離 (CQRS：Command Query Responsibility Segregation)

SaaSovation チームはユーザー別の情報が増加したことをきっかけにコマンドとクエリを分離する CQRS を採用した。

### コマンドクエリ責務分離とは

CQRS は CQS（Command Query Separation）のアーキテクチャ版で、Bertrand Meyer が提唱した「オブジェクト指向入門（下）」で説明されている。

- **1 コマンド (Write)** — オブジェクトの状態を変更するメソッド。値を返してはいけない。
- **2 クエリ (Read)** — メソッドが参照型で値を返す。状態を変更してはいけない。

【図 4.4】 CQRSパターンとDDDを組み合わせた場合の処理の流れ

処理の流れ：
- コマンド側：UIからコマンドを受け付け → コマンドハンドラ → ドメインモデル → リポジトリ → 書き込みDB
- クエリ側：UIからクエリを受け付け → クエリモデル → クエリサービス → 読み取りDB（非正規化）

## 4.11 DDDにおけるCQRSでの流れ

### (1) コマンド処理（コマンドプロセッサ/コマンドハンドラ）

コマンドプロセッサは、コマンドオブジェクトを受け取り、対応するコマンドハンドラに処理を委譲する。コマンドはドメインオブジェクトのメソッドを呼び出し、状態変化を起こす。コマンドの実行後は非同期に読み取りモデル（クエリモデル）のデータストアを更新する。

### (2) コマンドモデル (Write モデル)

通常、複数のコマンドハンドラが存在する。コマンドモデルの更新処理は複数の手法がある：
1. **コマンドプロセッサ** — アプリケーション層の管理者として振る舞い、コマンドに対応したハンドラにディスパッチする。
2. **トランザクションスクリプト** — データの変更を直接的な手続きとして記述する。
3. **コマンドモデル用テキストストア**

### (3) コマンドモデル用テキストストア

コマンドモデルの更新には基本的にリレーショナルデータベースが使われる。データのスキーマがドメインオブジェクトに沿った形で設計される。

### (4) クエリモデル（ビュー）を更新（イベントのサブスクライバ）

サブスクライバ（購読者）が発行されたドメインイベントを受信して処理を行い、非正規化されたビュー用データストアを更新する。

### (5) クエリモデル非正規化データストア

クエリモデルのデータストアは、画面表示に最適化された非正規化データが格納される。高速なデータ参照が可能になる。

### (6) クエリモデル (Read モデル)

クエリモデルは、画面表示のためのデータを提供するシンプルな層。ORM などを使わず生のSQLやXMLでシンプルに実装される場合が多い。

### (7) クエリ結果 (Read モデル)

データベースからクエリによってデータを取得し、クライアントに返す。

### CQRS で注意/非採用の指針

「結果整合性」とは「厳密な整合性（Strong Consistency）」と対比される概念で、データの更新が全ノードに即時反映されないが、最終的には一致する状態を指す。

CQRSで注意すべき点：
- CQRS は結果整合性を持つシステムで利用されることが多い
- 厳密な整合性が必要な場合はCQRSが向かない場合もある
- すべてのシステムにCQRSを採用すべきではない（オーバーエンジニアリングになりうる）

## 4.12 イベント駆動アーキテクチャ

イベント駆動アーキテクチャは、イベントのやり取りによってシステムの処理を組み立てる方式。SaaSovation チームは時間がかかる処理のユーザーストレスやタイムアウトへの対応のために採用した。

### 電話番号一覧から条件一致する件数を探すサンプル

【図 4.5】 ヘキサゴナルシステムにおけるイベント連鎖イメージ

ヘキサゴナルシステムにおいてイベントを発行すると、イベントは複数の連鎖を経て各コンポーネントにポーリングされ、最終的に全ハンドラが実行される。

イベント駆動の実装例：「パイプ＆フィルター」パターン、「長期プロセス（サーガ）」を利用した電話番号一覧からの条件一致処理。

### パイプ＆フィルターによるパイプライン

【図 4.6】「パイプ＆フィルター」による「長期プロセス」の処理フロー

イベント駆動のシンプルな例が「パイプ＆フィルター」というスタイル。

フィルター（処理器）が渡された情報の一部を処理してパイプ（次の処理）に渡すという形式。各フィルターは独立して動作し、数が多くなるとそれぞれのコンポーネントは小さく分割されてテストがしやすくなる。

### 長期プロセス（サーガ）による並列パイプライン

連続した処理の中にサブステップが多く、時間がかかる場合は「長期プロセス（サーガ）」を使う。サーガは状態を管理し、各サブプロセスを非同期に実行できる。

### イベントを記録して再現できるイベントソーシング

【図 4.7】 イベントソーシングを活用したDDDのシステム（図、page_0067）

SaaSovation チームは政府より法的な監査の要求があり、イベントソーシングを導入した。

イベントソーシングとは、システムの状態変化をイベントとして記録し続けることで、過去の任意の時点でシステムの状態を再現できる方式。

- 状態ではなくイベントログを永続化する
- イベントを再生（リプレイ）することで、現在の状態を再構築できる
- コンプライアンス、監査ログ、デバッグなどに威力を発揮する

### データファブリック（グリッドコンピューティング）

データファブリックとはイベントソーシングの上に乗るアーキテクチャであり、すべてのオブジェクトの状態変化をイベントとして記録し、分散環境に展開する方式（例：Pivotal GemFire、Oracle In-Memory、Coherence など）。

用途例：
- キャッシュ保存
- マルチアプリケーションへのレプリカ提供
- イベント複製による全データのバックアップ
- 結果整合性のサポート
- 拡張更新処理によるBI（ビジネスインテリジェンス）の処理管理
