# 用語集（第4章）

## レイヤ化アーキテクチャ

ソフトウェアを「ユーザーインターフェイス層」「アプリケーション層」「ドメイン層」「インフラストラクチャ層」の4つの層に分割して構築するアーキテクチャ。上位の層から下位の層への依存のみを許容する。DDDで最初に採用されることが多いシンプルなアーキテクチャ。

## DIP（依存関係逆転の原則）

Dependency Inversion Principle。Robert C. Martin が提唱した SOLID 原則の「D」。上位モジュールが下位モジュールに依存するのではなく、両者が抽象（インターフェイス）に依存するよう設計する原則。これによりドメイン層がインフラストラクチャ層に直接依存することを避けられる。

## ヘキサゴナルアーキテクチャ

Alistair Cockburn が提唱したアーキテクチャ。「ポート＆アダプター」アーキテクチャとも呼ばれる。ドメインをシステムの中心に配置し、外部との接続を「ポート」とそれに対応する「アダプター」によって実現する。テスト容易性と変更への柔軟性が高い。

## ポート＆アダプター

ヘキサゴナルアーキテクチャを構成する概念。「ポート」はドメインが外部と通信するためのインターフェイス（契約）。「アダプター」はポートを実装し、具体的な技術（REST、メッセージング、DBなど）と繋ぐ部品。

## SOA（サービス指向アーキテクチャ）

Service Oriented Architecture。ソフトウェアをサービスの集合として設計するアーキテクチャスタイル。SaaSovation の設計では ESB（エンタープライズサービスバス）を使ってヘキサゴナルアーキテクチャ内のドメインとサービスを接続する。

## REST

Representational State Transfer。Roy T. Fielding の博士論文に登場したアーキテクチャスタイル。以下4つの特性を持つ：
1. リソースを重要な URI で識別可能
2. ステートレスな通信
3. HTTP メソッド（GET/POST/PUT/DELETE）でリソース操作を表現
4. ハイパーメディア（HATEOAS）によるリソース間ナビゲーション

## RESTful

REST の特性に従ったシステムやサービスの呼び方。

## HATEOAS

Hypermedia As The Engine of Application State。REST の特性の一つ。レスポンスに関連リソースへのリンクを含めることで、クライアントがサーバーの知識なしにリソース間を遷移できるようにする。

## CQRS（コマンドクエリ責務分離）

Command Query Responsibility Segregation。コマンド（状態変更）とクエリ（状態参照）を別々のモデルに分離するアーキテクチャパターン。読み取りモデルを非正規化することで、複雑なクエリを高速に処理できる。

## CQS（コマンドクエリ分離）

Command Query Separation。Bertrand Meyer が提唱した設計原則。メソッドは状態を変更する「コマンド」か値を返す「クエリ」のどちらか一方であるべき、という原則。CQRSはこれをアーキテクチャレベルに拡張したもの。

## コマンド（CQRS）

CQRSにおけるコマンドは、ドメインオブジェクトの状態を変更する命令。戻り値を返さない。コマンドハンドラ（またはコマンドプロセッサ）が受け取り、対応するドメインモデルに処理を委譲する。

## クエリ（CQRS）

CQRSにおけるクエリは、データを参照して返す命令。状態を変更しない。読み取りに最適化された非正規化データストアから直接データを取得することが多い。

## 結果整合性

Eventual Consistency。分散システムにおいて、すべてのノードへの更新が即時に反映される「厳密な整合性（Strong Consistency）」とは異なり、最終的には全ノードのデータが一致する性質。CQRSはこの結果整合性を前提とすることが多い。

## イベント駆動アーキテクチャ

Event-Driven Architecture。システム内の状態変化を「イベント」として発行し、それを受け取ったサブスクライバが処理を行うアーキテクチャスタイル。非同期処理や疎結合を実現しやすい。

## パイプ＆フィルター

Pipe and Filter パターン。データをフィルター（処理ステップ）に通してパイプ（チャンネル）で次へ渡す処理パターン。各フィルターは独立して動作し、並列化やテストが容易。

## 長期プロセス（サーガ）

Long Running Process / Saga。複数のサブステップからなる長時間かかる処理を管理するパターン。各サブプロセスをイベントで連鎖させ、状態を管理しながら非同期に実行する。分散トランザクションの代替として使用される。

## イベントソーシング

Event Sourcing。ドメインオブジェクトの現在の状態を保存するのではなく、状態を変化させたイベントのログを記録することで状態を管理する手法。過去の任意の時点でシステムの状態を再構築（リプレイ）できる。監査ログやコンプライアンス対応に有効。

## データファブリック（グリッドコンピューティング）

Data Fabric。イベントソーシングを基盤として、すべてのオブジェクトの状態変化をイベントとして記録し、分散環境に展開するアーキテクチャ。Pivotal GemFire、Oracle In-Memory Coherence などが実装例。キャッシュ、レプリカ提供、結果整合性のサポートなどに使われる。

## プレゼンテーションモデル（ビューモデル）

ドメインオブジェクトをUIに直接渡すのではなく、UI表示に特化したデータ構造として変換したもの。ドメイン層とプレゼンテーション層の関心を分離する。

## DTO（データ転送オブジェクト）

Data Transfer Object。層間でデータを受け渡すための単純なデータ構造。ユーザーインターフェイス層への入力をDTOとして受け取り、アプリケーションサービスやドメインオブジェクトへ渡す。

## ESB（エンタープライズサービスバス）

Enterprise Service Bus。サービス間のメッセージングを仲介するミドルウェア。SOAにおいてサービスを疎結合に統合するために使用される。
