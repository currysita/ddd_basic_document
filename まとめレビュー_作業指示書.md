# 章まとめレビュー 作業指示書
- 以下は画像ファイルがあった頃のレビューに使用した物です。今は画像ファイルがなく、このままでは使用できない事を踏まえてください。
- 将来的に本文のレビューが必要になったときに、観点が必要になる可能性があります。そのためこのまま残してあります。

## レビューの目的

各章の `第XX章まとめ.md` を画像（原書スキャン）と目次と照合し、節構造・内容・表記の正確さを検証する。章まとめは問題集の解答根拠となるため、誤りがあると問題集全体に波及する。

---

## レビュー前に必ず読むファイル

| ファイル | 目的 |
|---|---|
| 対象章の画像ファイル（`images\chapters\第XX章_*\page_XXXX.png`） | 原書との照合。節タイトル・内容・図番号の確認 |
| `目次.md` | 節番号・節タイトルの正式名称の確認 |
| 対象章の `用語集\第XX章_用語集.md` | 用語定義の照合 |

> **注意**：画像ファイルは `.jpg` ではなく `.png` 形式。検索時は `*` パターンで拡張子を確認してから操作すること。

---

## レビューの観点

### 1. 節構造（最優先）

- 目次の節番号・節タイトルと `## X.Y タイトル` の見出しが一致しているか
- 節番号の重複・欠番・順序ミスがないか（例：8.3が2つある、8.4が3つあるなど）
- 節タイトルの表記が目次と一字一句一致しているか（誤字・略称を使っていないか）
- 本来サブセクション（`###`）にすべき内容が独立した節（`##`）になっていないか

### 2. 内容の正確さ

- 画像（原書）の記述と整合が取れているか
- 定義・手順・図の説明が画像と一致しているか
- 原書にない記述（AIが補完・推測した内容）が混入していないか
- 原書にある重要な記述が欠落していないか
- 図番号（「図X.Y」）が原書と一致しているか

### 3. 用語・表記

- 用語集の定義と整合しているか
- 誤字・誤変換がないか
- カタカナ表記が一貫しているか（例：「サブスクライバー」と「サブスクライバ」の混在）
- クラス名・メソッド名・インターフェース名のスペルが正しいか

### 4. 構造の明瞭さ

- 節の冒頭で「この節で何を説明するか」が分かるか
- 箇条書き・図・コードブロックが適切に使われているか
- 節間の繋がりが自然か（「続けて」「次に」などの誘導が原書に即しているか）

---

## 修正ルール

- 節タイトルは **目次の表記に完全一致** させる
- 内容の修正根拠は必ず画像（原書）で確認する。確認できない場合は修正せず「要確認」として記録する
- 画像では読み取れない細部（コード・脚注など）は既存の記述を維持し、不確かな場合は `<!-- 要確認 -->` コメントを付ける
- 修正内容はレビュー記録ファイルに記録する

---

## 表・図が読み取りにくい場合の対応

スキャン画像で表や図の文字が小さく、内容を正確に読み取れない場合は、**修正せずにユーザーに相談する**。

### 相談のタイミング

- 表のセル内の文字が潰れて読めない
- 図の凡例・ラベルが不明瞭
- コードブロックの一部が欠けている

### 相談の仕方

レビュー中に発見したら、その場で以下を伝える：

```
【要確認】XX節の表X.Xの内容が画像では読み取りにくいです。
別途、高解像度の画像または該当ページの画像を提供いただけますか？
```

### ユーザーからの画像提供

ユーザーが章フォルダ内に別途画像ファイルを配置する場合がある（例：`表01_2.png`）。  
レビュー中に読み取りにくい箇所があった場合は、対象章フォルダ内を `*` パターンで検索し、追加画像がないか確認すること。

### 相談後の処理

- 提供画像で内容を確認できたら修正し、レビュー記録の「状態」を「解決済み」に更新する
- 提供されなかった場合は「要確認」のまま残す

## レビュー記録の保存先

```
第XX章_*/まとめレビュー記録.md
```

例：`第08章_ドメインイベント_出来事を記録して活用\まとめレビュー記録.md`
