# 第5章「エンティティ」〜一意な識別子で同一性を識別〜

## 章の概要

DDDにおけるエンティティの概念を解説する章。エンティティとは何か、値オブジェクトとの違い、DDDでよく起きるモデリングの失敗、エンティティの発見・設計・実装方法を順に学ぶ。  
事例として SaaSOvation（マルチテナント SaaS システム）の「ユーザーエンティティ」「従業員エンティティ」を使って具体的な実装が示される。

---

## 5.1 エンティティと値オブジェクト

### エンティティとは

DDD における「エンティティ（Entity）」とは、一意な識別子（ID）を持ち、その識別子によって同一性を判断するオブジェクトのこと。  
同じ属性値を持つ別のオブジェクトがあっても、ID が異なれば別のエンティティとみなす。

> 「エンティティとは、主に同一性によって定義されるオブジェクトのことである」  
> （DDD 本 91 ページより）

### 値オブジェクトとの違い

【図 5.1 エンティティ（Entity）と値オブジェクト（Value Object）】

| 比較軸 | エンティティ | 値オブジェクト |
|--------|------------|--------------|
| 同一性の基準 | 識別子（ID） | 属性値の等価性 |
| 可変性 | ミュータブル（状態変化あり） | イミュータブル（不変） |
| 主な用途 | ライフサイクルを持つ物 | 概念的な値・属性のまとめ |

例：名前・住所などは値オブジェクト。ユーザー・注文などはエンティティ。

### DDD でよく起きる失敗

- ドメインモデリングを行わず、テーブル設計やデータ中心アプローチになってしまう。
- エンティティの同一性よりも細かい属性や振る舞いに気を取られ、ドメインの本質を見失う。
- Evans 氏が指摘するように、エンティティの同一性に注目してシンプルに保つことが重要。

## （4）「一意な識別子」の設計

エンティティを一意に識別する「一意な識別子（Identity）」を設計する。  
SaaSOvation の「テナント ID」「ユーザー名」を例に設計する。チャネル ID には「テナント ID (UUID)」を含めている。

### 一意な識別子の生成パターン

【図 5.5 「一意な識別子」の4つの生成パターン】

| # | 生成者 | 例 | メリット | デメリット |
|---|--------|---|---------|-----------|
| 1 | ユーザーが「入力」 | USR123 | 識別子を生成する仕組みが不要 | 重複せず適切な識別子を利用者に登録してもらうワークフローやチェックの仕組みが必要 |
| 2 | アプリケーションが「生成」 | "912649a0-0ed2-4071-8af6-1a350d4adbd0"（UUID のみ）や "2017-12-31-USER-1a350d4adbd0"（意味ある値と UUID の組み合わせ） | UUID（GUID）の生成アルゴリズムを用いて一意な値を軽量に生成 | 人間には理解しがたい文字列になる（そのため通常は利用者には非表示） |
| 3 | 永続化メカニズム（DB）が「生成」 | 10001 | Oracle のシーケンスなどの仕組みを用いて連番を簡単に生成 | DB 依存が高く性能面の課題がある（早期生成時におけるキャッシュの仕組みの検討が必要） |
| 4 | 他の境界づけられたコンテキストから「割り当て」 | USR123 や UUID など | 識別子生成の仕組みを他のコンテキストに託せる | 他のコンテキストに対して識別子を取得するための検索／マッチング／割り当ての仕組みが必要 |

SaaSOvation の「ユーザーエンティティ」では、テナント ID は「アプリケーションでの生成（UUID 生成）」を行い、ユーザー名は「ユーザーが入力」という形を取っている。

### 一意な識別子の生成タイミング（早期、遅延）

【図 5.5 テナントの識別子の生成（Java）コード例】

```java
public class TenantId implements Serializable {
    private String id;

    public TenantId(String anId) {
        super();
        this.setId(anId);
    }

    private void setId(String anId) {
        // ...
        this.id = anId;
    }
}
```

【表 5.2 一意な識別子の生成タイミング】

| # | 生成タイミング | メリット | デメリット |
|---|-------------|---------|-----------|
| 早期 | オブジェクト生成時（new 時）に ID を確定させる | 常に ID が存在するため整合性が高い | ID 生成コストが高い場合のパフォーマンス問題 |
| 遅延 | DB 永続化時や後処理で ID を割り当てる | 早期生成のコストを避けられる | 識別子未確定の状態でオブジェクトが存在する期間がある |

### 一意な識別子で、エンティティの同一性を判定（`equals`/`hashCode` メソッド）

一意な識別子を用いてエンティティの同一性を判定する。  
エンティティには `equals` メソッドを実装し、識別子の値が等しければ同一のエンティティと判定する。

【図 5.6 一意な識別メソッドにおける一意な識別子の確認（Java）】

```java
public class User extends Entity {
    // ...
    @Override
    public boolean equals(Object anObject) {
        boolean equalObjects = false;
        if (anObject != null && this.getClass() == anObject.getClass()) {
            User typedObject = (User) anObject;
            equalObjects =
                this.tenantId().equals(typedObject.tenantId()) &&
                this.username().equals(typedObject.username());
        }
        return equalObjects;
    }

    @Override
    public int hashCode() {
        // ...
        return hashCode;
    }
}
```

この例では `tenantId` と `username` の組み合わせが一意な識別子となっている。  
`equals` の実装によって同一エンティティの比較が正しく行われる。

### 一意な識別子を変更しない「不変性」

注意点として、一意な識別子を設定したら、その識別子の値を変更しないようにする。  
もし、識別子が設定済みにもかかわらず、変更処理が呼び出された場合はエラーとなるようにしなければいけない。  
通常は、一意な識別子のセッターメソッドにアサーションを記載する。

### O/R マッパー用の ID 項目「代理識別子」

最近の実装では、DB にアクセスするライブラリとして Hibernate や Entity Framework といった O/R マッパーを使用することも多い。  
「代理識別子」とは、O/R マッパーが使用するシステム的な識別子のことである。

### 「レイヤスーパータイプ」で共通処理を実装

代理識別子（システム上のキー）は一意な識別子（ビジネス上のキー）ではないため、外部から見えないようにしておくことが望ましい。  
代理識別子を外部から見えなくする方法として、レイヤ共通の処理を記述する「レイヤスーパータイプ」を導入することができる。

【図 5.7 レイヤスーパータイプで代理識別子を管理（Java）】

```java
public abstract class Entity {
    private long id;
    // ...
    protected long id() {
        return this.id;
    }
    // id のセッターは protected で絞り込み、外部から見えないようにしている
}

public class User extends Entity {
    private TenantId tenantId;
    private String username;
    // ...
}
```

このように `Entity` の親クラス（レイヤスーパータイプ）で `id` という代理識別子を提供し、可視範囲を protected で絞り込み、外部から見えないようにしている。

## （6）エンティティの作成方法（コンストラクタ／ファクトリ）を検討

### コンストラクタ

エンティティは一意な必須項目を初期化するようにすること。  
そのため、コンストラクタの引数に必要なパラメータを全て設定し、引数が1つでも欠ける場合はコンストラクタを失敗させる。

## （7）エンティティのバリデーションを検討

### 属性／プロパのバリデーション

エンティティに注目してモデリングを行う場合、バリデーションについてチェックするタイミングは以下の3種類がある：

1. **要件条件（precondition）** ：呼び出される前に確定できるものは設定メソッドで確認する
2. **事後条件（postcondition）** ：呼び出された後に状態メソッドで状態を確認する
3. **不変条件（invariant）** ：データが常に最新化し続け壊れているかチェックする

【図 5.11 エンティティのバリデーション（整合性型）（Java）】

```java
public class Employee extends Entity {
    // ...
    public void changeEmployeeType(EmployeeType aType) {
        // precondition（事前条件）
        DomainRegistry.employeeRepository().assertEmployeeTypeChangeable(this);
        
        // 実際の変更処理
        this.setEmployeeType(aType);
    }

    protected void assertEmployeeTypeChangeable(EmployeeType aType) {
        // ...
    }
}
```

### オブジェクト全体のバリデーション

全体のバリデーションでチェックを管理したい場合は、バリデーションクラスを分離する方法もある。

【図 5.12 エンティティを整合するためのオブジェクト全体のバリデーション（Java）】

```java
public class EmployeeValidator extends ValueObjectValidator {
    // ...
    public void validate() {
        // ...
        // バリデーション検証
    }
}
```

Word Compositeパターン（言葉・型合成）を活用して、バリデーション・グループを組み合わせることができる。  
バリデーションを1つにしたい場合は、その検証をエキスパートなどのチェックにかけることが合理的。  
なお、バリデーション自体のチェックは、「テスト駆動開発」ベースでチェックを行うこともできる。

## 5.3 まとめ

以上、本章では DDD のエンティティについて紹介した。  
ここではドメインの要件を理解してエンティティを発見し、成長させていく方法を学んだ。  
また、それらをコーディングする方法について紹介した。  
次章では DDD の「値オブジェクト」を紹介する。

### 本章のポイント

- エンティティは「一意な識別子（ID）」によって同一性を識別するオブジェクト
- 値オブジェクトとの最大の違いは「同一性の判断基準」（ID か属性値か）
- モデリングの初期段階では識別子と最小限の属性・振る舞いに絞り込む
- 一意な識別子の生成パターンは4種類（ユーザー入力・アプリ生成・DB生成・他コンテキストから割り当て）
- UUID による早期生成が一般的だが、生成タイミング（早期／遅延）の選択も重要
- `equals`/`hashCode` は識別子ベースで実装する
- 代理識別子（O/R マッパー用のシステムキー）はレイヤスーパータイプで隠蔽する
- バリデーションは「属性単位・オブジェクト全体・複数オブジェクト」の3レベルで設計する
- コンストラクタ・ファクトリで生成時の完全性を保証する
