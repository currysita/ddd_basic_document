## 10.1　DDDにおける集約

### 外部から集約を操作できる「集約ルート」

外部から集約を操作する場合、代表するオブジェクトである集約ルート（ルートエンティティ）を通してのみ操作できます。集約ルート以外のオブジェクトを外部から直接参照したり、変更することはできません。

【図 10.1　DDDの「集約」のイメージ】

- 集約の外からは「集約ルート」のみアクセス可能
- 集約ルートを経由して内部の集約オブジェクト（エンティティ・値オブジェクト）を操作する
- 集約内部には「みんなで整合性を保とうね」という関係性のオブジェクトが含まれる
- 集約の外部にいるオブジェクトは集約ルートの識別子しか持てない

集約のオブジェクトが担う役割は次の2つです。

- **トランザクション整合性**（整合性・一貫性）
- **業務ルールの4つの特徴**（整合性、一意性、妥当性、制約）

DDDの著書『実践ドメイン駆動設計』（第10章）では、トランザクション整合性を保てなくなるようなオブジェクトをグループ化して「集約」とするよう提唱しています。また、トランザクション整合性の観点から整合性が必要なもの（結果整合性でよいもの）についても論じています。

## 10.2　実践ドメイン駆動設計のサンプルプロジェクトで考える集約

それでは、SaaSovation を例に実際についていきましょう。前章まで2つ集約を例に考えてきましたが、ここでは全プロジェクトのバックログシステム（課題管理）を例に説明します。

### 巨大な集約の問題点

例えば、何をモデリングしたとして、巨大な集約を作りがちです。もちろん、「プロジェクト」を管理すればよい、直感的にもユーザーのすべてのデータを1つのオブジェクトに収めたいと思いでしょうが、次の問題が生じます。

【図 10.2　巨大な集約の問題点】

- モデリングエラーが入り込む
- トランザクション競合が発生する
- メモリの使用量が高くなる

### 小さな集約の特徴

小さな集約の設計を検討することが必要です。これを実行することによって小さな集約を複数作成することになります。

【図 10.3　小さな集約（4つの集約に分割し一意な識別子でフォーカス）】

この図では大きな集約を4つに分割した例が示されています。バックログアイテムにフォーカスして、以下のように整理します。

| | 大きな集約 | 小さな集約 |
|---|---|---|
| 設計難易度 | 難 | 易 |
| トランザクション整合性 | 高 | 高（集約内は保証） |
| スケーラビリティ | 低 | 高 |
| 競合頻度 | 高 | 低 |

【表 10.1　大きな集約と小さな集約の比較】

こうすることで、トランザクション競合をなくすことができます。

## 10.3　集約の設計におけるルール

本章では集約の設計に関するルールを説明します。

### (1) 小さな集約を設計する

先ほどの通りのように、パフォーマンスや スケーラビリティの対応からも集約を小さくすることが重要です。集約の内部にあるオブジェクトのトランザクション整合性を保てるようにするためには、集約は1つのエンティティ（集約ルート）と、いくつかのオブジェクトのみにするのがよいでしょう。

### (2) 他のオブジェクトは、その一意な識別子を使用する

集約の外からはオブジェクトへの直接参照はできないようにしました。エンティティへの参照ではなく、一意な識別子（Value Object）で参照することが重要です。直接参照が入り込んでしまうと、1つのトランザクションで複数の集約を変更することになりがちです。

## 10.4　基本ルールに反する例外パターン

これまで、集約の設計に基本ルールを紹介してきましたが、あえて、わたしたちに見える違反するようなルールもあります。

### 例外1．UIの利便性を優先

複数の集約を1度に取得するようなユーザーインターフェイスが必要な場合（イベント）を想定したとき、集約は一緒に操作しないようにしなければなりません。

### 例外2．設計的な仕組みでの対処

ユースケースによっては、トランザクション整合性の確保よりも、扱いやすいコードを優先するケースがあります。複数のオブジェクトを一度のトランザクションで変更するのが適切な場合もあります。

### 例外3．グローバルトランザクションの強制

ある場合には、グローバルなトランザクション境界なく、整合性を保証できないケースがあります。場合によっては、その強制が必要なこともあります。

### 例外4．クエリのパフォーマンス改善

技術的な観点から集約の境界をまたいでクエリを実行することが適切な場合もあります。この場合、直接参照が有効なケースがあります。

## 10.5　集約の実装コード例

設計方針と例外パターンに続けて、実際に、集約のコードについて見ていきましょう。

### ルートエンティティの選択

まず、集約を作る場合、外部からの操作を担う1つのエンティティを集約ルートとして選択します。ここでは、ルートエンティティである「BacklogItem」クラスのコード（C# \*2 / Java \*1）を選択しています。

参照先：
- \[*1\] https://github.com/VaughnVernon/IDDD_Samples/blob/master/iddd_agilepm/src/main/java/com/saasovation/agilepm/domain/model/product/backlogitem/BacklogItem.java
- \[*2\] https://github.com/VaughnVernon/IDDD_Samples_NET/blob/master/iddd_agilepm/Domain.Model/Products/BacklogItems/BacklogItem.cs

**リスト 10.1　集約ルート「BacklogItem」のコード（C#）**

```csharp
// 集約ルート（ルートエンティティ）「バックログアイテム」
public class BacklogItem : EntityWithCompositeId
{
    // コンストラクタ
    public BacklogItem(...)
    {
            （省略）
    }

    // 自分の一意な識別子（値オブジェクト）
    public BacklogItemId BacklogItemId { get; private set; }

    // 他集約「テナント」の一意な識別子（値オブジェクト）
    public TenantId TenantId { get; private set; }

    // 他集約「プロダクト」の一意な識別子（値オブジェクト）
    public ProductId ProductId { get; private set; }

    // 自分のプロパティ「バックログアイテムタイプ」（値オブジェクト）
    public BacklogItemType { get; private set; }

    // 自分のプロパティ「バックログアイテムステータス2」（値オブジェクト）
    public BacklogItemStatus { get; private set; }

    // 自分のプロパティ「カテゴリ」
    public string Category { get; private set; }
}
```

集約ルートエンティティとして、このエンティティを集約するためのメンバーの一意な識別子（値オブジェクト）が参照します。

### 集約の部品を極力、値オブジェクトにする

集約内のオブジェクトは、最大限エンティティよりも値オブジェクトを使うことを考えます。エンティティはメンバーレベルのオブジェクトを使う場合、その値オブジェクト（Value Object）のメソッドを使うことを推奨します。

- オブジェクトテスト：ユニットテストが行いやすい
- 基本パラメータとして渡される値オブジェクトのメソッド
- 集約の部分インスタンス生成のためのオブジェクトのメソッド
- 自分自身のIDでインスタンスを生成するオブジェクトのメソッド

## 10.6　集約のコーディング時に役立つ法則

その後の集約コーディングを行うにあたって、集約のコーディングを行いやすくするためにいくつかのヒントを紹介します。

### デメテルの法則

デメテルの法則は、最低限のメンバーにアクセスし、オブジェクトのメンバーのオブジェクトへのアクセスについて、最低限のコードで行うことです。これにより、コードが疎結合になります。

次のことをしないのが、オブジェクトとメソッドのルールです：
- オブジェクトの X / Z のメソッドを直接呼び出す
- 基本パラメータとして渡されたオブジェクトの X のメソッドを直接呼び出す
- 集約の部分インスタンス生成のためのオブジェクトの X のメソッドを直接呼び出す
- 自分自身の ID でインスタンスを生成するオブジェクトの X のメソッドを直接呼び出す

これに反しているのは、オブジェクトテスト/Zのメソッドを直接呼び出す、ということです。

### 「命じる、たずねるな（Tell-Don't-Ask）」の指針

「命じる、たずねるな」の概念は、オブジェクトに対して結果を求めるよりも、行動を命じるべきという考えです。オブジェクトに意思決定の責任を与えることで、カプセル化を高め、依存を減らすことができます。

### 集約での依存性の注入を避ける

別の集約を識別子で参照する場合、参照先の集約はそのままでは読み込まれません。依存性の注入（DI）によって集約に別の集約を直接渡すことは避けます。

## 10.7　まとめ

本章の最後に集約について整理し、オブジェクトをまとめて管理する集約の設計についてまとめました。次の章では、DDDの「ファクトリ」について説明します。

### 集約の取得方法

集約を取得する際には2つの方法があります。

**方式1．集約の中からリポジトリを呼び出す**

「与えられたドメインモデルの1つ」という区分からの選択が可能な場合、リポジトリを経由して集約を取得します。

**方式2．集約の外（リポジトリまたはドメインサービス）から、事前に呼び出しておく**

外から集約を取得する際には、外のメインサービスを経由して取得します。

以上、集約の設計方針として、コーディングを行うと整合性に優れた集約を実装できます。
