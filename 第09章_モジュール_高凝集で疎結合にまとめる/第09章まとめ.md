# 第9章「モジュール」〜高凝集で疎結合にまとめる〜

## 章扉（page_0131）

前章でシステム間の連携を実現するところができる「ドメインイベント」について説明しました。本章は、DDD 各章を「モジュール」という観点で解説します。モジュールはプログラミング言語における「名前空間（パッケージ）」に相当します。DDD では概念的に関連するものをまとめて分類したり、高凝集で疎結合にシステムを管理する際に使います。また、ドメインエキスパートとのコミュニケーションにも活用できます。

---

## 9.1　モジュールとは

モジュールとは、オブジェクトをまとめて管理する仕組みのことを指し、プログラミング言語においては Java の `package`、C# の `namespace` として知られています。DDD ではソースファイルの物理的な整理だけでなく、**モデリング要素**として活用します。

### DDDでのモジュールとは

「モジュール」は、モデルをまとめて管理する仕組みです。Java では `package`、C# では `namespace` が使われます。本書では Java の `package`、C# の `namespace` をまとめて「モジュール」と呼びます。

DDDにおける**モジュールの特徴**：
- ユビキタス言語でモジュール名を命名する
- システム全体を**高凝集**で**疎結合**にまとめる

【図】図9.1　DDDの「モジュール」の概念図

図には `Java` と `C#` のパッケージ構成が示されており、モジュール間の依存関係として：
- モジュール間を疎結合にする
- モジュール内は高凝集にする

という二つの原則が図示されている。

---

## コラム：高いほど良い「凝集度」

凝集度（cohesion）とは、モジュールの責任がどれほど一貫しているかの尺度です。

関数型凝集（functional cohesion）と呼ばれる最も高い凝集度では、モジュール内のすべての要素が単一のタスクの実行に向けて協力しています。凝集度の高いシステムはシステムのメンテナンス性が向上するといわれています。

参考：https://ja.wikipedia.org/wiki/凝集度

---

## コラム：低いほど良い「結合度」

結合度（coupling）とは、コンポーネント間の依存の強さを示す尺度です。

データ結合（data coupling）と呼ばれる最も低い結合度では、コンポーネントは引数を介してのみ通信します。結合度の低いシステムはシステムのメンテナンス性が向上するといわれています。

参考：https://ja.wikipedia.org/wiki/結合度

---

## 9.2　モジュールの設計方法

モジュール設計の要点は、**必要な機能だけを公開し、他のモジュールが依存しやすい構造を作らないこと**です。オブジェクト指向プログラミングの原則（SOLID など）に従ってモジュールを設計するのが理想的です。また、ドメインのコンセプトが明確になったら積極的にモジュール分割を検討するとよいでしょう。

### 実践ドメイン駆動設計におけるモジュールの設計ルール

実践ドメイン駆動設計ではモジュールを設計する目安として、次のように提案しています。

**表9.1　実践ドメイン駆動設計におけるモジュールの設計ルール**

| 番号 | 設計ルール |
|------|-----------|
| 1 | モジュールはドメイン上の概念に沿って設計する |
| 2 | モジュールは関連するものをまとめ、無関係なものは分ける |
| 3 | モジュールを単純なコード入れ物にしない。責任を明確にする |
| 4 | クラスを構成するようにモジュールを設計する |
| 5 | 凝集力のある概念で 1 つのモジュールにまとめるのが適切である |
| 6 | 結合度を下げるためにモジュールの実装の詳細を隠蔽する |
| 7 | ドメインのユビキタス言語を使ってモジュールに名前を付ける |
| 8 | ドメインのコンセプトが変わったらモジュールもリファクタリングする |

これらのルールを守り、**凝集力があり結合が疎なモジュール**を設計することが DDD の基本となります。

## 9.3　モジュールの命名規約

モジュールの命名は、階層構造にて記述します。階層構造の先頭には、組織名を使うことが一般的です。SaaSovation の例を見ていきましょう。

### トップレベルのモジュール名

```
// Java の場合
com.saasovation

// C# の場合
SaaSovation
```

トップレベル名として組織名を設定しています。Java の場合、インターネットドメイン名（com や jp）が一般的なため、ここでは `com` という名前で始めています。このような命名規約を使うことで、JAR や DLL において他プロジェクトとの衝突を避けることができます。

### トップレベル下のコンテキスト名

トップレベルの組織名を決めた後は、その下の名前を検討します。ポイントは、idovation / collabovation / projectovation といったプロダクト名を入れていないことです。その代わり「境界づけられたコンテキスト」に沿った名前を追加しています。

```csharp
// リスト9.2　コンテキスト名をモジュール名に追加（C#）
SaaSOvation.IdentityAccess
SaaSOvation.Collaboration
SaaSOvation.AgilePM
```
> ※ 原書（リスト9.2〜9.7）のコードでは `SaaSOvation`（大文字O）と表記されている。トップレベル（リスト9.1）の `SaaSovation` との表記ゆれは原書どおり。

コンテキスト名が必ずしも正しいわけではありませんが、SaaSovation では、この名称がユビキタス言語を示す上でわかりやすいといった理由で命名しています。この場合、プロダクト名の変更の影響を受けないメリットがあります。

### 重要なサブモジュール名

さらに続けて、コンテキスト名の下位にあたる階層名について考えてみましょう。

```csharp
// リスト9.3　「Domain」をサブモジュール名に設定（C#）
SaaSOvation.IdentityAccess.Domain
SaaSOvation.Collaboration.Domain
SaaSOvation.AgilePM.Domain
```

ここでは「Domain」というモジュール名を追加しています。この配下にはドメインに関する情報を格納します（この Domain フォルダの直下にはクラスは存在せず、その下位のサブフォルダの中に格納します）。

```csharp
// リスト9.4　「Domain」内に「Model」モジュールを追加（C#）
SaaSOvation.IdentityAccess.Domain.Model
SaaSOvation.Collaboration.Domain.Model
SaaSOvation.AgilePM.Domain.Model
```

次に Domain 配下に「Model」を追加しています。この「Model」配下のサブモジュールには、値オブジェクト、エンティティ、イベントといったクラスやインターフェイスが定義されます。

---

### Domain.Model 配下の構成（page_0137）

【図】図9.2　モジュールの階層構造とドメインモデルに含まれるオブジェクト（C#）

図には SaaSOvation.AgilePM の階層構造が示されており、`Domain.Model` 配下に：
- エンティティ
- 値オブジェクト（Product）
- 集約
- ドメインイベント
- リポジトリ
- ファクトリ

が格納されることが示されている。

```csharp
// リスト9.5　モジュール名にて検討が必要な例（C#）
SaaSOvation.AgilePM.Domain.Service
```

このモジュール名は一見よさそうに見えますが、`Service` という名前は「ドメインモデル貧血症」という中身のないモデルがたくさん作られるきっかけになりがちです。7章にて紹介したように、ビジネスロジックをすべてサービスクラスに記述してしまわないように注意します。

```csharp
// リスト9.6　担当する機能のコンセプトを示すモジュール名を付与する（C#）
SaaSOvation.AgilePM.Domain.<コンセプト名>
```

そこで、ドメインモデル内でサービスという用語を使わず、ドメインのコンセプトを示す名前（Products、Tenants、Teams など）をつけます。このモジュール名はチームで検討し、チームの会話で利用します。このように、チームメンバーがドメインのコンテキスト内の概念を自然に理解できるよう、ユビキタス言語でモジュールを命名するとよいでしょう。

## 9.4　モジュール間の依存関係

命名方法に続けて、各モジュールにどのようなクラスが含まれているのかを見ていきましょう。また、それぞれのモジュールが持っている依存関係も見ていきましょう。

```
// リスト9.7　ドメインモデルに含まれるサブモジュール名（C#）
SaaSOvation.AgilePM.Domain.Model.Tenants
SaaSOvation.AgilePM.Domain.Model.Teams
SaaSOvation.AgilePM.Domain.Model.Products
```

アジャイルプロジェクト管理コンテキストのドメインモデル配下の場合、「テナント（Tenants）」「チーム（Teams）」「プロダクト（Products）」といったサブモジュールが存在しています。

### テナントモジュール

「テナント（Tenants）」モジュールに存在するクラスは、TeanatId（値オブジェクト）です。5 章にて紹介した一意な識別子で、個々のテナントを区別するキーとなります（この TeanatId は「アジャイルプロジェクト管理コンテキスト」だけの識別子ではなく、本来は「認証・アクセスコンテキスト」側で使われる識別子となります）。

【図】図9.3　Tenants モジュールの依存関係例（片方向の参照）

`Domain.Model.Tenants` モジュール（TenantID）は各モジュールから参照されている。

このテナントモジュール（TeanatId）は、アジャイルプロジェクト管理コンテキストの他のほとんどのモジュールから参照されます。しかし、逆方向の参照はしていないので、シンプルな依存関係といえます。

### チームモジュール

2 つ目の「チーム（Teams）」モジュールを見ていきます。このモジュールには、チームを管理するために使う 3 つの集約（ProductOwner、Team、TeamMember）が含まれています。

【図】図9.4　Teams モジュールの依存関係例（双方向の参照）

- Tenantsモジュールを参照し、Productsモジュールからも参照されている。

この図を見るとわかるように、Teams モジュールには Tenants モジュールへの参照と Products モジュールからの参照の両方が存在しています。このような双方向の依存関係については、できるかぎり避けることを推奨します。

### プロダクトモジュール

3 つ目の「プロダクト（Products）」モジュールの中には、多数クラス存在するため依存関係が複雑になっています（Sprint、BacklogItem、Release を含む）。

【図】図9.5　Products 配下のモジュール依存関係例（複雑な依存）

- Productsのモジュールは相互に参照し合っており、通常は望ましくない循環的な依存関係が示されている。
- 通常こうしてしまうと子側のモジュールのリファクタリングのために、親モジュールの修正が必要となる。

## 9.5　アプリケーションレイヤのモジュール分割

これまで、ドメインモデルのモジュールを整理してきました。その次の段階として、アプリケーションレイヤのモジュール分割を見てみましょう。DDD では、ドメインモデルを直接アプリケーションサービス/アプリケーションクライアントから操作するのではなく、アプリケーションサービスを介して操作します。このモジュールを適切に分割して管理することが重要です。

【図】図9.6　モジュールの階層構造とアプリケーション配下に含まれるオブジェクト（C#）

`Application配下`に、サブモジュール（Products、Notifications、Commands）が含まれ、他のサービスが格納されている。

```
// リスト9.8a　アプリケーションレイヤモジュールの例（C#）
SaaSOvation.AgilePM.Application.Sprints
SaaSOvation.AgilePM.Application.Teams
SaaSOvation.AgilePM.Application.Products
```

これは「アジャイルプロジェクト管理コンテキスト」では以下のようにモジュールを作っています。

```
// リスト9.8b　アプリケーションレイヤモジュールの例（別コンテキスト C#）
SaaSOvation.IdentityAccess.Application
```

各プロダクトの状況に合わせて、モジュールを細分化するかどうか検討するとよいでしょう。

## 境界づけられたコンテキストとモジュールの使い分け

なお、モジュールによる分割か、境界づけられたコンテキストによる分割かで悩むことがあるかもしれません。モジュールに関してはこれまで述べてきた通りクラス群でまとめる仕組みですが、境界づけられたコンテキストは、ユビキタス言語の境界線（ビジネス的な境界線）となります。

モデルが大きくなってきた場合、ドメインエキスパートの言葉に従い、どのような方法による分割が最適かを検討していくとよいでしょう。そのため最初の設計時は、無理して分割するのではなく、ひとまとめのままのほうが無難かもしれません。
