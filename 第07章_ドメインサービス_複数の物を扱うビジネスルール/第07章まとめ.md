# 第7章「ドメインサービス」〜複数の物を扱うビジネスルール〜

## 章の概要

前章までは個々のオブジェクト（エンティティ・値オブジェクト）を中心に扱ってきた。本章では、単一のオブジェクトでは自然に記述できないビジネスロジックを扱うための「ドメインサービス」について解説する。

---

## 7.1 DDD におけるサービスとは

DDD には「エンティティ」「値」「集約」「ドメインイベント」「サービス」という5つのパターンがある。そのなかのサービスには次の2種類がある。

- **ドメインサービス**：ドメインモデルの一部として、ドメイン固有のビジネスロジックを担うサービス
- **アプリケーションサービス**：ドメインモデルを利用するクライアント（第14章で詳述）

【図 7.1】 DDD におけるサービスの位置づけ（SaaSovation の C# サンプルソースより）

図は「ドメインサービス」と「アプリケーションサービス」がそれぞれ異なる層に位置することを示す。ドメインサービスはドメイン層の内部に存在し、アプリケーションサービスはアプリケーション層に存在する。

### ドメインサービスの特徴

ドメインサービスは、ドメインモデルに対して単一オブジェクトに割り当てると不自然になるビジネスロジックを担う。複数のドメインオブジェクトを使って操作・変換・計算を行う場合に使う。

### アプリケーションサービスの特徴

アプリケーションサービスはドメインサービスとは異なり、ドメインオブジェクトを使うユーザーとのやり取りを調整・制御するコンポーネントである。トランザクション管理や「実質的なシステム用ユーズケース」などの役割を担う。詳細は「境界づけられたコンテキストの統合（第13章）」と「アプリケーション（第14章）」で説明される。

---

## 7.2 DDD の「ドメインサービス」と一般的な「サービス」の違い

一般的に「サービス」という言葉は、ビジネスロジックを担うための処理や Web サービスという意味で使われることが多い。REST や SOAP などのインターフェイスを実装したコンポーネントを「サービス」と呼ぶ場合もある。

しかし、DDD におけるドメインサービスは、一般的なサービス層とは異なる。

### ドメインサービスで実装する内容

ドメインサービスで実装するのは、**複数のエンティティや値オブジェクトを組み合わせた変換・計算処理**である。たとえば、複数の BacklogItem の優先度を集計して合計値を計算するような処理がその典型例。

単一のエンティティに割り当てると不自然になるビジネスロジックは、ドメインサービスに移す。

### DDD 本におけるドメインサービスの解説

DDDの本では、Evans 氏の次の文章で説明されている：

> 「ビジネスロジックをエンティティや値オブジェクトに自然に割り当てることができないことがある。そのような場合、ドメインサービスを使うべきである。」

---

## 7.3 ドメインサービスの例

それぞれの役割を担うドメインサービスの具体的な利用方法を見てみよう。

### SaaSovation のコード例

ここでは SaaSovation のドメインサービスのソースコード（`BusinessPriorityCalculator`）を見てみよう。

> 参照：https://github.com/VaughnVernon/IDDD_Samples/blob/master/iddd_agilepm/src/main/java/com/saasovation/agilepm/domain/model/product/backlogitem/BusinessPriorityCalculator.java

**リスト 7.1 ドメインサービス `BusinessPriorityCalculator`（Java）**

```java
// ビジネス優先度計算用ドメインサービス
public class BusinessPriorityCalculator {

    // リポジトリ
    private BacklogItemRepository backlogItemRepository;

    // コンストラクタ
    public BusinessPriorityCalculator(
            BacklogItemRepository aBacklogItemRepository) {

        super();

        this.backlogItemRepository = aBacklogItemRepository;
    }

    // 「ビジネス優先度を合計する」メソッド
    public BusinessPriorityTotals businessPriorityTotals(
            TenantId aTenantId,
            ProductId aProductId) {

        int totalBenefit = 0;
        int totalPenalty = 0;
        int totalCost = 0;
        int totalRisk = 0;

        // リポジトリから BacklogItem のコレクションを取得
        Collection<BacklogItem> outstandingBacklogItems =
                this.backlogItemRepository()
                    .allOutstandingProductBacklogItems(aTenantId, aProductId);

        // 複数の Backlog アイテムの集計処理を実施
        for (BacklogItem backlogItem : outstandingBacklogItems) {
            if (backlogItem.hasBusinessPriority()) {
                BusinessPriorityRatings ratings =
                        backlogItem.businessPriority().ratings();

                totalBenefit += ratings.benefit();
                totalPenalty += ratings.penalty();
                totalCost    += ratings.cost();
                totalRisk    += ratings.risk();
            }
        }

        // 戻り値である BusinessPriorityTotals の値オブジェクトを設定
        BusinessPriorityTotals businessPriorityTotals =
                new BusinessPriorityTotals(
                        totalBenefit,
                        totalPenalty,
                        totalBenefit + totalPenalty,
                        totalCost,
                        totalRisk);

        return businessPriorityTotals;
    }

    private BacklogItemRepository backlogItemRepository() {
        return this.backlogItemRepository;
    }
}
```

このドメインサービスでは、リポジトリを利用してバックログアイテムを取得し計算を行い、ビジネス優先度合計を示す値オブジェクトを生成している。これまで解説してきた通り、複数のドメインオブジェクトを組み合わせ、変換を行う処理を実装していることがわかる。

## 7.4 ドメインサービスの失敗例

次に、ドメインサービスを導入するときの注意点について紹介する。

### ドメインサービスの多用・誤用

ドメインサービスの多用は、ユビキタス言語がビジネスロジックを記述しきれていない場合に起きる。100 もの トランザクションスクリプトをドメインサービスとして登録してしまうほど、ドメインサービスを使い過ぎてしまうことがある。

### 悪い実装を導く「ミニレイヤ」の提供

「ミニレイヤ」とは、ドメインサービスが提供してしまうことで、ドメインモデルを薄くしてしまう誤った設計パターンを指す。これはドメインサービスがビジネスロジックを集中させ過ぎる形になり、エンティティや値オブジェクトがただのデータコンテナと化す問題を引き起こす。

### アプリケーションサービス層での誤用

別の失敗例として、ドメインサービスをアプリケーションサービス実装に移してしまうことが挙げられる。

例えば、発注先として「`BusinessPriorityCalculator`」の計算を、アプリケーションサービスで直接実装してしまう場合がある。これはドメインのロジックがアプリケーション層へ流出することを意味し、ドメインモデルの貧血化につながる。

このようなアプリケーションサービスにはドメインモデルの知識が存在しないため、本来の「アプリケーション（第14章）」に対する自然な責任を持つことができなくなってしまう。

## 7.5 ドメインサービス導入時のポイント

ここでは、ドメインサービス導入のポイントについてまとめる。

- セパレートインターフェイスをどこに置くかについて説明する
- サービスの生成方法を検討する

### リファクタリングによるドメインサービスの導入

実践ドメイン駆動設計のサンプルを確認する。（`BusinessPriorityTotals`）ソフトウェア、プロダクトクラスからドメインサービスへリファクタリングするとどうなるかをみてみよう。

【図 7.2】 リファクタリング後、ドメインサービスを導入する例

- **リファクタリング前**：アプリケーション層のクラスがリポジトリや複数の集約から直接データを取得して計算する
- **リファクタリング後**：ドメインサービス（`BusinessPriorityCalculator`）がリポジトリと集約の橋渡しをして計算し、アプリケーション層は薄くなる

このように、複数のドメインオブジェクトを組み合わせて変換・計算する処理をドメインサービスに移すことで、ドメインモデルの凝集度が高まる。

### 単一責任原則（SRP 原則）

SaaSOvation のチームでは、プロダクトクラスの役割が肥大化していたため、`BusinessPriorityCalculator` という単一の責任を持つドメインサービスへ分割する判断をした。

単一責任原則（SRP）は、「クラスを変更する理由は1つだけであるべき」という原則であり、Robert C. Martin 氏の著書（アジャイルソフトウェア開発の奥義、2004年）に記されている。

> 参照：https://www.youtube.com/watch?v=o5D3D04NwH4（Robert C. Martin 氏の動画）

ドメインサービスは1つのことに集中することが重要である。2つ以上の責任を持つドメインサービスは、責任が増えるにつれて分離を検討すべきである。

## 7.6 ドメインサービスの設計方法

最後に、ドメインサービスの設計方法として「セパレートインターフェイス」について解説する。

### セパレートインターフェイスとは

セパレートインターフェイスとは、Martin Fowler 氏が著書「エンタープライズアプリケーションアーキテクチャのパターン（P of EAA）」で提唱したパターンである。

> 参照：https://martinfowler.com/local/phrasys/separateInterface.html

【図 7.3】 セパレートインターフェイス

- 左側：ドメイン層のパッケージ
- 右側：インフラストラクチャ層のパッケージ

ドメイン層に「`EncryptionService`」というインターフェイスを定義し、実装をインフラストラクチャ層に置くことができる。これにより、ドメイン層がインフラストラクチャ層に依存しない設計が可能になる。

### セパレートインターフェイスを導入する理由

通常の開発では、システム層の境界をまたぐ形でインターフェイスを実装することがある。たとえば、`Jar` ファイルや `.NET DLL` によってドメインモデルのパッケージ（ドメイン層）からアプリケーションサービスにアクセスし、処理をしている場合がある。

SaaSOvation の場合、複数サービスに共通する MOM（メッセージ指向ミドルウェア）などのアダプタによるドメインサービスへのアクセスを整理するために、セパレートインターフェイスを用いている。このような形にすることで、ドメインモデルがどのサービスからでも使えるようになる。

【図 7.4】 実装クラスとセパレートインターフェイスの使い分け（C#）

### 実装クラスとセパレートインターフェイスの使い分け

なお、必ずしもセパレートインターフェイスが必要なわけではない。実装クラスをドメイン層に置き、インターフェイスを持たずに直接利用するケースも多くある。


### サービスの生成方法（セパレートインターフェイスの場合）

セパレートインターフェイスを利用したサービスの生成方法は、インターフェイスをどう実装するかによる方法がある。

- コントラクタやメソッドのパラメータを使って、インスタンスを直接渡す
- オブジェクトを生成するファクトリを使う
- Spring や Unity Container などの IoC/DI コンテナを使って、サービスのインスタンスを生成する（推奨）

実践ドメイン駆動設計では最終手段として DI が挙げられ、ファクトリの適用も選択肢に入る（詳細は第11章 ファクトリで説明される）。

---

## 7.7 まとめ

本章では DDD のサービスについて解説し、DDD においてサービスには「ドメインサービス」と「アプリケーションサービス」の2種類が存在することを確認した。

- 複数のドメインオブジェクトを組み合わせた変換・計算処理は、ドメインサービスに実装する
- ドメインサービスはドメインモデル（エンティティ・値オブジェクト）の一部として位置づけられる
- ドメインサービスの使いすぎ・誤用は「ミニレイヤ」や「貧血ドメインモデル」につながるため注意が必要
- セパレートインターフェイスを用いることで、ドメイン層とインフラストラクチャ層の依存関係を整理できる
- サービスのインスタンス生成には DI コンテナやファクトリを活用する

以上が DDD のドメインサービスについての説明である。
