# 第14章「アプリケーション」〜ドメインモデルを利用するクライアント〜

前章ではシステム間のデータ連携方式を扱う「境界づけられたコンテキストの統合」について紹介した。最終章となる本章では、IDDD本の第14章「アプリケーション」について解説する。ドメインモデルを利用するアプリケーションについて理解しつつ、どのようにしてドメインモデルの情報をUI層に公開していくのかについて紹介する。

---

## 14.1 ドメインモデルを利用するアプリケーションとは

「アプリケーション」の広義の意味は「システム」全体と同じだが、本章では、ドメインモデルを使用するクライアントである「UI層」「アプリケーション層」について紹介する。

### UI層とは

利用者が操作する「UI層」のソフトウェア例：

- サーバー側で処理するWebシステム
- ブラウザ側で描画するWeb APIサービス
- WinFormsやSWTやSwingといったネイティブ「GUIクライアント」
- スマートフォンアプリケーション

通常、利用者に使い勝手のよいユーザーインターフェイスを提供するように努める。

### アプリケーション層（アプリケーションサービス）とは

「ドメインサービス」（7章）との比較で軽く紹介したように、「アプリケーションサービス」はドメインモデルの調整役を果たすクライアントである。

アプリケーションサービスの主なタスク：

- ユーザーの行動（ユースケース）に応じた調整役を実施
- リポジトリからアグリゲートを取り出し、ドメインのコマンドを実行する（またはそのほかのドメインサービスを利用する）
- データベーストランザクションの境界を管理し、最後にリポジトリにアグリゲートを保存する

【図14.1】ドメインモデルを使用するクライアント

- 左：ユーザーインターフェイス層（プレゼンテーション層）— Webアプリケーション、APIサービス、ネイティブクライアント（WinForm/Swing等）、スマートフォンアプリケーション／ユーザー操作に伴う表示・更新の描画
- 中央：アプリケーション層 — アプリケーションサービス／利用用途（ユースケース）に応じた調整役を実施
- 右：ドメイン層 — ドメインサービス、集約、値オブジェクト、エンティティ／ビジネス業務を表現

---

## 14.2 ドメインモデルを利用するアーキテクチャ

【図14.2】ドメイン層を取り囲むアーキテクチャ

機能のチーム別に応じてアーキテクチャを変化させても、基本的な考え方は変わらない。具体的には以下の2種類が紹介されている：

- **レイヤードアーキテクチャ**：UI層 → アプリケーション層 → ドメイン層 → インフラストラクチャ層
- **ヘキサゴナルアーキテクチャ（ポート＆アダプタ）**：ドメインを中心に外部とポートで接続

## 14.3 ドメイン層とUI層における情報の受け渡し方法

ドメイン層の情報をUIに渡す3つの方法を紹介する。

【図14.3】ドメイン層とUI層における情報の受け渡し方法

図の構成：
- 「データ照会時」：ドメイン → アプリケーション → UI
- 「データ更新時」：UI → アプリケーション → ドメイン

### （1）複数の集約情報をDTOに詰め替え

推奨方法として、複数の集約情報をDTO（Data Transfer Object）に詰め替えてUIに渡す。DTOはプレゼンテーション専用のデータ構造であり、ドメインモデルの内部構造をUIに漏らさない。

DTO用のアセンブラとして `CalendarWebDTODataTransformer` や `CalendarWebDTOAssembler` といったクラスを利用する例が示されている。

参考実装：https://github.com/VaughnVernon/IDDD_Samples/blob/master/iddd_agilepm/src/main/java/com/saasovation/agilepm/application/

### （2）複数の集約への参照として集約IDのみを送信

次の方法として、Vernor氏が提案するデータ転送オブジェクト（DTO）に集約IDのみを含める方法がある。UIがドメインオブジェクト（DTO）を直接操作できる場合に有効。

### （3）ユースケースに最適化したクエリによる取得

ユースケース専用のクエリ（CQRSのクエリサイド）を用いてUIが必要なデータを直接取得する方法。ドメインモデルを介さず、読み取り専用のデータモデルで応答する。

### 複数クライアントに対応するための切り替え

Web形式(JSON/XML)、ネイティブクライアントなど複数のクライアントに対応するため、Transformerクラスを切り替える。例：

- `CalendarWebDTOTransformer`
- `CalendarWebDTODataTransformer`
- `CalendarWebDTOTransformer`（別実装）

## 14.3 複数の境界づけられたコンテキストの統合

境界づけられたコンテキストをまたいだ場合、1つのアプリケーションサービスが複数のドメイン（境界づけられたコンテキスト）に対してコマンドを実行する場合がある。

【図14.4】複数コンテキストにまたがる場合の統合（2パターン）

- **1つ目の方法**：1つのUI操作に対し、複数の境界づけられたコンテキストに対して、アプリケーションサービスが直接コマンドを実行する（図14.4 左）
- **2つ目の方法**：境界づけられたコンテキスト側が結果をドメインイベントとして発行し、その他の境界づけられたコンテキストがそのイベントを購読する形で連携する（図14.4 右）。トランザクションの管理は各コンテキスト側に委ねられる（13章参照）。

## 14.4 UI層での描画に関わるモデル

最後に、アプリケーションサービスからUIへのデータ公開方法についてまとめる。

### 「ビュー」と「プレゼンテーションモデル」

UIを描画するフレームワークはさまざまだが、多くの場合：

- UIの表示（描画）を行う「**ビュー**」
- UIの状態とUI固有ロジックを持つ「**プレゼンテーションモデル**」

の2つから構成されることが多い。プレゼンテーションモデルの状態に基づいてビューが画面をレンダリングする。

### さまざまなモデルの役割

【図14.5】層別によって使用されるモデルの種類

| 層 | モデル |
|---|---|
| ユーザーインターフェイス層 | ビュー（描画を担当）、プレゼンテーションモデル（UI画面のデータや責任としての抽象化） |
| アプリケーション層 | DTO/DPO（データ構造の公開と受け渡しを担当） |
| ドメイン層 | ドメインモデル（ビジネスや業務のあり方を表現） |

- **ドメインモデル**はユビキタス言語を用いて、ビジネスや業務そのものを表現する
- **DTO/DPO**はクライアントに対して、ユースケースに応じたデータ構造の公開と受け渡しを担当する
- **プレゼンテーションモデル**はUI画面のデータや責任を抽象化する
- また、MVVMフレームワークのツールを使ってプレゼンテーションモデルを実現することも可能。IIDDDの例ではRESTFulで動くツールのプレゼンテーションモデルとなる例が示されている。

### UI層にドメインモデルを公開するか判断

DPO を利用する場合、UI層からドメインモデルを公開することもできる。

【表14.1】クライアントにドメインモデルを公開することのポイント

| 観点 | 公開あり | 公開なし |
|------|---------|---------|
| 分離 | ×（低い） | ○（高い） |
| 実装コスト | ○（小さい） | ×（大きい） |
| セキュリティ | ×（漏洩リスク） | ○（隠蔽） |
| テスト容易性 | △ | ○ |
| バリデーション | ×（ドメイン側で対応） | ○（DTOで別途対応） |
| モデルの変更影響 | ×（波及大） | ○（波及小） |

ドメインモデルを公開しないことでUIとモデルを分離できるが、パリデーション、セキュリティ、モデル変更の波及を考慮するとDTOの利用が推奨される。

## 14.5 アプリケーションサービスにおける実装

最後に、アプリケーションサービスの実装例を示す。

参考：https://github.com/VaughnVernon/IDDD_Samples/blob/master/iddd_identityaccess/src/main/java/com/saasovation/identityaccess/application/IdentityApplicationService.java

```java
// アプリケーションサービス
@Transactional
public class IdentityApplicationService {

    // ドメインサービス
    // ...

    public void provisionTenant(ProvisionTenantCommand aCommand) {
        return
            this.tenantProvisioningService().provisionTenant(
                aCommand.getTenantName(),
                aCommand.getTenantDescription(),
                new FullName(
                    aCommand.getAdministratorFirstName(),
                    aCommand.getAdministratorLastName()
                ),
                new EmailAddress(aCommand.getAdministratorEmailAddress()),
                new PostalAddress(
                    aCommand.getAdminstratorAddressStreetAddress(),
                    aCommand.getAdminstratorAddressCity(),
                    aCommand.getAdminstratorAddressStateProvince(),
                    aCommand.getAdminstratorAddressPostalCode(),
                    aCommand.getAdminstratorAddressCountryCode()
                ),
                new Telephone(aCommand.getAdminstratorPrimaryTelephone()),
                new Telephone(aCommand.getAdminstratorSecondaryTelephone())
            );
    }
    // ...
}
```

このメソッドの引数「`ProvisionTenantCommand`」は、Commandパターンを使用して入力コマンドを整理している。必要なコマンドDTOに応じた最小限のデータを持つ。

## 14.6 まとめ

本章ではアプリケーション（ドメインモデルを利用するクライアント）について紹介した。まとめると以下のとおり：

- UI層（プレゼンテーション層）はユーザーが操作するインターフェイスを提供し、ビューとプレゼンテーションモデルで構成されることが多い
- アプリケーション層（アプリケーションサービス）はUI層とドメイン層の調整役であり、ユースケースに応じた処理をまとめ、トランザクション境界を管理する
- ドメインモデルの情報をUI層に渡す方法として、DTOへの詰め替え・集約IDのみ送信・ユースケース特化クエリの3種類がある
- レイヤードアーキテクチャやヘキサゴナルアーキテクチャにより、ドメイン層を外部から守ることができる
- ドメインモデルをUI層に直接公開するかどうかは、分離・セキュリティ・実装コストのトレードオフを考慮して判断する
