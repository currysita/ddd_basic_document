# 用語集：第8章「ドメインイベント」

## ドメインイベント（Domain Event）

ドメインの専門家が気にかける、ドメイン内で起きた「出来事」を表すオブジェクト。「注文された」「在庫が切れた」「決済が完了した」などが例として挙げられる。Eric Evans の DDD 本では「ドメインの専門家が気にかけるドメイン内で起きたあること」と定義される。不変（イミュータブル）なオブジェクトとして設計される。

## ドメインイベントパブリッシャー（Domain Event Publisher）

ドメインイベントを発行（出版）する役割を担うクラス。集約がコマンドを処理した結果としてイベントを生成し、パブリッシャー経由で外部（サブスクライバー）に通知する。

## ドメインイベントサブスクライバー（Domain Event Subscriber）

パブリッシャーが発行したドメインイベントを受信（購読）して処理する役割を担うクラス。受信後の処理には同期処理（トランザクション実行）と非同期処理（キューへの格納）の 2 種類がある。

## イベントストーミング（Event Storming）

ドメインイベントを起点にシステムの振る舞いを設計・発見するワークショップ手法。付箋などを用いてドメインエキスパートと開発者が協力してイベントを洗い出す。

## occurredOn

ドメインイベントが発生した日時を保持するプロパティ。`DomainEvent` インターフェースが持つ共通プロパティの一つ。コンストラクタ内で `new Date()` によりセットされる。

## 最終的整合性（Eventual Consistency）

複数の集約やサービスをまたぐ操作において、即時の整合性ではなく「最終的には整合性が取れる」ことを許容する設計方針。ドメインイベントを非同期で処理することで実現される。

## 不変（Immutable）

一度生成されたオブジェクトの状態が変更されない設計。ドメインイベントクラスはすべてのフィールドを `final` として宣言し、setter を提供しないことで不変性を保つ。

## DomainEvent インターフェース

すべてのドメインイベントクラスが実装する基底インターフェース。`occurredOn()` メソッドを定義し、イベント発生日時を返す。

## 軽量なオブザーバー（Lightweight Observer）

パブリッシャーとサブスクライバが同じプロセス／スレッドで動作し、イベントを発行した同じトランザクション内で処理されるシンプルなイベント送受信方式。IDDD 本が推奨する基本的な実装パターン。

## DomainEventPublisher

イベントをサブスクライバに通知するシステム共通サービス。IDDD 本のサンプルコードで Java / C# 版が提供されている。パブリッシャーはサブスクライバの登録と、対応するイベント種別への配信を担う。

## DomainEventSubscriber

イベント受信時の処理を定義するシステム共通インターフェース。メールの送信やイベントストアへの保存など、処理別にインターフェイスを実装する。

## イベントストア（Event Store）

発生したドメインイベントをすべてストレージに記録するコンポーネント。メッセージング基盤のキュー、監査ログ、分析データ、イベントソーシングによる集約の再構築などに活用できる。ストレージには RDB や NoSQL を使用する。

## イベントソーシング（Event Sourcing）

集約の現在の状態を直接保存する代わりに、発生したすべてのドメインイベントを記録し、それを再生することで集約の状態を再構築するパターン。イベントストアと組み合わせて使用される。

## 重複排除（Idempotency）

受信側が複数回同じイベントを受け取った場合に、処理が重複しないようにする仕組み。イベントに一意な識別子を付与することで実現する。

## 結果整合性（Eventual Consistency）

複数の境界づけられたコンテキストをまたぐ操作において、即時の整合性ではなく「最終的には整合性が取れる」ことを許容する設計方針。ドメインイベントを他コンテキストへ非同期送信することで実現される。

## ストアドイベント（Stored Domain Event）

ドメインイベントをイベントストアに永続化するためのデータクラス。`EventId` やイベントクラス名などのメタ情報とイベントのシリアライズデータを保持する。

## 自立型サービス（Autonomous Service）

他のサービスに依存せず独立して動作するサービス。ドメインイベントを活用して疎結合に連携することで実現される。障害時の影響範囲を最小化し、個別のスケーリングが可能になる。

## RabbitMQ

AMQP（Advanced Message Queuing Protocol）を使用したオープンソースのメッセージングミドルウェア。P2P のキューイングモデルでメッセージング通信を実現する。IDDD 本でイベント送受信の実装例として紹介されている。

## AMQP（Advanced Message Queuing Protocol）

メッセージングキューのための標準プロトコル。RabbitMQ が採用しており、P2P のメッセージング通信を実現する。

## べき等性（Idempotency）

同じ操作を何度繰り返しても結果が変わらない性質。メッセージング基盤では NW 障害などによりメッセージが重複配送される可能性があるため、受信側がべき等性を保って重複処理を防ぐ必要がある。
