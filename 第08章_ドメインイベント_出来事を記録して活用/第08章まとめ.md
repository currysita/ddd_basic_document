# 第8章「ドメインイベント」〜出来事を記録して活用〜

## 章の概要

前章で紹介した複数のオブジェクトを扱うビジネスルールである「ドメインサービス」について説明しました。本章では、対比的なコンセプトである「ドメインイベント」について解説します。ドメインイベントを利用することで、関連するシステムどうしの同期的な結合を排除するとともに、業務的な出来事を記録して活用することができます。

---

## 8.1 ドメインインイベントとは

「ドメインイベント」は (2) 複数の集約のデータをモデリングする方法として 2 つのプロパティを設定します。本節では、ドメインイベントをどのようにモデリングするかについて説明します。

ドメインイベントとは、システム内で発生した「出来事」を表すオブジェクトです。たとえば、「本を購入した」「在庫が切れた」「決済が完了した」といった出来事が該当します。以前のバージョンは「発生した出来事（ドメインイベント）」として `DDD の集約` と考えられていましたが、現在のドメインイベントスポット 2 つのプロパティを含む「する事に」といったチームが注目しています。

上述の通り、事業者のドメインからすると、ドメインイベントは DDD 本（Eric Evans DDD）274 ページで以下のように定義されています。

> ドメインイベントとは、ドメインの専門家が気にかけるドメイン内で起きたあることです。

---

## ドメインイベントのモデリング方法

「コンテキスト」は (2) 2 つのオブジェクトの振る舞いをモデリングする方法として 2 つのプロパティを設定します。(1) イベント発生でドメインイベントを直接「コマンド」から「集約」へと送信します。たとえば、以下のような流れとなります。

**【図 8.1 ドメインイベントの発見方法】**

- コマンド（例：「注文する」）→ 集約（例：注文集約）→ ドメインイベント（例：注文されました）という流れ
- 集約（例：在庫集約）→ ドメインイベント → 別の集約（例：出荷集約）という流れ

また、イベント「コマンド」からどう集約を取りまとめるかについては、定義のアウトプットとして集約をシステムに「最終的な整合性」が取れるように実装します。

---

## 8.2 ドメインイベントがもたらすメリット

ドメインイベントには、結果的なシステム全体の整合性を保ちながら、サービス間の疎結合を実現する様々なメリットがあります。

### バッチ処理に関する考え方の変化

イベントを有効活用することで考え方が変わります。従来は「毎朝のシステムが集計処理を行う」と「1 日 1 回のバッチで集計が動く」という設計をしていました。しかし、ドメインイベントを活用することで、毎回ドメインイベントを記録・適用するだけで集計処理のコストを下げることができます。

### 集約の制約からの開放

「集約は 100 行以内」というトランザクションの整合性を高めると「1 つのトランザクションで複数の集約をまたぐ」という問題が発生します。しかしレシーバーとセンダーの組み合わせで分割したイベントを「バックグラウンドで」実行できるようにすれば、スケーラブルなベースサービスへの移行もシームレスに行えます。

---

## 8.3 ドメインインインイベントの導入

イベントを導入する場合は、ドメインイベントスポットという「イベントストーミング」と呼ばれる伝統的なイベント技法が利用できます（ただしすべての起点となるわけではなく、代替的な伝統的なイベントと比較して最も効率的な場合に選択します）。

---

## 8.3 イベントを用いた処理の例

SaaS/Dreams というプロジェクト管理コマンドツールに「バックログアイテムをコミットする」コマンドを実施した場合、以下のステップでイベントが処理されます。

1. 集約のコマンド「スプリントバックアップアイテムをコミットする」が実行される
2. 集約のコマンドで、コミットなどスプリントへの処理が行われる
3. 「ドメインインインイベント（パブリッシャー）」がイベントを発射（出版）する
4. 「サブスクライバー」がイベントを受信（管理）する
5. 「サブスクライバー」でメールを送信する

**【図 8.2 イベントの処理（出版）と受信（購読）の流れ】**

- DDD での出版の処理は「パブリッシャー」と行い、イベントの受取は「サブスクライバー」と行う処理を指す
- サブスクライバーが処理を受けてからは、大きく 2 つのフローがある：
  1. 同期処理：イベントを受取り、トランザクション実行
  2. 非同期処理：イベントをキューに格納し、後で実行

---

## 8.4 「ドメインインイベント」クラスのモデリング

続いて、イベントを格納する「ドメインインイベント」クラスの設計について説明します。

イベントのデータを保管するため、ドメインインイベントのクラスには通常バックグラウンドクラスとして 1 つのマップから情報を取得します。例えば、クラスの実装に「バックリンクモデル」として定義した場合は、必ずイベントの受取を開始した後にドメインインイベントとして、その情報を受取したイベントのデータを基底クラスに記述します。

---

## 8.4 ドメインインイベントの共通項目

DDD の本（DDDD 本）の通りに、ドメインインイベントのクラスには 2 つのプロパティを設定します。

1. `イベント発生日時`（`occurredOn`）：イベントが発生した日時を保持する
2. `集約のルートID`（例：`BacklogItemId`）：イベントが発生した集約の識別子を保持する

これら 2 つのプロパティを基底クラス（`DomainEvent` インターフェースや `DomainEventPublisher` クラス）として実装しておくことで、すべてのドメインイベントクラスに共通のプロパティを持たせることができます。

---

## 8.4 ドメインインインイベントの共通項目

DDD 本（Eric Evans）184 ページより、クラスに以下を実装します。

```java
// DomainEvent インターフェース
public interface DomainEvent {
    public Date occurredOn();
}
```

```java
// BacklogItemCommitted ドメインインイベントクラス（例）
public final class BacklogItemCommitted implements DomainEvent {
    private Date occurredOn;
    private BacklogItemId backlogItemId;
    private SprintId committedToSprintId;
    private TenantId tenantId;

    public BacklogItemCommitted(
        TenantId aTenantId,
        BacklogItemId aBacklogItemId,
        SprintId aCommittedSprintId) {

        super();
        this.occurredOn = new Date();
        this.backlogItemId = aBacklogItemId;
        this.committedToSprintId = aCommittedSprintId;
        this.tenantId = aTenantId;
    }

    // getter...
}
```

**【図 8.3 イベントクラスの実装（BackingItemCommitted クラス）】**

`BackingItemCommitted` は `BacklogItemId`・`SprintId`・`TenantId` の 3 つのプロパティを持ち、イベント発生時刻は `occurredOn` として コンストラクタ内で `new Date()` によりセットされます。

---

## 不変なドメインインインイベントクラス

通常、ドメインインインイベントクラスは「不変（イミュータブル）」として設計されます。一度作成されたイベントオブジェクトは変更されないため、すべてのフィールドは `final` として宣言し、setter は提供しません。

---

## イベントの一意な識別子

ドメインイベントクラスには「一意な識別子」は存在していませんが、イベントを使って外部のコンテキストと連携したり、メッセージング基盤を使ったりする場合には、一意な識別子が必要となる場合があります（一般的に、イベントの受信側は複数回イベントが送信されてくることを考慮し、「一意な識別子」を用いて重複排除を行います）。

## 8.5 基本的なイベント送受信方法

続けて、基本的なイベントの連携方式を見ていきましょう。イベントの連携はパブリッシャー（出版側）と、サブスクライバ（購読側）による組み合わせで構成されます。

### 軽量なオブザーバーを用いたイベントの送受信

IDDD 本では、シンプルにイベントを送受信するには「軽量なオブザーバー」がよいとしています。オブザーバーとは、状態の変化を他のオブジェクトに通知するデザインパターンの 1 つです。

**【図 8.4 軽量なオブザーバーを用いた基本的なイベントの流れ（IDDD 本 図 8-2 を和訳）】**

軽量なオブザーバーを用いて、イベントを処理する流れは次の通りです。

1. **【UI や Web サービス】**：アプリケーションサービスのコマンドを呼び出す
2. **【アプリケーションサービス】**：イベント受信時に実行するサブスクライバをパブリッシャーに登録する
3. **【アプリケーションサービス】**：集約のコマンドを呼び出す
4. **【集約】**：ドメインイベントを生成する
5. **【集約】**：パブリッシャーを用いて、イベントを発信する
6. **【パブリッシャー】**：サブスクライバに対応するイベントの種類を確認して、イベントを配信（実行）する
7. **【サブスクライバ】**：自分が対応するイベントが配信されてきた場合は、イベントに適した処理を実行する

軽量なオブザーバーの特徴は、パブリッシャーとサブスクライバが同じプロセス／スレッドで動作し、イベントを発行した同じトランザクションにて処理される点です。

### イベントを発信するパブリッシャー

パブリッシャーは、サブスクライバにイベント通知を送る機能を提供します。IDDD 本のサンプルコードでは `DomainEventPublisher`（Java / C#）というシステム共通のサービスとして提供されています。

### イベントを受信するサブスクライバ

サブスクライバは、イベント受信時に行う機能を提供します。例えば、メールを送ったり、イベントストア（後述）に情報を保存したりします。IDDD 本のサンプルでは `DomainEventSubscriber`（Java / C#）というシステム共通のインターフェイスを用いて、処理別にインターフェイスを実装します。

## 8.6 他システム（他の境界づけられたコンテキスト）へのイベント送信

続けて、他の境界づけられたコンテキストへイベントを送信する方法について説明します。リモートの他のコンテキストへイベントを送信するには大きく 3 種類のイベントストア方式があります。

**【表 8.1 メッセージモデルとメッセージング基盤管理方法を選ぶ方法】**

| No | 概要 | ドメインイベント格納 | メッセージ処理 |
|----|------|---------------------|----------------|
| 1  | 非同期メッセージ基盤 | ドメインイベントをメッセージング基盤（RabbitMQ など）に送る | メッセージング基盤がメッセージを配送する |
| 2  | 分散トランザクション処理 | ドメインイベントをローカルのイベントストアに保存する | メッセージング基盤がメッセージを配送する |
| 3  | イベントストア | ドメインイベントをイベントストアに保存する | REST などで他のサービスが取得する |

ここでは、IDDD 本として特に特徴的な 3 番目のイベントストア方式を中心に紹介します。

また、他のコンテキストへイベントを送信するときには結果整合性が取れるようにする必要があります。REST で送信する方法、メッセージングミドルウェア（RabbitMQ）で送信する方法などがあります。

### 結果整合性と遅延の許容範囲

なお、他のコンテキストへのイベント送信をするときには結果整合性が取れるようにする必要があります。「アクティブなコンテキスト側」が他のコンテキストへ送信する際には「結果整合性」が必要となります。この場合は、サブスクライバが処理したイベントに基づいてデータを最終的に整合させます。

## 8.7 イベントストア

イベントストアとは、発生したドメインイベントをすべてストレージに記録するためのコンポーネントです。

**【図 8.5 イベントストアと REST を用いた他システムへのイベント連携】**

イベントストアには、次のようなメリットが備わっています。

- メッセージング基盤を通してドメインイベントを発行するキューとして使用できる
- モデルで発生したイベントをすべて履歴として記録できる（調査や監査に利用）
- イベントストアのデータを使って、分析や予測ができる（後から集計が可能）
- イベントを用いてリポジトリの構築を行い、集約の再構築ができる（イベントソーシングにてデータパッチやデバッグも可能）

### イベントストアの実装イメージ

イベントストアの実装例としては、「イベントストア（EventStore）」インターフェイスを利用してイベントの履歴情報をストレージへ格納するものが考えられます。イベントストアのストレージは、MySQL のような RDB や、LevelDB のような NoSQL などから最適な製品を選択します。どのストレージを選択するにしても、ドメインモデルの情報とイベント履歴の整合性が保たれるように注意します。

**【図 8.6 イベントストアの実装イメージ】**

ストレージには「ストアドイベント（StoredDomainEvent）」というデータクラスを格納します。ストアドイベントには以下のプロパティがあります。

- `EventId`（テナントアサプライズ）
- `DomainEventPublisher`（クラス名）

ナチュラルなイベントストアをシンプルに実装することで、イベントをイベントストアに格納し、ドメインイベントクラスからインターフェイスを呼び出してイベントを格納することができます。

---

## 8.8 自立型のサービスのメリット

他の境界づけられたコンテキストと連携する場合、ドメインイベントを使うと「自立型のサービス」にできるとメリットとして挙げられます。自立型のサービスは、他のサービスに依存しないシステムとして動くことを指します。他のサービスに依存しないため、それぞれのシステムが独立して動作し、スケーリングや障害時のシステム影響を最小化できます。

### REST による通知 API

イベント通知用の REST 送信では、結果タグ（レスポンスタグ）と連絡タグ（データタイプ）の組み合わせで記述できるようにします。そうすると、イベント数が膨大になった場合でも、チャンクを分割して取得できるようにします。サーバー側に処理をすることでキャッシュを使うことができます。

---

## 8.9 メッセージングミドルウェアによる連携方法

前章では REST を使ったイベントと連携について紹介しましたが、REST API を実装する以外にもメッセージングミドルウェアとイベントを連携させる方法があります。メッセージングミドルウェアとしては ActiveMQ・RabbitMQ・Akka・Microsoft Azure などのプラットフォームがあります。ここでは、IDDD 本でも紹介している RabbitMQ を用いて説明します。

### RabbitMQ を用いたイベントの送受信

RabbitMQ は AMQP（Advanced Message Queuing Protocol）というメッセージングプロトコルを使用した、オープンソースなミドルウェアです。AMQP は、キューイングモデルで P2P のメッセージング通信を実現するプロトコルです。

RabbitMQ によるイベント送受信の流れは次の通りです。

1. イベントストアからイベントを一定時間ごとに取り出してメッセージ送信する
2. 他サービスが受取ったリクエストをキューに入れる、サーバー内でリクエストを処理する
3. メッセージのキューを受け取った後、コンシューマーにルーティングする
4. 配送されたメッセージの処理が完了したら、コンシューマーが RabbitMQ に「ACK（アクノリッジメント）」を返す

### メッセージング基盤を使うときの注意点

メッセージング基盤を利用する場合には「べき等性」を保たせる必要があります。これは NW の障害や端末側での障害時に、同一のメッセージが複数回配送される可能性があるためです。受信側で重複メッセージを識別して処理が重複しないように対応が必要になります。また、配信順序が保証されないことを前提にして設計することも重要です。

## 8.10 まとめ

本章ではドメインイベントについて、メリット・モデリング方法・送受信方法・他コンテキストへの送信方法などの観点から学びました。DDD では「ドメインイベント」を活用することで、ドメイン内で起きた出来事をドメインモデルとして表現し、関連サービスどうしの疎結合を実現できます。イベントを活用することでシステム全体の結合度を下げ、スケーラブルで保守性の高いアーキテクチャを実現できます。
