# 第8章「ドメインイベント」〜出来事を記録して活用〜

## 章の概要

前章で紹介した複数のオブジェクトを扱うビジネスルールである「ドメインサービス」と対比的なコンセプトである「ドメインイベント」について解説します。ドメインイベントを利用することで、関連するシステムどうしの同期的な結合を排除するとともに、業務的な出来事を記録して活用することができます。

---

## 8.1 ドメインイベントとは

「ドメインイベント」とは、システム内で発生した「出来事」を表すオブジェクトです。たとえば「本を購入した」「在庫が切れた」「決済が完了した」といった出来事が該当します。

DDD 本（Eric Evans）では次のように定義されています。

> ドメインイベントとは、ドメインの専門家が気にかけるドメイン内で起きたあることです。

### ドメインイベントのモデリング方法

ドメインイベントを発見するには、コマンド（例：「注文する」）→ 集約（例：注文集約）→ ドメインイベント（例：「注文されました」）という流れで考えます。

**【図 8.1 ドメインイベントの発見方法】**

- コマンド → 集約 → ドメインイベント（例：注文集約が「注文されました」イベントを発行）
- 集約 → ドメインイベント → 別の集約（例：在庫集約が「在庫切れ」イベントを発行 → 出荷集約が受信）

---

## 8.2 ドメインイベントがもたらすメリット

ドメインイベントには、システム全体の結合度を下げながら整合性を保つ様々なメリットがあります。

### バッチ処理に関する考え方の変化

従来は「毎朝のバッチで集計処理を行う」という設計が一般的でした。しかしドメインイベントを活用することで、出来事が発生するたびにイベントを記録・適用するだけでよくなり、集計処理のコストを大幅に下げることができます。

### 集約の制約からの開放

「1つのトランザクションで複数の集約をまたぐ」という制約問題が発生する場面でも、ドメインイベントを使ってバックグラウンドで非同期に処理を分割することで、スケーラブルな設計が可能になります。

---

## 8.3 ドメインイベントの導入

イベントを導入する場合は、**イベントストーミング**と呼ばれる手法が利用できます。イベントストーミングとは、ドメインエキスパートと開発者が付箋を使って協力しながらドメインイベントを洗い出すワークショップ手法です。

また、イベントを用いた処理の具体例として、SaaSovation のプロジェクト管理ツールで「バックログアイテムをコミットする」コマンドを実施した場合、以下のステップでイベントが処理されます。

1. 集約のコマンド「スプリントにバックログアイテムをコミットする」が実行される
2. 集約がコマンドを処理する（スプリントへの紐付けなど）
3. 「ドメインイベント（パブリッシャー）」がイベントを発行（出版）する
4. 「サブスクライバ」がイベントを受信（購読）する
5. 「サブスクライバ」がメール送信などの処理を実行する

**【図 8.2 イベントの処理（出版）と受信（購読）の流れ】**

サブスクライバが処理を受け取った後は、大きく3つのフローがあります：

1. **同期処理**：イベントを受け取り、同じトランザクション内で実行する
2. **非同期処理**：イベントをキューに格納し、後で実行する（結果整合性）
3. **分散処理**：別システム（境界づけられたコンテキスト）へメッセージとして送信する

---

## 8.4 「ドメインイベント」クラスのモデリング

続いて、イベントを格納する「ドメインイベント」クラスの設計について説明します。

### ドメインイベントの共通項目

実践ドメイン駆動設計の通り、ドメインイベントのクラスには最低限以下の2つのプロパティを設定します。

1. `occurredOn`（イベント発生日時）：イベントが発生した日時を保持する
2. 集約のルートID（例：`BacklogItemId`）：イベントが発生した集約の識別子を保持する

これらを基底クラス（`DomainEvent` インターフェース）として実装しておくことで、すべてのドメインイベントクラスに共通のプロパティを持たせることができます。

**【図 8.3 イベントクラスの実装（BacklogItemCommitted クラス）】**

```java
// DomainEvent インターフェース
public interface DomainEvent {
    public Date occurredOn();
}
```

```java
// BacklogItemCommitted ドメインイベントクラス（例）
public final class BacklogItemCommitted implements DomainEvent {
    private Date occurredOn;
    private BacklogItemId backlogItemId;
    private SprintId committedToSprintId;
    private TenantId tenantId;

    public BacklogItemCommitted(
        TenantId aTenantId,
        BacklogItemId aBacklogItemId,
        SprintId aCommittedSprintId) {

        super();
        this.occurredOn = new Date();
        this.backlogItemId = aBacklogItemId;
        this.committedToSprintId = aCommittedSprintId;
        this.tenantId = aTenantId;
    }

    // getter...
}
```

`BacklogItemCommitted` は `BacklogItemId`・`SprintId`・`TenantId` の3つのプロパティを持ち、イベント発生時刻はコンストラクタ内で `new Date()` によりセットされます。

### 不変なドメインイベントクラス

ドメインイベントクラスは「不変（イミュータブル）」として設計します。一度作成されたイベントオブジェクトは変更されないため、すべてのフィールドを `final` として宣言し、setter は提供しません。

### イベントの一意な識別子

ドメインイベントクラスには基本的に「一意な識別子」は不要です。ただし、イベントを外部コンテキストと連携したり、メッセージング基盤を使ったりする場合には一意な識別子が必要となります。受信側が複数回イベントを受け取ることを考慮し、「一意な識別子」を用いて重複排除（べき等性の確保）を行います。

## 8.5 基本的なイベント送受信方法

続けて、基本的なイベントの連携方式を見ていきましょう。イベントの連携はパブリッシャー（出版側）とサブスクライバ（購読側）の組み合わせで構成されます。

### 軽量なオブザーバーを用いたイベントの送受信

実践ドメイン駆動設計では、シンプルにイベントを送受信するには「軽量なオブザーバー」がよいとしています。オブザーバーとは、状態の変化を他のオブジェクトに通知するデザインパターンの1つです。

**【図 8.4 軽量なオブザーバーを用いた基本的なイベントの流れ（実践ドメイン駆動設計 図 8-2 を和訳）】**

軽量なオブザーバーを用いてイベントを処理する流れは次の通りです。

1. **【UI や Web サービス】**：アプリケーションサービスのコマンドを呼び出す
2. **【アプリケーションサービス】**：イベント受信時に実行するサブスクライバをパブリッシャーに登録する
3. **【アプリケーションサービス】**：集約のコマンドを呼び出す
4. **【集約】**：ドメインイベントを生成する
5. **【集約】**：パブリッシャーを用いて、イベントを発信する
6. **【パブリッシャー】**：サブスクライバに対応するイベントの種類を確認して、イベントを配信（実行）する
7. **【サブスクライバ】**：自分が対応するイベントが配信されてきた場合は、イベントに適した処理を実行する

軽量なオブザーバーの特徴は、パブリッシャーとサブスクライバが同じプロセス／スレッドで動作し、イベントを発行した同じトランザクションにて処理される点です。

### イベントを発信するパブリッシャー

パブリッシャーは、サブスクライバにイベント通知を送る機能を提供します。実践ドメイン駆動設計のサンプルコードでは `DomainEventPublisher`（Java / C#）というシステム共通のサービスとして提供されています。

### イベントを受信するサブスクライバ

サブスクライバは、イベント受信時に行う機能を提供します。例えば、メールを送ったり、イベントストア（後述）に情報を保存したりします。実践ドメイン駆動設計のサンプルでは `DomainEventSubscriber`（Java / C#）というシステム共通のインターフェイスを用いて、処理別にインターフェイスを実装します。

## 8.6 他システム（他の境界づけられたコンテキスト）へのイベント送信

続けて、他の境界づけられたコンテキストへイベントを送信する方法について説明します。リモートの他コンテキストへイベントを送信するには、大きく3種類の方式があります。

**【表 8.1 メッセージモデルとメッセージング基盤管理方法】**

| No | 概要 | ドメインイベント格納 | メッセージ処理 |
|----|------|---------------------|----------------|
| 1  | 非同期メッセージ基盤 | ドメインイベントをメッセージング基盤（RabbitMQ など）に直接送る | メッセージング基盤がメッセージを配送する |
| 2  | 分散トランザクション処理 | ドメインイベントをローカルのイベントストアに保存する | メッセージング基盤がメッセージを配送する |
| 3  | イベントストア＋REST | ドメインイベントをイベントストアに保存する | REST などで他のサービスが取得する |

実践ドメイン駆動設計では特に3番目のイベントストア方式を中心に紹介しています。

### 結果整合性と遅延の許容範囲

他のコンテキストへのイベント送信をするときには「結果整合性」が取れるようにする必要があります。サブスクライバが処理したイベントに基づいてデータを最終的に整合させる設計となります。

---

## 8.7 イベントストアとは

イベントストアとは、発生したドメインイベントをすべてストレージに記録するためのコンポーネントです。

**【図 8.5 イベントストアと REST を用いた他システムへのイベント連携】**

イベントストアには、次のようなメリットが備わっています。

- メッセージング基盤を通してドメインイベントを発行するキューとして使用できる
- モデルで発生したイベントをすべて履歴として記録できる（調査や監査に利用）
- イベントストアのデータを使って、分析や予測ができる（後から集計が可能）
- イベントを用いてリポジトリの構築を行い、集約の再構築ができる（イベントソーシングにてデータパッチやデバッグも可能）

### イベントストアの実装イメージ

**【図 8.6 イベントストアの実装イメージ】**

イベントストアの実装例としては、`EventStore` インターフェイスを利用してイベントの履歴情報をストレージへ格納します。ストレージには `StoredDomainEvent`（ストアドドメインイベント）というデータクラスを格納します。

ストレージは MySQL のような RDB や LevelDB のような NoSQL などから最適な製品を選択します。どのストレージを選択するにしても、ドメインモデルの情報とイベント履歴の整合性が保たれるように注意します。

---

## 8.8 自立型のサービスのメリット

他の境界づけられたコンテキストと連携する場合、ドメインイベントを使うと「自立型のサービス」にできます。自立型のサービスとは、他のサービスに依存しないシステムとして動くことを指します。他のサービスに依存しないため、それぞれのシステムが独立して動作し、スケーリングや障害時のシステム影響を最小化できます。

### REST による通知 API

イベント通知用の REST 送信では、イベント数が膨大になった場合でも、チャンクを分割して取得できるように設計します。サーバー側でキャッシュを使うことも有効です。

---

## 8.9 メッセージングミドルウェアによる連携方法

REST API を実装する以外にも、メッセージングミドルウェアとイベントを連携させる方法があります。メッセージングミドルウェアとしては ActiveMQ・RabbitMQ・Akka・Microsoft Azure などのプラットフォームがあります。実践ドメイン駆動設計では RabbitMQ を用いて説明しています。

### RabbitMQ を用いたイベントの送受信

RabbitMQ は AMQP（Advanced Message Queuing Protocol）というメッセージングプロトコルを使用した、オープンソースなミドルウェアです。

RabbitMQ によるイベント送受信の流れは次の通りです。

1. イベントストアからイベントを一定時間ごとに取り出してメッセージ送信する
2. メッセージのキューを受け取った後、コンシューマーにルーティングする
3. 配送されたメッセージの処理が完了したら、コンシューマーが RabbitMQ に「ACK（アクノリッジメント）」を返す

### メッセージング基盤を使うときの注意点

メッセージング基盤を利用する場合には「べき等性」を保たせる必要があります。NW の障害や端末側での障害時に、同一のメッセージが複数回配送される可能性があるためです。受信側で重複メッセージを識別して処理が重複しないよう対応が必要になります。また、配信順序が保証されないことを前提にして設計することも重要です。

---

## 8.10 まとめ

本章ではドメインイベントについて、メリット・モデリング方法・送受信方法・他コンテキストへの送信方法などの観点から学びました。DDD では「ドメインイベント」を活用することで、ドメイン内で起きた出来事をドメインモデルとして表現し、関連サービスどうしの疎結合を実現できます。イベントを活用することでシステム全体の結合度を下げ、スケーラブルで保守性の高いアーキテクチャを実現できます。
